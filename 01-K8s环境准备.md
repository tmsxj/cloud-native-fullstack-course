## 2. 理论基础

### 2.1 Kubernetes 集群组件

| 组件 | 类型 | 说明 |
|------|------|------|
| **kube-apiserver** | 控制面 | 集群入口，所有组件通过它与 API Server 交互，支持水平扩展 |
| **etcd** | 控制面 | 分布式 KV 存储（键值数据库），保存集群全部状态数据（堆叠模式运行在 Master 上） |
| **kube-scheduler** | 控制面 | 调度器，负责将 Pod 分配到合适的 Node（节点） |
| **kube-controller-manager** | 控制面 | 控制器管理器，运行 Node/ReplicaSet/Endpoint 等控制器 |
| **kubelet** | 数据面 | Node Agent（节点代理），负责 Pod 生命周期管理 |
| **kube-proxy** | 数据面 | 维护节点网络规则，实现 Service（服务）路由 |
| **containerd** | 数据面 | 容器运行时（CRI，容器运行时接口），负责镜像管理和容器运行 |
| **CoreDNS** | 插件 | 集群 DNS，为 Service 提供名称解析 |

### 2.2 kubeadm 核心概念

- **kubeadm init**：在 Master 节点初始化控制面，生成证书、启动静态 Pod
- **kubeadm join**：Worker 节点加入集群，使用 Bootstrap Token（引导令牌）完成 TLS Bootstrap
- **kubeadm token**：管理加入令牌，默认 24 小时过期
- **kubeadm reset**：重置节点，清除所有 K8s 配置和数据
- **Certificate Authority (CA，证书授权中心)**：kubeadm 使用自签名 CA 签发所有证书，默认有效期 1 年（v1.28+）

### 2.3 TLS Bootstrap 流程

TLS Bootstrap（TLS 引导）是 K8s 自动证书分发机制，允许新节点自动获取合法证书。

```
Worker Node (kubelet)                  API Server
      |                                    |
      |--- 1. 使用 Bootstrap Token 请求 --->|
      |    (CSR, CSRRequest)               |
      |                                    |
      |<-- 2. 返回 RBAC 审批结果 -----------|
      |                                    |
      |--- 3. 获取签名的客户端证书 -------->|
      |                                    |
      |<-- 4. 返回证书（有效期 1 年）-------|
      |                                    |
      |--- 5. 使用正式证书与 API 通信 ----->|
```

### 2.4 系统要求

| 项目 | Master 节点 | Worker 节点 |
|------|------------|-------------|
| CPU | >= 2 核 | >= 2 核 |
| 内存 | >= 2 GB | >= 2 GB |
| 磁盘 | >= 20 GB | >= 20 GB |
| 网络 | 节点间互通 | 节点间互通 |
| OS | Ubuntu 22.04 LTS | Ubuntu 22.04 LTS |
| 容器运行时 | containerd 1.7.x | containerd 1.7.x |

#### 3.2.4 Ansible Playbook - 系统初始化

```bash
cat > /root/k8s-prepare.yml << 'EOF'
---
- name: K8s 集群环境准备
  hosts: k8s_all # 目标主机组
  become: yes # 提权执行
  tasks:
    - name: 关闭 Swap
      command: swapoff -a
      changed_when: false

    - name: 移除 fstab 中的 swap
      lineinfile:
        path: /etc/fstab
        regexp: 'swap'
        state: absent

    - name: 关闭防火墙
      command: ufw disable
      changed_when: false

    - name: 加载内核模块
      modprobe:
        name: "{{ item }}"
        state: present
      loop:
        - overlay
        - br_netfilter

    - name: 持久化内核模块
      copy:
        content: |
          overlay
          br_netfilter
        dest: /etc/modules-load.d/k8s.conf

    - name: 配置内核参数
      sysctl:
        name: "{{ item.name }}"
        value: "{{ item.value }}"
        state: present
        reload: yes
      loop:
        - { name: 'net.bridge.bridge-nf-call-iptables', value: '1' }
        - { name: 'net.bridge.bridge-nf-call-ip6tables', value: '1' }
        - { name: 'net.ipv4.ip_forward', value: '1' }
        - { name: 'net.ipv4.conf.all.forwarding', value: '1' }
        - { name: 'vm.swappiness', value: '0' }

    - name: 安装依赖包
      apt:
        name:
          - apt-transport-https
          - ca-certificates
          - curl
          - gnupg
          - ipvsadm
          - ipset
          - conntrack
          - socat
          - ebtables
          - ethtool
          - jq
          - chrony
        state: present
        update_cache: yes

    - name: 启动 chrony
      service:
        name: chrony
        state: started
        enabled: yes

    - name: 配置 limits
      copy:
        content: |
          * soft nofile 655360
          * hard nofile 655360
          * soft nproc 655360
          * hard nproc 655360
          * soft memlock unlimited
          * hard memlock unlimited
        dest: /etc/security/limits.d/k8s.conf

    - name: 验证内核模块
      command: lsmod | grep -E 'overlay|br_netfilter'
      changed_when: false
      register: kernel_modules

    - name: 显示内核模块
      debug:
        var: kernel_modules.stdout_lines
EOF
```

#### 3.2.5 执行 Playbook

### 3.4 etcd 备份恢复

#### 3.4.1 etcd 备份脚本

```bash
cat > /root/etcd-backup.sh << 'SCRIPT'
#!/bin/bash
# etcd 备份脚本 - 在任意 Master 节点执行
# 建议通过 crontab 每小时执行一次

ETCDCTL_API=3
ETCD_CERT_DIR="/etc/kubernetes/pki/etcd"
BACKUP_DIR="/data/etcd-backup"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/etcd-backup-${DATE}.db"

mkdir -p ${BACKUP_DIR}

# 执行快照备份
ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \ # etcd服务地址
    --cacert=${ETCD_CERT_DIR}/ca.crt \ # CA证书
    --cert=${ETCD_CERT_DIR}/server.crt \ # 客户端证书
    --key=${ETCD_CERT_DIR}/server.key \ # 客户端私钥
    snapshot save ${BACKUP_FILE}

# 压缩备份文件
gzip ${BACKUP_FILE}

# 保留最近 7 天的备份
find ${BACKUP_DIR} -name "etcd-backup-*.db.gz" -mtime +7 -delete

echo "[$(date)] etcd backup completed: ${BACKUP_FILE}.gz"
SCRIPT

chmod +x /root/etcd-backup.sh
```

#### 3.4.2 设置定时备份

```bash
# 每小时备份一次
echo "0 * * * * root /root/etcd-backup.sh >> /var/log/etcd-backup.log 2>&1" \
    > /etc/cron.d/etcd-backup
```

#### 3.4.3 etcd 恢复

```bash
# 1. 停止所有控制面组件（在所有 Master 节点执行）
systemctl stop kubelet
crictl stop $(crictl ps -q --label component=kube-apiserver)
crictl stop $(crictl ps -q --label component=etcd)
crictl stop $(crictl ps -q --label component=kube-scheduler)
crictl stop $(crictl ps -q --label component=kube-controller-manager)

# 2. 在 Master-1 上恢复（先移除旧数据）
ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    snapshot restore /data/etcd-backup/etcd-backup-20260524_100000.db.gz \
    --data-dir=/var/lib/etcd-restore

# 3. 替换 etcd 数据目录
mv /var/lib/etcd /var/lib/etcd.bak
mv /var/lib/etcd-restore /var/lib/etcd
chown -R etcd:etcd /var/lib/etcd

# 4. 重启所有 Master 节点的 kubelet
systemctl start kubelet

# 5. 验证集群恢复
kubectl get nodes
kubectl get pods -n kube-system
```

## 4. 配置详解

### 4.1 IPVS 模式配置

IPVS（IP Virtual Server，IP虚拟服务器）是 Linux 内核的高性能负载均衡技术，kube-proxy 使用 IPVS 模式比 iptables 模式性能更高。

```bash
# 在所有节点加载 IPVS 内核模块
cat > /etc/modules-load.d/ipvs.conf << 'EOF'
ip_vs # IPVS核心模块
ip_vs_rr # 轮询调度算法
ip_vs_wrr # 加权轮询调度
ip_vs_sh # 源地址哈希
nf_conntrack # 连接跟踪
EOF

modprobe ip_vs
modprobe ip_vs_rr
modprobe ip_vs_wrr
modprobe ip_vs_sh
modprobe nf_conntrack
```

### 4.2 /etc/fstab 配置示例

```bash
# 正确的 fstab（无 swap 条目）
UUID=xxxx-xxxx  /  ext4  defaults  0  1
UUID=yyyy-yyyy  /boot  ext4  defaults  0  2
```

### 4.3 内核参数详解

| 参数 | 值 | 说明 |
|------|-----|------|
| `net.bridge.bridge-nf-call-iptables` | 1 | 使 bridge 流量经过 iptables（Calico/Flannel 等 CNI 插件必需） |
| `net.ipv4.ip_forward` | 1 | 启用 IP 转发（Pod 跨节点通信必需） |
| `vm.swappiness` | 0 | 禁止使用 swap（K8s 要求） |
| `kernel.pid_max` | 4194303 | 增大 PID 上限，支持大量容器 |

CRI（容器运行时接口）是 K8s 与容器运行时交互的标准接口。

## 7. 高频面试题

### Q1: Kubernetes 集群由哪些核心组件组成？各组件的作用是什么？ [难度: 中]

**答案：** Kubernetes 采用经典的控制面-数据面架构。控制面包含四个核心组件：kube-apiserver 是集群的唯一入口，所有内部组件通信和外部用户请求都通过它，它负责数据校验、持久化和 RBAC 鉴权，支持水平扩展；etcd 是基于 Raft 协议的分布式 KV 存储，保存集群的全部状态数据，是整个集群的"数据库"；kube-scheduler 负责将未调度的 Pod 根据资源需求、亲和性/反亲和性、污点容忍等策略分配到合适的 Node 上；kube-controller-manager 运行多个内置控制器（Node Controller、ReplicaSet Controller、Deployment Controller 等），通过 Watch API Server 实现声明式 API 的调谐循环。数据面包含 kubelet（Node Agent，管理 Pod 生命周期）、kube-proxy（维护网络规则实现 Service 路由）和容器运行时（如 containerd，负责实际运行容器）。这种架构实现了控制逻辑与业务负载的完全解耦。

### Q2: kubeadm init 的完整流程是什么？ [难度: 中]

**答案：** kubeadm init 分为多个 Phase 依次执行。首先是 Preflight Check，检查系统环境（swap 关闭、内核模块加载、端口占用等）是否满足要求。然后是 certs Phase，生成 CA 证书（自签名）并用 CA 签发所有组件证书（API Server、etcd Server/Peer、Front Proxy、SA 等）。接着是 kubeconfig Phase，生成 admin.conf、kubelet.conf 等配置文件。之后是 control-plane Phase，生成静态 Pod 的 YAML 文件到 /etc/kubernetes/manifests/，由 kubelet 自动拉起 etcd、API Server、Scheduler、Controller Manager。然后是 etcd Phase，等待 etcd 集群就绪。接着是 upload-config Phase，将 kubeadm-config 和 kubelet-config 上传到 ConfigMap。之后是 bootstrap-token Phase，创建用于节点加入的 Bootstrap Token。最后是 post-init Phase，安装 CoreDNS 并等待控制面完全就绪。整个过程是幂等的，可以安全地重复执行。

### Q3: etcd 在 Kubernetes 中的角色是什么？为什么需要备份？ [难度: 中]

**答案：** etcd 是 Kubernetes 集群的唯一持久化存储层，所有集群状态数据（包括 Node 信息、Pod 规格、Secret、ConfigMap、Service 定义、Deployment 状态等）都存储在 etcd 中。当 API Server 收到创建/更新/删除请求后，数据最终会被持久化到 etcd。如果 etcd 数据丢失，整个集群的状态将无法恢复，所有工作负载定义都会丢失。因此 etcd 备份是 Kubernetes 运维中最关键的灾难恢复手段。生产环境中建议：使用 `etcdctl snapshot save` 定期备份（建议每小时一次），备份文件存储到独立的存储系统（如 S3、NFS），定期验证备份的可恢复性。etcd 使用 Raft 一致性协议，在 3 节点部署下可容忍 1 个节点故障，5 节点下可容忍 2 个节点故障。

### Q4: Kubernetes 证书体系是怎样的？证书有效期是多久？ [难度: 高]

**答案：** Kubernetes 使用 PKI（Public Key Infrastructure，公钥基础设施）体系进行双向 TLS 认证。kubeadm 初始化时会创建一个自签名的根 CA（/etc/kubernetes/pki/ca.crt），然后用这个 CA 签发所有组件证书。证书分为几类：集群 CA 签发的证书包括 API Server 证书（含 SAN 列表，包含所有 Master IP 和 VIP）、kubelet 客户端证书、前端代理证书；etcd CA 签发的证书包括 etcd Server 证书、etcd Peer 证书（节点间通信）；ServiceAccount 使用独立的 RSA 密钥对签发 JWT Token（JSON Web Token）。在 Kubernetes 1.15 之前，证书有效期默认为 1 年，需要手动续期。从 1.15 开始 kubeadm 引入了自动证书轮转（默认 1 年），kubelet 会在证书剩余有效期不足 80% 时自动请求续期。可以使用 `kubeadm certs check-expiration` 查看所有证书的有效期，使用 `kubeadm certs renew all` 手动续期。

### Q5: 为什么 Kubernetes 要求关闭 Swap？ [难度: 低]

**答案：** Kubernetes 要求关闭 Swap 主要有三个原因。第一，性能一致性：Swap 会导致内存分配延迟不可预测，当系统内存不足时，内核会将不常用的内存页交换到磁盘，导致容器进程响应时间大幅增加（从纳秒级变为毫秒级），这对于延迟敏感型应用是不可接受的。第二，调度准确性：kube-scheduler 根据节点实际可用内存进行调度决策，如果启用了 Swap，报告的可用内存与实际可用内存不一致，可能导致调度到资源不足的节点。第三，QoS 保证：Kubernetes 根据 Pod 的资源请求和限制实现 QoS（Quality of Service，服务质量）分级（Guaranteed/Burstable/BestEffort），Swap 的存在会破坏内存限制的语义，BestEffort Pod 可能消耗大量 Swap 空间影响其他 Pod。可以通过 `kubelet --fail-swap-on=false` 强制允许 Swap，但不推荐在生产环境使用。

### Q6: TLS Bootstrap 是什么？工作原理是什么？ [难度: 高]

**答案：** TLS Bootstrap 是 Kubernetes 提供的一种自动证书分发机制，允许新加入的 kubelet 节点在没有预置证书的情况下自动获取合法的客户端证书。工作流程如下：首先，管理员通过 `kubeadm token create` 创建一个 Bootstrap Token（格式为 `[a-z0-9]{6}.[a-z0-9]{16}`），该 Token 关联了特定的 RBAC 角色（system:bootstrappers:kubeadm:default-node-token）。然后，新节点的 kubelet 使用该 Token 向 API Server 提交一个 CSR（Certificate Signing Request）。API Server 收到 CSR 后，根据 Token 关联的 RBAC 规则进行审批。审批通过后，controller-manager 自动签发证书并返回给 kubelet。kubelet 获得证书后，使用正式证书与 API Server 通信。整个过程实现了"零接触"的节点加入，是 kubeadm join 的底层机制。Bootstrap Token 默认 24 小时过期，可以通过 `--ttl=0` 设置永不过期（不推荐）。

### Q7: Ansible 批量配置 K8s 节点有哪些优势？ [难度: 低]

**答案：** Ansible 在 K8s 环境准备中有以下核心优势。第一，幂等性：Ansible 的大多数模块是幂等的，重复执行不会产生副作用，这对于环境一致性至关重要。第二，并行执行：Ansible 支持对多个主机并行执行任务，可以同时配置所有节点，大幅提升效率。第三，YAML 声明式语法：Playbook 使用人类可读的 YAML 格式，易于版本管理和代码审查。第四，无需 Agent：Ansible 基于 SSH 协议通信，不需要在被管理节点上安装额外的 Agent。第五，丰富的模块库：提供 sysctl、modprobe、apt、service 等模块，覆盖系统配置的所有方面。第六，错误处理：支持条件判断、循环、错误重试等高级特性。在生产环境中，建议将 Ansible Playbook 纳入 Git 版本管理，实现基础设施即代码（IaC），确保环境可复现。

### Q8: 什么是静态 Pod？它与普通 Pod 有什么区别？ [难度: 中]

**答案：** 静态 Pod 是由 kubelet 直接管理的 Pod，不经过 API Server 调度。kubelet 会监视指定目录（默认 /etc/kubernetes/manifests/）下的 YAML/JSON 文件，自动创建和管理对应的 Pod。与普通 Pod 的主要区别：第一，静态 Pod 不通过 API Server 创建，因此不会被 Scheduler 调度，始终运行在定义它的节点上。第二，静态 Pod 的名称会在节点名称后追加后缀（如 etcd-k8s-master1）。第三，API Server 上会看到对应的 Mirror Pod（只读副本），但无法通过 kubectl 删除。第四，静态 Pod 不支持大部分控制器功能（如 ReplicaSet、Deployment 管理）。在 kubeadm 部署的集群中，etcd、API Server、Scheduler 和 Controller Manager 都以静态 Pod 的形式运行在 Master 节点上，这是堆叠 etcd 拓扑的基础。静态 Pod 的配置文件修改后，kubelet 会自动重启对应的 Pod。

### Q9: kubeadm init 时 --control-plane-endpoint 参数的作用是什么？ [难度: 中]

**答案：** `--control-plane-endpoint` 参数用于指定控制面的统一访问地址，是实现高可用集群的关键参数。该地址可以是一个 DNS 名称或 IP 地址（配合负载均衡器使用）。当指定此参数时，kubeadm 会将此地址写入所有 kubeconfig 文件和证书的 SAN（Subject Alternative Name）列表中，所有节点（包括 Worker 和其他 Master）都通过这个地址与 API Server 通信。在 3 Master 高可用架构中，通常在前面部署一个 HAProxy 或 Nginx 负载均衡器（如 192.168.1.50），将请求分发到 3 个 Master 的 6443 端口。如果不指定此参数，kubeadm 会使用第一个 Master 节点的 IP 作为 API Server 地址，导致无法实现真正的多 Master 高可用。在后续添加 Master 节点时，也需要使用 `kubeadm join --control-plane` 参数指向同一个 endpoint。

### Q10: 如何验证 Kubernetes 集群的健康状态？ [难度: 低]

**答案：** 验证集群健康状态需要从多个维度检查。第一，节点状态：`kubectl get nodes` 确认所有节点处于 Ready 状态。第二，系统组件：`kubectl get pods -n kube-system` 确认所有核心组件（etcd、api-server、scheduler、controller-manager、coredns）处于 Running 状态。第三，组件健康检查：`kubectl get componentstatuses`（虽然已废弃但仍可用）或直接检查各组件日志。第四，etcd 健康检查：`etcdctl endpoint health --endpoints=https://127.0.0.1:2379` 确认 etcd 集群健康。第五，网络连通性：创建测试 Pod 验证跨节点通信和 DNS 解析。第六，系统资源：`kubectl top nodes` 确认资源使用率正常。第七，证书有效期：`kubeadm certs check-expiration` 确认证书未过期。建议将这些检查整合为一个健康检查脚本，定期执行并告警。

### Q11: Kubernetes 中 kubelet 的工作原理是什么？ [难度: 高]

**答案：** kubelet 是运行在每个 Node 上的核心组件，负责 Pod 的全生命周期管理。其工作原理基于 Watch-List 机制：kubelet 通过 List-Watch API Server 获取分配到本节点的 Pod 规格（PodSpec），然后通过 CRI（Container Runtime Interface，容器运行时接口）接口调用容器运行时（如 containerd）创建和管理容器。kubelet 的核心职责包括：Pod 生命周期管理（创建、删除、更新 Pod）、健康检查（ExecAction、HTTPGet、TCPSocket 三种探针）、资源管理（CPU/内存限制的 cgroup 设置）、卷管理（挂载/卸载存储卷）、网络管理（通过 CNI 插件配置 Pod 网络）、镜像管理（拉取容器镜像）、状态上报（向 API Server 上报 Pod 状态）。kubelet 还支持多种驱动（Docker、containerd、CRI-O），通过 CRI 标准化接口与运行时交互。kubelet 的配置通过 /var/lib/kubelet/config.yaml 管理，可以通过 kubelet 的命令行参数或配置文件进行调优。

### Q12: 什么是 RBAC？在 Kubernetes 中如何工作？ [难度: 中]

**答案：** RBAC（Role-Based Access Control，基于角色的访问控制）是 Kubernetes 的默认授权模式。RBAC 包含四个核心对象：Role 定义命名空间级别的权限规则（如对 Pod 的 get/list/watch/create 操作）；ClusterRole 定义集群级别的权限规则；RoleBinding 将 Role 绑定到用户/组/ServiceAccount（服务账号）；ClusterRoleBinding 将 ClusterRole 绑定到目标。当 API Server 收到请求时，认证（Authentication，身份验证）阶段确认请求者身份（如客户端证书、Bearer Token），然后授权（Authorization，权限校验）阶段遍历所有适用的 RoleBinding/ClusterRoleBinding，检查请求者是否具有执行该操作的权限。RBAC 的设计遵循最小权限原则，默认情况下新创建的 ServiceAccount 没有任何权限。在 TLS Bootstrap 过程中，kubeadm 预先创建了 system:bootstrappers 组的 RBAC 规则，允许新节点提交 CSR 并自动获取证书。

## 8. 故障排查案例

### 案例 1: kubeadm init 报错 "[ERROR Swap]: running with swap on is not supported"

**现象：**
```
[ERROR Swap]: running with swap on is not supported. Please disable swap
[preflight] If you know what you are doing, you can make a check non-fatal with `--ignore-preflight-errors=swap`
```

**排查步骤：**
1. 检查 swap 状态：`free -h` 或 `swapon --show`
2. 确认 swap 是否在 fstab 中：`cat /etc/fstab | grep swap`
3. 检查是否有 systemd swap 服务：`systemctl status dev-sda2.swap`

**解决方案：**
```bash
# 立即关闭 swap
swapoff -a

# 永久关闭（从 fstab 中移除）
sed -i '/swap/d' /etc/fstab

# 禁用 systemd swap
systemctl mask swap.target

# 验证
free -h
swapon --show  # 应无输出
```

### 案例 2: 节点 NotReady，kubelet 日志报 "network plugin is not ready"

**现象：**
```bash
kubectl get nodes
# NAME           STATUS     ROLES           AGE   VERSION
# k8s-worker1    NotReady   <none>          10m   v1.28.15
```

kubelet 日志：
```
Container runtime network not ready: NetworkReady=false reason:NetworkPluginNotReady message:Network plugin returns error: cni plugin not initialized
```

**排查步骤：**
1. 检查 CNI 插件是否安装：`ls /opt/cni/bin/`
2. 检查 CNI 配置文件：`ls /etc/cni/net.d/`
3. 检查 kubelet 的 CNI 配置：`ps aux | grep kubelet | grep cni`
4. 检查网络插件 Pod 状态：`kubectl get pods -n kube-system | grep -E 'calico|flannel|cilium'`

**解决方案：**
```bash
# 如果 CNI 插件未部署，需要先部署（参考模块 05）
# 如果 CNI 配置文件缺失，检查 containerd 配置
cat /etc/containerd/config.toml | grep sandbox_image

# 重启 kubelet
systemctl restart kubelet

# 检查 CNI 插件日志
kubectl logs -n kube-system -l k8s-app=calico-node --tail=50
```

### 案例 3: kubeadm join 报错 "discovery token ID not found"

**现象：**
```
error execution phase preflight: couldn't validate the identity of the API Server: could not find a JWS signature in the cluster-info ConfigMap for token ID "abcdef"
```

**排查步骤：**
1. 检查 token 是否过期：在 Master 上执行 `kubeadm token list`
2. 检查 token 格式是否正确
3. 检查集群的 cluster-info ConfigMap：`kubectl get cm cluster-info -n kube-public -o yaml`

**解决方案：**
```bash
# 在 Master 节点上重新生成 token
kubeadm token create --print-join-command

# 使用输出的完整 join 命令在 Worker 节点上执行
# 例如:
# kubeadm join 192.168.1.50:6443 --token xxxxx.xxxxxxxx \
#   --discovery-token-ca-cert-hash sha256:xxxxxxxxxxxxx
```

### 案例 4: etcd 集群不健康，API Server 无法启动

**现象：**
```bash
kubectl get nodes
# The connection to the server 192.168.1.50:6443 was refused

# etcdctl 检查
etcdctl endpoint health --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key
# 输出: https://127.0.0.1:2379 is unhealthy: failed to connect
```

**排查步骤：**
1. 检查 etcd Pod 状态：`crictl ps -a | grep etcd`
2. 检查 etcd 数据目录权限：`ls -la /var/lib/etcd/`
3. 检查磁盘空间：`df -h /var/lib/etcd`
4. 检查 etcd 日志：`crictl logs <etcd-container-id>`

**解决方案：**
```bash
# 如果是磁盘空间不足
df -h
# 清理空间后重启 kubelet
systemctl restart kubelet

# 如果是数据损坏，从备份恢复（参考 3.4.3 节）
# 如果是证书过期
kubeadm certs renew etcd-server
systemctl restart kubelet
```

### 案例 5: Ansible Playbook 执行失败 "UNREACHABLE"

**现象：**
```
fatal: [k8s-worker2]: UNREACHABLE! => {
    "msg": "Failed to connect to the host via ssh: Permission denied (publickey,password)."
}
```

**排查步骤：**
1. 手动测试 SSH 连接：`ssh root@192.168.1.55`
2. 检查 SSH 配置：`cat /root/.ssh/config`
3. 检查目标节点的 SSH 服务：`systemctl status sshd`
4. 检查目标节点的 authorized_keys：`cat /root/.ssh/authorized_keys`

**解决方案：**
```bash
# 重新配置 SSH 免密登录
ssh-keygen -t rsa -N '' -f /root/.ssh/id_rsa
ssh-copy-id root@192.168.1.55

# 如果目标节点禁用了密码登录，需要手动复制公钥
cat /root/.ssh/id_rsa.pub | ssh root@192.168.1.55 "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"

# 验证
ansible k8s-worker2 -m ping
```

### 案例 6: 证书过期导致集群不可用

**现象：**
```bash
kubectl get nodes
# Error from server (ClientCertificateHasExpired): client certificate has expired

# 检查证书有效期
kubeadm certs check-expiration
# 显示多个证书已过期
```

**排查步骤：**
1. 检查所有证书的有效期：`kubeadm certs check-expiration`
2. 检查 kubelet 证书自动轮转是否启用：`ps aux | grep kubelet | grep rotate`
3. 检查系统时间是否正确：`timedatectl`

**解决方案：**
```bash
# 1. 续期所有证书
kubeadm certs renew all

# 2. 更新 kubeconfig 文件
cp /etc/kubernetes/admin.conf $HOME/.kube/config
chown $(id -u):$(id -g) $HOME/.kube/config

# 3. 重启所有控制面组件
systemctl restart kubelet

# 4. 在其他 Master 节点上复制新的证书和配置
# 从 Master-1 复制到 Master-2/3:
# scp /etc/kubernetes/pki/* root@k8s-master2:/etc/kubernetes/pki/
# scp /etc/kubernetes/admin.conf root@k8s-master2:/etc/kubernetes/
# scp /etc/kubernetes/kubelet.conf root@k8s-master2:/etc/kubernetes/

# 5. 验证
kubeadm certs check-expiration
kubectl get nodes
```

## 9. Windows容器支持概述（进阶）

> 本节介绍Windows节点池和Windows容器的基本概念。
> 
> **适用场景**: 传统.NET Framework应用迁移、混合Linux/Windows环境
> **注意**: Windows容器镜像较大（~3GB），资源消耗高于Linux容器

### 9.1 Windows容器基础

**Windows容器类型**:

| 类型 | 隔离级别 | 内核共享 | 适用场景 |
|------|----------|----------|----------|
| **Windows Server容器** | 进程隔离 | 共享内核 | 高密度部署、性能优先 |
| **Hyper-V容器** | 虚拟化隔离 | 独立内核 | 强隔离、多租户 |

**架构对比**:
```
Linux容器:
┌─────────────────────────────────────┐
│           Linux Node                │
│  ┌─────────┐ ┌─────────┐           │
│  │Container│ │Container│           │
│  └────┬────┘ └────┬────┘           │
│       └──────┬────┘                │
│              ▼                      │
│        Linux Kernel                 │
└─────────────────────────────────────┘

Windows容器 (进程隔离):
┌─────────────────────────────────────┐
│          Windows Node               │
│  ┌─────────┐ ┌─────────┐           │
│  │Container│ │Container│           │
│  └────┬────┘ └────┬────┘           │
│       └──────┬────┘                │
│              ▼                      │
│       Windows Kernel                │
└─────────────────────────────────────┘
```

### 9.2 Windows节点池限制

**关键限制**:

| 限制项 | 说明 |
|--------|------|
| **网络模式** | 仅支持Overlay网络（VXLAN）或L2Bridge，不支持纯路由模式 |
| **CNI插件** | 需要使用win-overlay或win-l2bridge |
| **存储** | 支持SMB CSI、Azure Disk/File CSI |
| **资源** | Windows节点需要更多内存（建议8GB+） |
| **镜像** | Windows基础镜像较大（mcr.microsoft.com/windows/servercore ~3GB） |
| **版本匹配** | 容器OS版本必须与节点OS版本匹配 |

**不支持的功能**:
- HostPID

## 10. 生产环境建议

### 10.1 操作系统层面

1. **内核版本**：建议使用 5.15+ LTS 内核，获得更好的容器支持和安全补丁
2. **文件系统**：etcd 数据目录建议使用 XFS 文件系统，性能优于 ext4
3. **磁盘**：Master 节点使用 SSD，etcd 对磁盘 I/O 延迟非常敏感（建议 < 10ms）
4. **时间同步**：必须配置 NTP/chrony，集群节点时间偏差不超过 500ms
5. **ulimit**：调高文件描述符和进程数限制，避免大规模部署时遇到瓶颈

### 10.2 安全加固

1. **SSH 加固**：禁用密码登录，仅允许密钥认证；修改默认端口
2. **防火墙**：虽然关闭了 ufw，但建议在交换机层面做网络隔离
3. **内核安全**：启用 `kernel.unprivileged_bpf_disabled=1`，限制非特权用户使用 BPF
4. **审计日志**：启用 auditd，记录关键系统调用
5. **自动更新**：配置 unattended-upgrades 自动安装安全补丁

### 10.3 高可用设计

1. **etcd 集群**：生产环境至少 3 节点，建议 5 节点
2. **负载均衡**：使用 HAProxy + Keepalived 实现 API Server 的 VIP 高可用
3. **备份策略**：etcd 每小时备份，保留 7 天，异地存储
4. **灾难恢复**：定期进行恢复演练，确保备份可用
5. **监控告警**：对 etcd 的大小、Leader 变更、磁盘延迟等关键指标设置告警

### 10.4 运维规范

1. **变更管理**：所有集群变更通过 Git 管理的 Ansible Playbook 执行
2. **文档记录**：记录集群拓扑、网络规划、证书有效期等信息
3. **定期巡检**：每周检查证书有效期、节点状态、资源使用率
4. **版本管理**：K8s 版本升级遵循 N-1 策略，一次只升级一个次版本
5. **容量规划**：监控资源使用趋势，提前规划扩容
