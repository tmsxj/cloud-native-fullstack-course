# 模块33：计算机基础与内核

---

## 一、理论核心

### 1.1 进程调度与状态转换

操作系统通过进程调度器管理CPU资源的分配，理解进程状态和调度策略是分析系统性能的基础。

| 进程状态 | 含义 | 触发条件 | 排查命令 |
|----------|------|----------|----------|
| R（Running） | 运行态 | 正在CPU上执行或在运行队列中等待 | `ps aux` |
| S（Sleeping） | 可中断睡眠 | 等待某事件完成（如IO） | `ps aux` |
| D（Disk Sleep） | 不可中断睡眠 | 等待IO完成，无法被信号唤醒 | `ps aux` |
| T（Stopped） | 停止态 | 收到SIGSTOP信号 | `ps aux` |
| Z（Zombie） | 僵尸态 | 已终止但父进程未回收 | `ps aux` |

| 调度策略 | 适用场景 | 特点 | Linux实现 |
|----------|----------|------|-----------|
| CFS（完全公平调度） | 普通进程 | 按虚拟运行时间（vruntime）排序，红黑树实现 | SCHED_NORMAL |
| 实时调度FIFO | 硬实时任务 | 优先级高的进程一直运行，直到主动放弃 | SCHED_FIFO |
| 实时调度RR | 软实时任务 | 同优先级进程时间片轮转 | SCHED_RR |
| 批量调度 | 后台任务 | 低优先级，CPU空闲时运行 | SCHED_BATCH |

**关键要点**：
- CFS调度器的核心思想是：每个进程获得公平的CPU时间，优先级高的进程vruntime增长慢
- 实时进程的优先级（1-99）高于普通进程（100-139），但实时进程可能导致普通进程饥饿
- 上下文切换开销包括：保存/恢复寄存器、切换地址空间、刷新TLB

---

### 1.2 虚拟内存与页面置换

Linux采用虚拟内存机制，让每个进程拥有独立的地址空间，通过页表映射到物理内存。理解内存分配和页面置换是排查OOM和性能问题的核心。

| 概念 | 含义 | 触发条件 | 排查命令 |
|------|------|----------|----------|
| 页表（Page Table） | 虚拟地址到物理地址的映射表 | 每次内存访问 | `cat /proc/<pid>/maps` |
| TLB（转译后备缓冲） | 页表的高速缓存 | 每次内存访问先查TLB | `perf stat -e dTLB-loads` |
| 缺页中断（Page Fault） | 访问的页不在物理内存中 | 首次访问或页被换出 | `sar -B` |
| 页面置换（Swap） | 将不活跃的页换出到磁盘 | 物理内存不足 | `vmstat` |
| 写时复制（COW） | fork时共享页，写入时才复制 | fork子进程 | `strace -e trace=fork` |

| 页面置换算法 | 策略 | 优点 | 缺点 |
|--------------|------|------|------|
| LRU（最近最少使用） | 淘汰最久未访问的页 | 符合局部性原理 | 实现复杂，需维护访问时间 |
| Clock（时钟算法） | 近似LRU，使用访问位 | 实现简单，开销小 | 精度不如LRU |
| FIFO（先进先出） | 淘汰最早进入的页 | 实现最简单 | 可能淘汰热点页（Belady异常） |

**关键要点**：
- 缺页中断分为：major fault（磁盘IO）和minor fault（仅分配页框），major fault性能影响大
- swap使用>0即表示物理内存不足，应告警
- Linux使用LRU的近似实现（活跃/非活跃链表），配合swapiness参数控制换出积极性

### 1.3 文件系统与IO栈

Linux文件系统采用VFS（虚拟文件系统）统一接口，底层支持多种具体文件系统。理解IO栈是排查磁盘性能问题的关键。

| 层级 | 组件 | 作用 | 优化方向 |
|------|------|------|----------|
| 用户层 | 应用IO调用 | read/write/mmap | 批量读写、异步IO |
| VFS层 | 虚拟文件系统 | 统一接口，屏蔽具体文件系统差异 | - |
| 具体文件系统 | ext4/xfs/btrfs | 管理inode、目录、数据块 | 选择合适的文件系统 |
| 页缓存层 | Page Cache | 缓存文件数据，减少磁盘IO | 调整dirty_ratio |
| 块层 | Block Layer | IO调度、请求合并 | 选择合适的IO调度器 |
| 设备驱动 | SCSI/NVMe驱动 | 与硬件交互 | 更新驱动 |
| 硬件层 | HDD/SSD/NVMe | 物理存储 | 选择高性能硬件 |

| 文件系统 | 适用场景 | 最大文件大小 | 最大卷大小 | 特点 |
|----------|----------|------------|------------|------|
| ext4 | 通用场景 | 16TB | 1EB | 成熟稳定，广泛支持 |
| xfs | 大文件、高并发 | 8EB | 8EB | 高性能，延迟分配 |
| btrfs | 快照、数据完整性 | 16EB | 16EB | 功能丰富，但稳定性稍差 |
| zfs | 企业级存储 | 16EB | 256ZB | 数据完整性强，内存需求大 |

**关键要点**：
- Page Cache是文件IO性能的关键，读文件时先查Page Cache，写文件时先写入Page Cache（延迟刷盘）
- `dirty_ratio`（默认20%）控制脏页占总内存比例，超过此比例强制刷盘，调大可减少刷盘频率但增加丢数据风险
- ext4的日志模式有ordered（默认）、journal、writeback三种，ordered性能最好，journal最安全

## 二、实操演练

### 任务1：分析进程调度与上下文切换

**任务目标**：定位高上下文切换导致的CPU系统态飙高

**操作步骤**：

```bash
# 步骤1：查看上下文切换统计
vmstat 1 5
# 预期输出：cs（context switch）列显示每秒上下文切换次数

# 步骤2：查看详细的上下文切换
pidstat -w 1 -p <pid>
# 预期输出：cswch/s（自愿切换）和nvcswch/s（非自愿切换）

# 步骤3：查看自愿切换原因（进程主动放弃CPU）
cat /proc/<pid>/status | grep voluntary_ctxt_switches
# 预期输出：自愿上下文切换次数

# 步骤4：查看非自愿切换原因（被调度器抢占）
cat /proc/<pid>/status | grep nonvoluntary_ctxt_switches
# 预期输出：非自愿上下文切换次数

# 步骤5：使用perf分析上下文切换热点
perf stat -e context-switches,cpu-migrations -p <pid> sleep 10
# 预期输出：10秒内的上下文切换和CPU迁移次数

# 步骤6：查看线程迁移（跨CPU核心）
perf stat -e migrations -p <pid> sleep 10
# 预期输出：CPU迁移次数，频繁迁移影响缓存命中率

# 步骤7：绑定进程到指定CPU（减少迁移）
taskset -c 0,1 <command>
# 预期输出：进程仅在CPU 0和1上运行

# 步骤8：验证修复
vmstat 1 5
# 预期输出：cs列下降到正常范围
```

**注意事项**：
- 自愿切换通常是IO等待，非自愿切换通常是时间片用完被抢占
- 大量线程（>CPU核心数*2）会导致频繁上下文切换，降低效率
- CPU亲和性（taskset）可减少缓存失效，但可能导致负载不均衡

### 任务2：分析缺页中断与内存分配

**任务目标**：定位大量缺页中断导致的性能问题

**操作步骤**：

```bash
# 步骤1：查看缺页中断统计
sar -B 1 5
# 预期输出：pgpgin/s、pgpgout/s、fault/s、majflt/s

# 步骤2：查看进程的缺页中断
pidstat -r 1 -p <pid>
# 预期输出：minflt/s（小缺页）和majflt/s（大缺页）

# 步骤3：使用perf分析缺页中断
perf stat -e page-faults,major-faults -p <pid> sleep 10
# 预期输出：10秒内的缺页中断次数

# 步骤4：查看进程的内存映射
pmap -x <pid>
# 预期输出：各内存区域的地址、大小、RSS、Dirty页

# 步骤5：查看进程的内存统计
cat /proc/<pid>/status | grep -E "VmRSS|VmSize|VmData|VmStk"
# 预期输出：实际物理内存、虚拟内存、数据段、栈大小

# 步骤6：使用strace跟踪内存分配
strace -e trace=mmap,munmap,brk -p <pid>
# 预期输出：进程的内存分配和释放系统调用

# 步骤7：调整swappiness（控制换出积极性）
cat /proc/sys/vm/swappiness
# 预期输出：60（默认值）
echo 10 > /proc/sys/vm/swappiness
# 预期输出：降低swap积极性，优先使用物理内存

# 步骤8：验证修复
sar -B 1 5
# 预期输出：majflt/s下降到接近0
```

**注意事项**：
- minor fault是正常行为（首次访问或COW），major fault才需要关注（磁盘IO）
- 大量major fault通常表示内存不足或mmap文件未预热
- swappiness=0不表示禁用swap，而是尽量不换出

### 任务3：分析文件系统性能与Page Cache

**任务目标**：定位文件系统IO瓶颈，优化Page Cache使用

**操作步骤**：

```bash
# 步骤1：查看Page Cache使用情况
cat /proc/meminfo | grep -E "Cached|Buffers|Dirty|Writeback"
# 预期输出：Cached（页缓存）、Dirty（脏页）、Writeback（正在回写）

# 步骤2：查看脏页回写参数
cat /proc/sys/vm/dirty_ratio
cat /proc/sys/vm/dirty_background_ratio
# 预期输出：20和10（默认值，百分比）

# 步骤3：查看文件系统缓存命中率
cat /proc/sys/fs/file-nr
# 预期输出：已分配文件句柄 / 已使用 / 最大值

# 步骤4：清空Page Cache（测试用）
echo 3 > /proc/sys/vm/drop_caches
# 预期输出：清空pagecache、dentries和inodes

# 步骤5：使用vmtouch预热文件到缓存
vmtouch -t /path/to/file
# 预期输出：将文件加载到Page Cache

# 步骤6：查看文件在缓存中的状态
vmtouch /path/to/file
# 预期输出：显示文件的缓存命中率

# 步骤7：调整脏页回写频率
echo 5 > /proc/sys/vm/dirty_ratio
echo 2 > /proc/sys/vm/dirty_background_ratio
# 预期输出：更早开始回写，减少突发IO

# 步骤8：验证修复
iostat -x 1 5
# 预期输出：IO更平滑，await降低
```

**注意事项**：
- `drop_caches`是安全的，只清理缓存不杀进程，但会导致后续IO变慢（需重新加载缓存）
- 数据库等重IO应用建议调大`dirty_ratio`，减少刷盘频率
- 文件句柄耗尽会导致"Too many open files"错误，需调整`fs.file-max`

### 任务4：分析系统调用与内核性能

**任务目标**：定位高频系统调用导致的性能瓶颈

**操作步骤**：

```bash
# 步骤1：使用strace统计系统调用
strace -c -p <pid>
# 预期输出：各系统调用的次数、耗时、错误数

# 步骤2：使用perf分析内核热点
perf top -p <pid>
# 预期输出：用户态和内核态的热点函数

# 步骤3：使用bpftrace跟踪特定系统调用
bpftrace -e 'tracepoint:raw_syscalls:sys_enter { @[comm] = count(); }'
# 预期输出：各进程的系统调用频率

# 步骤4：查看系统调用表
cat /usr/include/asm/unistd_64.h | grep __NR_
# 预期输出：系统调用号和名称对应关系

# 步骤5：分析read/write系统调用大小
strace -e trace=read,write -s 0 -p <pid>
# 预期输出：每次read/write的字节数，过小表示IO碎片化

# 步骤6：使用ftrace跟踪内核函数
echo function > /sys/kernel/debug/tracing/current_tracer
echo <kernel-function> > /sys/kernel/debug/tracing/set_ftrace_filter
cat /sys/kernel/debug/tracing/trace_pipe
# 预期输出：指定内核函数的调用跟踪

# 步骤7：查看系统调用开销（用户态vs内核态）
time -p <command>
# 预期输出：real/user/sys时间，sys高表示内核开销大

# 步骤8：验证修复
strace -c -p <pid>
# 预期输出：系统调用次数减少或耗时降低
```

**注意事项**：
- strace有性能开销（约20-30%），生产环境谨慎使用
- 高频小IO（如每次read 1字节）会导致大量系统调用，应改为批量读写
- 用户态实现（如DPDK、io_uring）可减少系统调用开销

### 任务5：排查内核panic与系统崩溃

**任务目标**：分析内核panic日志，定位系统崩溃根因

**操作步骤**：

```bash
# 步骤1：查看panic日志
dmesg | grep -i panic
# 预期输出：panic时的内核堆栈和错误信息

# 步骤2：查看oops（非致命内核错误）
dmesg | grep -i oops
# 预期输出：oops的位置和调用栈

# 步骤3：查看硬件错误（MCA/MCE）
dmesg | grep -i "machine check"
# 预期输出：CPU或内存硬件错误

# 步骤4：查看syslog中的内核消息
journalctl -k -b -1 | grep -E "panic|oops|BUG|ERROR"
# 预期输出：上一次启动的内核错误

# 步骤5：分析vmcore（如配置了kdump）
crash /usr/lib/debug/lib/modules/$(uname -r)/vmlinux /var/crash/*/vmcore
# 预期输出：进入crash工具，分析崩溃时的内存状态

# 步骤6：查看内核模块
cat /proc/modules | grep <suspected-module>
# 预期输出：可疑模块的加载状态和依赖

# 步骤7：检查内核参数
cat /proc/cmdline
# 预期输出：启动时传递的内核参数

# 步骤8：验证修复（更新内核或驱动）
yum update kernel
# 预期输出：安装新内核，重启后验证
```

**注意事项**：
- panic常见原因：硬件故障、驱动bug、内核参数错误、内存损坏
- kdump可在panic时自动捕获vmcore，是排查panic的关键工具
- 内核升级前建议在测试环境验证，避免新内核引入兼容性问题

## 三、面试真题

### 基础 高频 - Q1: 进程有哪些状态？如何查看？

> **参考答案**：
> 1. **R（Running）**：运行态，正在CPU上执行或在运行队列中等待
> 2. **S（Sleeping）**：可中断睡眠，等待某事件完成（如IO），可被信号唤醒
> 3. **D（Disk Sleep）**：不可中断睡眠，等待IO完成，无法被信号唤醒
> 4. **T（Stopped）**：停止态，收到SIGSTOP信号
> 5. **Z（Zombie）**：僵尸态，已终止但父进程未调用wait()回收
> 6. **查看命令**：`ps aux`的STAT列，`top`的S列，`cat /proc/<pid>/status`的State字段
>
> **延伸知识**：D状态进程最危险，无法通过kill终止，通常卡在磁盘IO或NFS挂载，只能等待IO完成或重启系统

---

### 基础 高频 - Q2: 什么是虚拟内存？为什么要用虚拟内存？

> **参考答案**：
> 1. **定义**：虚拟内存是每个进程独立的地址空间，通过页表映射到物理内存，让进程以为自己独占整个内存
> 2. **隔离性**：每个进程有独立的地址空间，一个进程崩溃不会影响其他进程
> 3. **连续性**：物理内存可以不连续，但虚拟地址空间是连续的，简化程序设计
> 4. **扩展性**：物理内存不足时，可将不活跃的页换出到磁盘（swap），让程序使用超过物理内存的地址空间
> 5. **共享性**：多个进程可映射到同一物理页（如共享库、COW fork）
>
> **延伸知识**：页表是多级结构（x86_64使用4级页表），TLB是页表的高速缓存，TLB miss会导致额外的内存访问开销

---

### 中等 高频 - Q3: 什么是缺页中断？major fault和minor fault有什么区别？

> **参考答案**：
> 1. **缺页中断**：进程访问的虚拟页不在物理内存中时，CPU触发缺页中断，由内核处理
> 2. **minor fault**：页已分配但不在内存中（如首次访问、COW、内存映射文件），无需磁盘IO，处理快
> 3. **major fault**：页需要从磁盘加载（如swap换入、可执行文件加载），需要磁盘IO，处理慢
> 4. **排查**：`sar -B`查看fault/s和majflt/s，`pidstat -r`查看进程的缺页情况
> 5. **优化**：减少major fault需保证物理内存充足，避免swap；预热mmap文件可减少运行时major fault
>
> **延伸知识**：大量major fault是内存不足的典型症状，会导致系统卡顿，应监控并告警

---

### 中等 高频 - Q4: Linux文件系统的IO栈是怎样的？Page Cache起什么作用？

> **参考答案**：
> 1. **IO栈层级**：用户层→VFS层→具体文件系统（ext4/xfs）→Page Cache→块层→设备驱动→硬件
> 2. **Page Cache作用**：缓存文件数据，减少磁盘IO。读文件时先查Page Cache，命中则直接返回；写文件时先写入Page Cache，延迟刷盘
> 3. **脏页回写**：`dirty_ratio`（默认20%）控制脏页比例，超过则强制刷盘；`dirty_background_ratio`（默认10%）控制后台刷盘阈值
> 4. **优化方向**：批量读写减少系统调用、选择合适的IO调度器（SSD用none）、调整脏页参数平衡性能和数据安全
> 5. **清空缓存**：`echo 3 > /proc/sys/vm/drop_caches`可清空Page Cache，但会导致后续IO变慢
>
> **延伸知识**：Direct IO绕过Page Cache直接读写磁盘，适用于数据库等自己管理缓存的应用

---

### 中等 中频 - Q5: 什么是写时复制（COW）？fork时发生了什么？

> **参考答案**：
> 1. **COW定义**：fork创建子进程时，不立即复制父进程的内存页，而是让父子共享同一物理页。当任一进程写入时，才复制该页
> 2. **fork过程**：①复制父进程的页表（标记为只读）②复制文件描述符表③复制信号处理表④设置子进程PID和PPID
> 3. **COW优点**：减少fork时的内存复制开销，提高fork速度
> 4. **COW缺点**：首次写入时触发缺页中断，复制页面，有短暂延迟
> 5. **排查**：大量COW可能导致minor fault增加，`strace -e trace=fork`跟踪fork调用
>
> **延伸知识**：execve替换进程映像时会释放COW共享的页，重新加载可执行文件到内存

---

### 困难 中频 - Q6: 讲一个你处理过的内核/系统级故障案例（STAR-R法）

> **参考答案**（以大量major fault导致系统卡顿为例）：
>
> **Situation**：数据分析平台，夜间批量任务运行时系统突然卡顿，SSH连接无响应，监控显示CPU的wa（IO等待）达到80%
>
> **Task**：作为运维工程师，定位系统卡顿根因并恢复服务
>
> **Action**：
> 1. 通过带外管理（IPMI）登录服务器，`vmstat 1`确认wa高，us和sy正常
> 2. `sar -B 1`发现majflt/s达到5000+，大量缺页中断
> 3. `ps aux --sort=-%mem`发现数据分析进程的RSS远小于VSZ，大量内存被换出到swap
> 4. `free -h`确认swap使用率达到90%，available内存接近0
> 5. 根因分析：批量任务内存需求超过物理内存，触发大量swap换入换出
> 6. 紧急处理：暂停部分非关键任务，释放内存压力
>
> **Result**：系统恢复响应，majflt/s下降到正常水平。后续将批量任务拆分为小批次，并增加物理内存
>
> **Reflection**：swap是性能杀手，一旦使用系统就会卡顿。改进措施包括：监控swap使用率告警、批量任务内存评估、考虑使用OOM Killer保护关键进程
>
> **延伸知识**：Linux的swappiness参数控制换出积极性，数据库等应用建议设为10，优先使用物理内存
