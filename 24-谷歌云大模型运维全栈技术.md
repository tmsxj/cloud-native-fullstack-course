## 一、理论核心

### 1.1 Vertex AI 平台架构

Vertex AI 是 Google Cloud 的统一机器学习平台，提供从数据标注、模型训练、模型部署到在线推理的全链路能力。运维工程师的核心关注点在于模型部署方式、资源调度、成本控制和可观测性集成。

| 对比维度 | Vertex AI 全托管 | Cloud Run 自托管 | GKE 自托管 |
|----------|-----------------|-----------------|------------|
| 运维复杂度 | 低，Google管理底层 | 中，需管理容器镜像 | 高，需管理节点和调度 |
| 弹性伸缩 | 自动按请求量伸缩 | 自动缩至零实例 | 需配置HPA/VPA |
| GPU支持 | 按需分配T4/A100/H100 | 仅CPU推理 | 支持GPU节点池 |
| 成本模型 | 按推理请求+运行时间计费 | 按请求+容器运行时间计费 | 按节点+Pod资源计费 |
| 适用场景 | 快速验证、低频推理 | 中等规模API服务 | 大规模生产、多模型编排 |
| 自定义程度 | 有限，受限于预置运行时 | 高，完全控制容器 | 最高，控制所有层 |

**关键要点**：
- Vertex AI Endpoint 支持多模型共享同一个推理节点，通过流量分配实现A/B测试和灰度发布
- Vertex AI Prediction 支持批量推理任务（Batch Prediction），适合离线数据处理场景
- Vertex AI Model Registry 提供模型版本管理，与CI/CD流水线集成实现模型自动发布
- Vertex AI 与 Cloud Operations（Monitoring/Logging/Trace）原生集成，推理延迟和错误率自动采集

### 1.2 Gemini 模型家族与选型

Gemini 是 Google 的多模态大模型家族，覆盖从轻量级到旗舰级的全系列。运维选型的核心依据是延迟、成本、精度三者的平衡。

| 模型 | 参数量 | 上下文窗口 | 延迟 | 适用场景 | 单价（每百万Token） |
|------|--------|-----------|------|---------|-------------------|
| Gemini 2.5 Flash | 未公开 | 1M Token | 低（<500ms） | 日志分析、告警分类、客服对话 | 输入$0.15/输出$0.60 |
| Gemini 2.5 Pro | 未公开 | 1M Token | 中（1-3s） | 复杂推理、架构分析、故障诊断 | 输入$1.25/输出$10.00 |
| Gemma 3 | 1B-27B | 128K Token | 极低（<200ms） | 端侧部署、轻量分类、本地推理 | 开源免费（云端推理按Compute Engine计费） |
| Gemini 3 Flash Preview | 未公开 | 1M Token | 低 | 智能代理主力、多模态任务 | 输入$0.10/输出$0.40 |

**关键要点**：
- 运维场景优先选 Gemini 2.5 Flash，成本仅为 Pro 的1/16，延迟更低，对日志分析、告警分类等任务精度足够
- Gemma 3 适合私有化部署场景，可在本地服务器或GKE中运行，无需调用云端API
- 所有 Gemini 模型支持 Function Calling，可直接调用 GCP API（如创建VM、查询监控指标）
- Vertex AI 支持模型微调（Supervised Fine-tuning），可用企业运维数据定制专属模型

### 1.3 AI Agent 运维架构

AI Agent（智能体）是 2025-2026 年大模型落地的核心形态。Google Cloud 提供三层 Agent 构建体系，运维工程师需要理解各层的定位和部署方式。

| 层级 | 产品 | 目标用户 | 部署方式 | 核心能力 |
|------|------|---------|---------|---------|
| 代码层 | Agent Development Kit（ADK） | 开发/运维团队 | Vertex AI Agent Engine / Cloud Run | 多Agent协作、工具定义、上下文管理 |
| 无代码层 | Google Agentspace | 非技术人员 | Google Workspace 内置 | 跨应用数据整合、工作流自动化 |
| 预置层 | Gemini Code Assist / Cloud Assist | 全员 | IDE插件 / Cloud Console | 代码生成、PR审查、云资源优化建议 |

**关键要点**：
- ADK（Agent Development Kit）是运维团队构建定制Agent的核心工具包，支持 Python 和 TypeScript
- Agent 可注册自定义工具（Tool），如"查询Prometheus指标"、"创建K8s资源"、"发送告警通知"
- Vertex AI Agent Engine 提供 Agent 的托管运行环境，自动处理并发、限流、日志采集
- ReAct（Reasoning + Acting）模式是 Agent 的核心推理范式：模型先推理需要调用哪个工具，再执行工具获取结果，循环直到得出最终答案

## 二、实操演练

### 任务1：启用 Vertex AI API 并部署 Gemini 推理端点

**任务目标**：在 GCP 项目中启用 Vertex AI，创建 Gemini 2.5 Flash 模型的在线推理端点，通过 REST API 发送运维日志分析请求。

**操作步骤**：

```bash
# 步骤1：启用 Vertex AI API（需拥有项目Owner或Editor角色）
gcloud services enable aiplatform.googleapis.com \
    --project=my-project-id \
    --region=asia-east1
# 预期输出：Operation "operations/xxx" completed successfully.

# 步骤2：验证API已启用
gcloud services list --enabled \
    --filter="name:aiplatform.googleapis.com" \
    --project=my-project-id
# 预期输出：NAME                    TITLE
#           aiplatform.googleapis.com  Vertex AI API

# 步骤3：通过REST API调用Gemini 2.5 Flash分析运维日志
# 无需手动创建Endpoint，Gemini模型支持直接调用（Serverless方式）
curl -X POST \
    "https://asia-east1-aiplatform.googleapis.com/v1/projects/my-project-id/locations/asia-east1/publishers/google/models/gemini-2.5-flash:generateContent" \
    -H "Authorization: Bearer $(gcloud auth print-access-token)" \
    -H "Content-Type: application/json" \
    -d '{
        "contents": [{
            "parts": [{
                "text": "分析以下K8s Pod日志，判断故障根因：\nERROR: connection refused to mysql-service:3306\nWARN: retrying in 5 seconds...\nERROR: timeout after 3 retries\nPod: order-service-7d4f5b-x2k9l\nNamespace: production"
            }]
        }]
    }'
# 预期输出：JSON格式的分析结果，包含故障根因判断和修复建议

# 步骤4：查看API调用配额和用量
gcloud beta ai quotas describe \
    --model=gemini-2.5-flash \
    --project=my-project-id \
    --region=asia-east1
```

**注意事项**：
- Gemini 模型的 Serverless 调用无需创建 Endpoint，直接通过 REST API 发请求即可，适合低频调用场景
- 高频生产场景需创建 Dedicated Endpoint（独享推理节点），避免冷启动延迟
- API Key 方式仅适合本地测试，生产环境必须使用 Service Account + IAM 权限控制

### 任务2：在 Cloud Run 上部署 LLM 推理服务

**任务目标**：将一个基于 Gemini API 的日志分析服务打包为容器镜像，部署到 Cloud Run，实现 HTTP API 接口。

**操作步骤**：

```bash
# 步骤1：创建项目目录和源码文件
mkdir llm-ops-service && cd llm-ops-service

# 创建 main.py（FastAPI推理服务）
cat > main.py << 'PYEOF'
from fastapi import FastAPI, HTTPException
from fastapi.middleware.cors import CORSMiddleware
import google.generativeai as genai
import os

app = FastAPI(title="LLM运维分析服务")

# 允许跨域（开发环境）
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_methods=["*"],
    allow_headers=["*"],
)

# 初始化Gemini客户端（使用环境变量中的API Key）
genai.configure(api_key=os.environ.get("GEMINI_API_KEY"))
model = genai.GenerativeModel("gemini-2.5-flash")

@app.post("/analyze")
async def analyze_log(request: dict):
    """分析运维日志，返回故障诊断结果"""
    log_content = request.get("log", "")
    context = request.get("context", "K8s生产环境")
    prompt = f"你是资深运维工程师。分析以下{context}日志，给出：1.故障根因 2.影响范围 3.修复命令\n\n日志内容：\n{log_content}"
    try:
        response = model.generate_content(prompt)
        return {"analysis": response.text, "model": "gemini-2.5-flash"}
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))

@app.get("/health")
async def health_check():
    """健康检查端点"""
    return {"status": "healthy", "service": "llm-ops-service"}
PYEOF

# 创建 requirements.txt
cat > requirements.txt << 'EOF'
fastapi==0.115.0
uvicorn==0.30.0
google-generativeai==0.8.0
EOF

# 创建 Dockerfile
cat > Dockerfile << 'EOF'
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY main.py .
EXPOSE 8080
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8080"]
EOF

# 步骤2：构建并推送镜像到 Artifact Registry
gcloud artifacts repositories create llm-repo \
    --repository-format=docker \
    --location=asia-east1 \
    --project=my-project-id

gcloud auth configure-docker asia-east1-docker.pkg.dev

docker build -t asia-east1-docker.pkg.dev/my-project-id/llm-repo/llm-ops-service:v1 .

docker push asia-east1-docker.pkg.dev/my-project-id/llm-repo/llm-ops-service:v1
# 预期输出：v1: digest: sha256:xxxx size: xxxx

# 步骤3：部署到 Cloud Run
gcloud run deploy llm-ops-service \
    --image=asia-east1-docker.pkg.dev/my-project-id/llm-repo/llm-ops-service:v1 \
    --region=asia-east1 \
    --platform=managed \
    --allow-unauthenticated \
    --memory=512Mi \
    --cpu=1 \
    --min-instances=0 \
    --max-instances=5 \
    --set-env-vars="GEMINI_API_KEY=your-api-key" \
    --timeout=300

# 步骤4：获取服务URL并测试
SERVICE_URL=$(gcloud run services describe llm-ops-service \
    --region=asia-east1 --format='get(status.url)')
echo "服务地址: $SERVICE_URL"

# 测试日志分析接口
curl -X POST "$SERVICE_URL/analyze" \
    -H "Content-Type: application/json" \
    -d '{"log": "Error: Pod CrashLoopBackOff, container exit code 137, OOMKilled", "context": "K8s生产环境"}'
# 预期输出：JSON格式的故障分析结果

# 步骤5：查看服务日志
gcloud logging read "resource.type=cloud_run_revision AND resource.labels.service_name=llm-ops-service" \
    --limit=20 \
    --region=asia-east1 \
    --format='table(timestamp,textPayload)'
```

**注意事项**：
- GEMINI_API_KEY 不要硬编码在镜像中，使用 Cloud Run 的 --set-env-vars 或 Secret Manager 注入
- 生产环境必须设置 --no-allow-unauthenticated，配合 IAM 认证控制访问权限
- min-instances=0 表示无请求时不计费，适合低频运维场景；高频场景建议设为1避免冷启动

### 任务3：在 GKE 上部署 Gemma 3 本地推理服务

**注意事项**：
- GPU节点池使用 --spot 参数创建抢占式实例，成本约为按需实例的30-60%，适合非实时推理场景

- 无GPU环境可使用 CPU 推理镜像，但延迟会从秒级增加到十秒级
- 生产环境建议配置 PodDisruptionBudget 防止节点维护时推理服务全部中断

### 任务4：构建 LLM 驱动的告警分析 Agent

**任务目标**：使用 Vertex AI Agent Engine 注册自定义工具，构建一个能自动查询 Prometheus 指标、分析告警、生成修复命令的运维 Agent。

**操作步骤**：

```bash
# 步骤1：安装Vertex AI SDK
pip install google-cloud-aiplatform google-adk

# 步骤2：创建Agent定义文件
cat > ops_agent.py << 'PYEOF'
"""LLM运维Agent：自动分析告警并生成修复方案"""
import vertexai
from vertexai import agent_engine

# 初始化Vertex AI
vertexai.init(project="my-project-id", location="asia-east1")

# 定义Agent可调用的工具
tools = [
    {
        "name": "query_prometheus",
        "description": "查询Prometheus监控指标，返回当前值和趋势",
        "parameters": {
            "type": "object",
            "properties": {
                "metric_name": {
                    "type": "string",
                    "description": "PromQL指标名称，如kube_node_status_condition"
                },
                "time_range": {
                    "type": "string",
                    "description": "时间范围，如1h、6h、24h"
                }
            },
            "required": ["metric_name"]
        }
    },
    {
        "name": "get_pod_logs",
        "description": "获取指定Pod的最近日志",
        "parameters": {
            "type": "object",
            "properties": {
                "namespace": {"type": "string"},
                "pod_name": {"type": "string"},
                "lines": {"type": "integer", "description": "日志行数，默认100"}
            },
            "required": ["namespace", "pod_name"]
        }
    },
    {
        "name": "execute_kubectl",
        "description": "执行kubectl命令并返回结果（只读命令）",
        "parameters": {
            "type": "object",
            "properties": {
                "command": {
                    "type": "string",
                    "description": "kubectl命令，如get pods -n production"
                }
            },
            "required": ["command"]
        }
    }
]

# 创建Agent
agent = agent_engine.Agent(
    name="ops-analyst-agent",
    display_name="运维分析Agent",
    instructions="""你是一个资深K8s运维专家。当收到告警或故障报告时：
1. 先用query_prometheus查询相关指标，确认故障范围
2. 再用get_pod_logs查看异常Pod的日志
3. 必要时用execute_kubectl查看集群状态
4. 综合分析后给出：故障根因、影响范围、修复命令、预防措施
回答使用中文，命令使用bash代码块格式。""",
    model="gemini-2.5-flash",
    tools=tools
)

# 部署Agent到Agent Engine
remote_agent = agent_engine.register(
    agent=agent,
    display_name="运维分析Agent",
    project="my-project-id",
    location="asia-east1"
)
print(f"Agent部署成功，ID: {remote_agent.name}")
PYEOF

python test_agent.py
```

## 三、面试真题

### 基础 高频 - Q1: Vertex AI、Cloud Run、GKE 三种部署LLM服务的方式有什么区别？如何选择？

> **参考答案**：
> 1. **Vertex AI 全托管**：Google管理底层基础设施，无需运维容器和节点，适合快速验证和低频推理场景，成本按请求量计费
> 2. **Cloud Run 容器化**：运维人员管理容器镜像，Google自动伸缩（可缩至零），适合中等规模API服务，成本按容器运行时间计费
> 3. **GKE 自托管**：运维人员管理节点池、Pod调度、GPU分配，适合大规模多模型编排和需要GPU加速的场景，成本按节点+Pod资源计费
>
> **选型原则**：验证阶段用 Vertex AI，单模型API服务用 Cloud Run，多模型生产集群用 GKE

---

### 中等 高频 - Q2: 如何在GCP上实现LLM驱动的运维告警自动分析？

> **参考答案**：
> 1. **告警接入层**：Cloud Monitoring 检测到异常指标后，通过 Pub/Sub 发送告警事件
> 2. **Agent处理层**：Vertex AI Agent Engine 上的运维Agent接收告警，自动调用工具（查询Prometheus指标、获取Pod日志、执行kubectl命令）
> 3. **分析输出层**：Agent综合多源数据后输出故障根因、影响范围和修复命令，通过通知渠道发送给值班人员
> 4. **闭环验证**：修复后Cloud Monitoring指标恢复，Agent确认故障解除
>
> **延伸知识**：Agent的工具注册机制（Function Calling）是实现自动化的关键，Gemini原生支持工具调用

---

### 中等 中频 - Q3: Gemini 2.5 Flash 和 Gemini 2.5 Pro 在运维场景中如何选型？

> **参考答案**：
> 1. **Gemini 2.5 Flash**：运维场景的首选，延迟低（<500ms）、成本低（Pro的1/16），适合日志分析、告警分类、命令生成等高频任务
> 2. **Gemini 2.5 Pro**：适合复杂推理场景，如架构分析、多步骤故障诊断、容量规划建议，精度更高但成本和延迟显著增加
> 3. **Gemma 3**：开源模型，适合私有化部署和数据安全要求严格的场景，可在本地或GKE中运行，不依赖云端API
>
> **延伸知识**：实际生产中可采用分级策略——Flash处理80%的常规告警，Pro处理20%的复杂故障

---

### 困难 中频 - Q4: 如何控制LLM运维服务的成本？有哪些优化手段？

> **参考答案**：
> 1. **模型分级**：常规告警用Flash（$0.15/M输入Token），复杂分析用Pro（$1.25/M输入Token），避免所有请求都用Pro
> 2. **Prompt优化**：精简Prompt长度，减少输入Token数；使用System Instruction固定角色设定，避免每次重复
> 3. **缓存策略**：对相同告警规则设置缓存，避免重复分析相同告警消耗Token
> 4. **弹性伸缩**：Cloud Run设置min-instances=0，无请求时不计费；GKE配置HPA按CPU利用率自动伸缩
> 5. **预算告警**：设置50%/80%/100%三级预算阈值告警，提前发现用量异常
> 6. **Token统计**：将推理日志导出到BigQuery，按天统计Token消耗和费用，识别高成本请求

---

### 中等 高频 - Q5: 什么是Vertex AI Agent Engine？运维Agent的核心架构是什么？

> **参考答案**：
> 1. **Vertex AI Agent Engine** 是GCP上的Agent托管运行环境，提供Agent的注册、部署、并发管理、日志采集等能力
> 2. **核心架构**：基于ReAct（Reasoning + Acting）模式——模型先推理需要调用哪个工具，执行工具获取结果，循环直到得出最终答案
> 3. **工具注册**：运维Agent注册自定义工具（如查询Prometheus、获取K8s日志、执行kubectl），模型通过Function Calling自动选择和调用工具
> 4. **部署方式**：Agent可部署在Vertex AI Agent Engine（托管）或Cloud Run（自托管容器）
>
> **延伸知识**：Google提供Agent Development Kit（ADK）工具包，支持Python和TypeScript开发自定义Agent

---

### 困难 低频 - Q6: 如何在GKE上部署开源大模型（如Gemma 3）的推理服务？有哪些关键配置？

> **参考答案**：
> 1. **GPU节点池**：创建带GPU的节点池（如g2-standard-4 + NVIDIA L4），使用Spot抢占式实例降低成本
> 2. **资源请求**：Pod设置GPU资源限制（nvidia.com/gpu: 1），配置tolerations让Pod调度到GPU节点
> 3. **推理镜像**：使用Google官方推理镜像（gemma-inference），通过环境变量指定模型ID
> 4. **健康检查**：配置readinessProbe和livenessProbe，确保推理服务就绪后才接收流量
> 5. **弹性伸缩**：配置HPA按CPU利用率自动伸缩副本数，配置PDB防止维护时服务中断
> 6. **可观测性**：通过Cloud Operations采集推理服务的延迟、吞吐量、GPU利用率等指标

---

### 基础 中频 - Q7: GCP的LLM运维服务和传统运维监控（Prometheus+Grafana）如何集成？

> **参考答案**：
> 1. **数据采集层**：Prometheus采集K8s指标，Loki采集日志，数据通过Agent的工具接口暴露给LLM
> 2. **分析层**：LLM Agent调用工具获取Prometheus指标和Loki日志，进行多源数据关联分析
> 3. **告警层**：Cloud Monitoring检测异常后触发Agent分析，Agent输出结构化诊断结果
> 4. **闭环层**：Agent生成的修复命令经人工确认后执行，执行结果反馈到监控大盘验证
>
> **延伸知识**：这种"LLM+可观测性"的融合模式是AIOps的核心方向，LLM负责理解和推理，可观测性组件负责数据采集

---

### 中等 低频 - Q8: 如何保证LLM运维服务的安全性？有哪些风险和防护措施？

> **参考答案**：
> 1. **API密钥管理**：使用Secret Manager存储Gemini API Key，通过环境变量注入，禁止硬编码在镜像中
> 2. **IAM权限控制**：Agent的工具调用配置最小权限原则，kubectl工具限制为只读命令白名单
> 3. **输入校验**：对用户输入进行长度限制和敏感词过滤，防止Prompt注入攻击
> 4. **访问控制**：Cloud Run服务关闭公开访问（--no-allow-unauthenticated），配置IAM认证
> 5. **审计日志**：启用Cloud Audit Logs记录所有API调用和Agent工具调用，便于事后追溯
> 6. **数据隔离**：企业敏感日志在发送给LLM前脱敏处理，避免泄露IP地址、密钥等信息
