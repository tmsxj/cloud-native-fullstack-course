# 模块29：K8S故障排查与根因分析

---

## 一、理论核心

### 1.1 K8S故障分层诊断模型

K8S故障排查遵循"从外到内、从控制面到数据面"的分层思路。生产环境中80%的故障可通过标准化排查路径在10分钟内定位根因。

| 层级 | 排查对象 | 核心命令 | 典型故障 |
|------|----------|----------|----------|
| 集群层 | 节点状态、资源容量 | `kubectl get nodes` | 节点NotReady、资源不足 |
| 控制面层 | API Server、etcd、Scheduler | `kubectl get componentstatuses` | 证书过期、etcd脑裂 |
| 网络层 | DNS、Service、Ingress | `kubectl get svc,endpoints` | CoreDNS延迟、Endpoint不同步 |
| 存储层 | PVC、PV、StorageClass | `kubectl get pvc,pv` | 挂载失败、容量不足 |
| 应用层 | Pod、Container、探针 | `kubectl describe pod` | OOMKilled、探针误杀 |

**关键要点**：
- 先确认故障范围：是单个Pod、单个节点还是整个集群
- `kubectl describe`和`kubectl logs`是排查起点，覆盖90%场景
- 控制面故障影响所有工作负载，优先排查

---

### 1.2 Pod生命周期与状态机

Pod从创建到运行经历Pending到ContainerCreating到Running到Terminating等状态，每个状态的异常都有明确排查方向。

| Pod状态 | 含义 | 排查重点 |
|---------|------|----------|
| Pending | 调度中或资源不足 | 节点资源、亲和性、PVC绑定 |
| ContainerCreating | 拉取镜像或创建容器 | 镜像权限、CNI网络、存储挂载 |
| CrashLoopBackOff | 容器反复崩溃 | 应用日志、探针配置、资源限制 |
| ImagePullBackOff | 镜像拉取失败 | 镜像名、仓库权限、网络连通 |
| Terminating | 终止中但未完成 | 优雅关闭、finalizer、volume卸载 |
| Unknown | 节点失联 | kubelet状态、节点网络、磁盘压力 |

**关键要点**：
- `kubectl describe pod`的Events字段是状态转换的完整记录
- CrashLoopBackOff需查看`--previous`日志定位崩溃原因
- Terminating超60秒需检查`terminationGracePeriodSeconds`

---

### 1.3 故障分类与责任边界

生产环境故障按责任边界分为开发侧、运维侧、共同处理三类，不同类型故障的排查思路和协作方式差异很大。

| 分类 | 故障数量 | 典型特征 | 排查主导方 |
|------|----------|----------|------------|
| 开发侧 | 8类 | 应用代码缺陷、服务治理缺失 | 开发主导，运维协助 |
| 运维侧 | 11类 | 基础设施问题、集群组件故障 | 运维主导，开发配合 |
| 共同处理 | 7类 | 资源配置协作、发布流程协作 | 双方协同，SRE牵头 |

**关键要点**：
- 开发侧故障最常见的是内存泄漏、连接池耗尽、优雅关闭缺失
- 运维侧故障最常见的是节点资源争抢、证书过期、CoreDNS延迟
- 共同处理故障最常见的是资源限制不匹配、探针参数不一致、多环境配置混乱

---

### 1.4 日志与事件聚合分析

K8S原生日志分散在节点和容器内，生产环境必须建立中心化日志采集体系才能高效排查跨Pod故障。

| 日志类型 | 位置 | 采集方式 | 保留策略 |
|----------|------|----------|----------|
| 容器标准输出 | `/var/log/containers/` | Fluent Bit/Fluentd | 7-30天 |
| 容器标准错误 | 同上 | 同上 | 同上 |
| kubelet日志 | `/var/log/pods/kube-system_kubelet*` | DaemonSet采集 | 30天 |
| 系统日志 | `/var/log/syslog`或`journalctl` | Node Exporter或rsyslog | 30天 |
| 审计日志 | API Server配置 | 独立存储 | 180天 |

**关键要点**：
- `kubectl logs`仅查看当前运行实例，崩溃后需加`--previous`
- `kubectl get events --sort-by=.lastTimestamp`按时间排序查看集群事件
- 生产环境必须部署日志聚合系统（Loki/EFK），单节点日志无法支撑分布式排查

## 二、实操演练

### 任务1：排查Pod一直处于Pending状态

**任务目标**：定位Pod无法调度的根因并恢复运行

**操作步骤**：

```bash
# 步骤1：查看Pod状态和事件
kubectl describe pod <pod-name> -n <namespace>
# 关注Events字段的Warning事件

# 步骤2：检查Pending原因分类处理
# 情况A：Insufficient cpu/memory
kubectl describe node <node-name> | grep -A5 "Allocated resources"
# 预期输出：显示节点已分配和可分配资源
# 处理：清理低优先级Pod或扩容节点

# 情况B：PersistentVolumeClaim not bound
kubectl get pvc <pvc-name> -n <namespace>
# 预期输出：STATUS为Pending
kubectl describe pvc <pvc-name>
# 预期输出：Events显示Provisioning失败原因
# 处理：检查StorageClass或手动创建PV

# 情况C：no nodes available to schedule pods
kubectl get nodes
# 预期输出：确认节点状态是否为Ready
# 处理：检查节点污点或Pod亲和性配置

# 步骤3：验证Pod是否正常运行
kubectl get pod <pod-name> -n <namespace>
# 预期输出：STATUS为Running，READY为1/1
```

**注意事项**：
- 多Pod同时Pending时优先检查集群资源总量
- 节点磁盘压力（DiskPressure）也会导致Pod无法调度
- 使用`kubectl get events --field-selector reason=FailedScheduling`批量查看调度失败事件

---

### 任务2：排查Pod频繁重启（CrashLoopBackOff）

**任务目标**：定位容器崩溃根因并修复

**操作步骤**：

```bash
# 步骤1：查看Pod重启次数和状态
kubectl get pod <pod-name> -n <namespace>
# 预期输出：RESTARTS列数字持续增加

# 步骤2：查看Pod事件确认重启原因
kubectl describe pod <pod-name> -n <namespace> | grep -A10 "Events"
# 预期输出：显示OOMKilled或Liveness probe failed等事件

# 步骤3：分类排查
# 情况A：OOMKilled
kubectl describe pod <pod-name> | grep "Last State"
# 预期输出：Last State: Terminated, Reason: OOMKilled
kubectl top pod <pod-name> --containers
# 预期输出：内存使用接近limit值
# 处理：调大内存limit或排查内存泄漏

# 情况B：应用崩溃（Exit Code非0）
kubectl logs <pod-name> -n <namespace> --previous
# 预期输出：应用错误堆栈或panic信息
# 处理：根据日志修复代码或配置

# 情况C：Liveness探针失败
kubectl get pod <pod-name> -o yaml | grep -A10 livenessProbe
# 预期输出：显示探针配置参数
# 处理：调整timeoutSeconds或修复探针接口

# 步骤4：验证修复结果
kubectl get pod <pod-name> -n <namespace> -w
# 预期输出：RESTARTS不再增加，STATUS保持Running
```

**注意事项**：
- OOMKilled不一定是内存泄漏，可能是limit设置过小
- `--previous`只能查看上一次崩溃日志，更早的需查看节点日志
- 使用`kubectl debug`可启动临时调试容器深入排查

---

### 任务3：排查服务间调用超时

**任务目标**：定位服务间通信延迟或失败的网络层根因

**操作步骤**：

```bash
# 步骤1：确认服务端Pod状态
kubectl get pods -l app=<service-name> -n <namespace>
# 预期输出：所有Pod状态为Running且READY

# 步骤2：检查Service和Endpoint
kubectl get svc <service-name> -n <namespace>
kubectl get endpoints <service-name> -n <namespace>
# 预期输出：Endpoint列表包含所有Pod IP
# 如果Endpoint缺失：检查Pod标签是否匹配Service选择器

# 步骤3：测试DNS解析
kubectl run -it --rm debug --image=busybox:1.28 -n <namespace> --restart=Never -- nslookup <service-name>
# 预期输出：解析到ClusterIP
# 如果解析失败：检查CoreDNS Pod状态

# 步骤4：测试网络连通性
kubectl exec -it <client-pod> -n <namespace> -- wget -qO- http://<service-name>:<port>/health
# 预期输出：返回HTTP 200
# 如果超时：检查NetworkPolicy是否阻挡

# 步骤5：检查NetworkPolicy
kubectl get networkpolicy -n <namespace>
# 预期输出：确认无策略阻挡客户端访问

# 步骤6：验证修复
kubectl exec -it <client-pod> -- curl -w "@curl-format.txt" -o /dev/null -s http://<service-name>:<port>
# 预期输出：time_total < 100ms
```

**注意事项**：
- 跨命名空间访问需使用FQDN：`service-name.namespace.svc.cluster.local`
- CoreDNS缓存可能导致解析延迟，使用`dig +trace`排查
- 节点conntrack表满会导致DNS包丢失，检查`nf_conntrack_count`

---

### 任务4：排查Ingress访问异常

**任务目标**：定位从外部到集群内服务的访问链路故障

**操作步骤**：

```bash
# 步骤1：检查Ingress资源状态
kubectl get ingress <ingress-name> -n <namespace>
# 预期输出：显示HOSTS和ADDRESS

# 步骤2：检查Ingress控制器Pod
kubectl get pods -n ingress-nginx
# 预期输出：所有Pod状态Running

# 步骤3：查看Ingress控制器日志
kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx --tail=50
# 预期输出：无ERROR级别日志

# 步骤4：检查Ingress配置
kubectl get ingress <ingress-name> -o yaml | grep -A5 "rules:"
# 预期输出：确认host和backend service名称正确

# 步骤5：检查Service后端
kubectl get svc <backend-service> -n <namespace>
kubectl get endpoints <backend-service> -n <namespace>
# 预期输出：Endpoint包含健康Pod IP

# 步骤6：测试访问链路
curl -v -H "Host: <ingress-host>" http://<ingress-ip>/
# 预期输出：HTTP 200，返回业务内容
# 如果502：后端Pod未就绪或崩溃
# 如果503：Service无可用Endpoint
# 如果404：Ingress路径规则不匹配

# 步骤7：验证TLS证书（HTTPS场景）
openssl s_client -connect <ingress-host>:443 -servername <ingress-host> </dev/null 2>/dev/null | openssl x509 -noout -dates
# 预期输出：显示证书有效期
```

**注意事项**：
- Ingress规则变更后需等待控制器reload（通常5-10秒）
- 证书过期需更新Secret并重启Ingress控制器
- 使用`kubectl ingress-nginx`插件可快速检查配置

---

### 任务5：排查节点NotReady状态

**任务目标**：定位节点失联根因并恢复节点

**操作步骤**：

```bash
# 步骤1：查看节点状态和条件
kubectl describe node <node-name> | grep -A20 "Conditions"
# 预期输出：显示Ready、DiskPressure、MemoryPressure等条件

# 步骤2：检查节点kubelet状态
ssh <node-ip> systemctl status kubelet
# 预期输出：Active: active (running)
# 如果inactive：systemctl start kubelet

# 步骤3：检查节点磁盘空间
ssh <node-ip> df -h
# 预期输出：确认/var/lib/docker和/var/log未占满
# 如果磁盘使用率>85%：清理日志或镜像

# 步骤4：检查节点内存
ssh <node-ip> free -h
# 预期输出：确认可用内存>10%
# 如果内存不足：终止非关键进程或扩容节点

# 步骤5：检查节点网络
ssh <node-ip> ping <apiserver-ip>
# 预期输出：网络连通
# 如果不通：检查防火墙或路由配置

# 步骤6：检查kubelet证书
ssh <node-ip> openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -dates
# 预期输出：证书未过期
# 如果过期：执行kubeadm alpha certs renew命令

# 步骤7：验证节点恢复
kubectl get node <node-name>
# 预期输出：STATUS为Ready
```

**注意事项**：
- 节点NotReady超过5分钟，Pod会被驱逐到其他节点
- 磁盘压力是节点NotReady的最常见原因
- 大规模集群中节点证书过期需批量处理

## 三、面试真题

### 基础 高频 - Q1: Pod一直处于Pending状态，如何排查？

> **参考答案**：
> 1. **查看Pod事件**：执行`kubectl describe pod`，重点查看Events字段的Warning事件，常见原因包括Insufficient cpu/memory、PersistentVolumeClaim not bound、no nodes available to schedule pods
> 2. **检查节点资源**：执行`kubectl describe node`查看Allocated resources，确认节点是否有足够CPU和内存
> 3. **检查PVC状态**：如果事件提示PVC问题，执行`kubectl get pvc`和`kubectl describe pvc`确认StorageClass和PV状态
> 4. **检查调度约束**：确认Pod的nodeSelector、亲和性、污点容忍是否匹配节点标签
>
> **延伸知识**：使用`kubectl get events --field-selector reason=FailedScheduling`可批量查看调度失败事件

---

### 基础 高频 - Q2: Pod频繁重启，如何定位根因？

> **参考答案**：
> 1. **确认重启原因**：执行`kubectl describe pod`查看Last State和Events，区分OOMKilled、Error、Liveness probe failed
> 2. **OOMKilled排查**：执行`kubectl top pod`确认内存使用是否接近limit，如果是则调大limit或排查内存泄漏
> 3. **应用崩溃排查**：执行`kubectl logs --previous`查看上一次运行的错误日志，根据堆栈定位代码问题
> 4. **探针失败排查**：检查livenessProbe的timeoutSeconds是否小于接口实际响应时间，或探针接口本身是否异常
>
> **延伸知识**：使用`kubectl debug`可启动临时调试容器，在不重启主容器的情况下深入排查

---

### 中等 高频 - Q3: 服务间调用出现间歇性超时，如何排查？

> **参考答案**：
> 1. **检查服务端状态**：确认后端Pod全部Running且Ready，执行`kubectl get endpoints`确认Endpoint列表完整
> 2. **测试DNS解析**：在客户端Pod执行`nslookup service-name`，确认CoreDNS解析正常，无延迟
> 3. **测试网络连通**：在客户端Pod执行`curl`或`wget`直接访问服务端Pod IP，排除Service层问题
> 4. **检查网络策略**：执行`kubectl get networkpolicy`确认无策略阻挡访问
> 5. **检查CoreDNS**：确认CoreDNS Pod运行正常，考虑启用NodeLocal DNSCache降低解析延迟
>
> **延伸知识**：节点conntrack表满会导致DNS包丢失，需监控`nf_conntrack_count`指标

---

### 中等 中频 - Q4: Ingress访问返回502/503，如何排查？

> **参考答案**：
> 1. **检查Ingress控制器**：确认Ingress控制器Pod全部Running，查看日志无ERROR
> 2. **检查Ingress规则**：确认host、path和backend service名称配置正确
> 3. **检查Service后端**：执行`kubectl get endpoints`确认Endpoint包含健康Pod IP
> 4. **区分错误码**：502表示后端Pod未就绪或崩溃，503表示Service无可用Endpoint，404表示路径规则不匹配
> 5. **检查证书**：HTTPS场景下确认TLS证书未过期，Secret配置正确
>
> **延伸知识**：Nginx Ingress的reload采用fork-exec模式，配置变更期间可能出现短暂中断

---

### 中等 高频 - Q5: 节点状态变为NotReady，如何处理？

> **参考答案**：
> 1. **检查kubelet状态**：SSH到节点执行`systemctl status kubelet`，确认kubelet正常运行
> 2. **检查磁盘空间**：执行`df -h`，确认/var/lib/docker和/var/log未超过85%，磁盘压力会导致节点NotReady
> 3. **检查网络连通**：确认节点与API Server网络连通，无防火墙阻挡
> 4. **检查证书过期**：执行`openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -dates`确认证书有效
> 5. **检查资源压力**：执行`free -h`和`top`确认节点无内存或CPU压力
>
> **延伸知识**：节点NotReady超过5分钟，节点上的Pod会被驱逐到其他节点，需关注业务连续性

---

### 困难 中频 - Q6: 大规模集群中Service Endpoint同步延迟导致流量503，如何排查和解决？

> **参考答案**：
> 1. **确认现象**：新Pod Ready后未及时接收流量，执行`kubectl get endpoints`发现Pod IP未加入Endpoint列表
> 2. **检查Endpoint Controller**：查看Controller日志，发现relist操作延迟增加
> 3. **根因分析**：集群节点数超过50时，Endpoint列表过长，默认`--endpoint-sync-period`（5秒）在高频变更场景下不足
> 4. **短期修复**：调大Endpoint Controller的`--endpoint-sync-period`到10秒
> 5. **长期方案**：升级到K8S 1.21+，启用EndpointSlice替代Endpoint，或拆分大型Service减少Endpoint数量
>
> **延伸知识**：EndpointSlice将Endpoint拆分为多个切片，每个切片最多100个Endpoint，大幅提升性能

---

### 困难 低频 - Q7: etcd集群响应变慢导致整个集群操作延迟，如何排查？

> **参考答案**：
> 1. **检查etcd健康状态**：执行`etcdctl endpoint health`，确认所有节点健康
> 2. **检查磁盘性能**：etcd对磁盘延迟敏感，执行`iostat -x`确认磁盘I/O无瓶颈，必须使用SSD
> 3. **检查etcd日志**：查看是否有`etcdserver: request timed out`错误
> 4. **检查数据量**：执行`etcdctl endpoint status`，确认数据库大小，定期执行defrag
> 5. **检查网络延迟**：确认etcd节点间网络延迟<10ms
>
> **延伸知识**：etcd数据目录必须使用SSD，HDD的延迟会导致etcd性能急剧下降，进而影响整个集群

---

### 困难 高频 - Q8: 讲一个你处理过的K8S生产故障案例（STAR-R法）

> **参考答案**（以内存泄漏导致OOM为例）：
>
> **Situation**：微服务架构，Java Spring Boot应用部署在K8S，日活50万用户，核心交易服务Pod每2-3天OOM重启一次
>
> **Task**：作为SRE工程师，定位内存泄漏根因并彻底修复
>
> **Action**：
> 1. 通过`kubectl describe pod`确认OOMKilled事件，监控显示内存持续增长
> 2. 进入容器执行`jmap -histo`，发现某业务对象实例数异常增长
> 3. 生成堆转储文件，使用MAT工具分析，定位到订单缓存未设置过期策略
> 4. 与开发确认：业务迭代引入缓存但未添加TTL
> 5. 临时调大Pod内存limit争取修复时间
>
> **Result**：开发修复代码添加缓存TTL后，内存从4GB稳定在1.5GB，不再重启
>
> **Reflection**：缓存必须设置过期策略是生产环境铁律。改进措施包括压测增加24小时内存泄漏检测、代码审查增加缓存检查点、生产配置内存使用率>80%告警
>
> **延伸知识**：面试中也可用Endpoint同步延迟或资源限制不匹配的案例，关键是用STAR-R结构讲清楚

### 困难 高频 - Q9: 讲一个运维侧处理过的K8S大规模集群故障案例

> **参考答案**（以Endpoint同步延迟为例）：
>
> **Situation**：K8S集群50+节点，微服务200+，日调用量10亿次，P99延迟要求<100ms。发布新版本时流量间歇性503，持续10-30秒，无明显配置错误日志
>
> **Task**：作为K8S平台工程师，定位503根因，消除发布期间的流量异常
>
> **Action**：
> 1. 发布期间监控显示503错误率上升，但Pod状态全部正常
> 2. 检查Service Endpoint，发现新Pod Ready后未立即加入Endpoint列表
> 3. 查看Endpoint Controller日志，发现relist操作耗时从毫秒级增加到秒级
> 4. 分析根因：集群50+节点导致Endpoint列表过长，默认`--endpoint-sync-period`（5秒）在高频变更场景下不足
> 5. 短期调大`--endpoint-sync-period`到10秒，长期升级到EndpointSlice
>
> **Result**：发布期间503错误消失，P99延迟从200ms恢复到80ms，Endpoint同步延迟从5-10秒降低到1-2秒
>
> **Reflection**：集群规模增长后需重新评估核心组件性能瓶颈。改进措施包括集群超50节点时评估Endpoint性能、发布流程增加灰度观察期、监控Endpoint同步延迟指标
>
> **延伸知识**：EndpointSlice将Endpoint拆分为多个切片，每个切片最多100个Endpoint，是大规模集群的必选方案

### 困难 中频 - Q10: 讲一个开发运维协同处理的K8S故障案例

> **参考答案**（以资源限制与突发流量不匹配为例）：
>
> **Situation**：电商大促系统，Java微服务K8S部署，日常QPS 1000，大促峰值QPS 10000。大促开始10分钟后核心服务Pod频繁重启，订单成功率下降
>
> **Task**：作为SRE工程师，快速恢复服务并建立大促资源保障机制
>
> **Action**：
> 1. 监控显示Pod CPU使用率达到limit（2核）触发限流，随后OOM重启
> 2. 检查资源限制：开发按日常峰值QPS 1000申请了2核CPU、4GB内存
> 3. 检查实际流量：大促峰值QPS 10000，是日常的10倍
> 4. 协作分析：开发按日常峰值申请资源未考虑大促场景，运维按常规阈值设置limit未对齐大促需求，双方均按标准流程操作但缺乏极端场景协同
> 5. 紧急调大资源limit（CPU 4核，内存8GB），启用HPA自动扩容
>
> **Result**：资源调整后服务稳定，订单成功率恢复到99.5%。大促前30分钟成功率下降到85%，调整后恢复
>
> **Reflection**：资源限制配置缺乏场景化考虑是根因。改进措施包括建立资源需求评审机制、开发提供场景化资源需求文档、大促前全链路压测验证资源配置、建立容量规划流程定期review
>
> **延伸知识**：资源限制不是一次性配置，需随业务场景持续优化。VPA（Vertical Pod Autoscaler）可辅助分析资源使用建议
