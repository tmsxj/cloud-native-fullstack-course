## 1. 概述与架构图

### 1.1 为什么需要高级调度？

> 📌 **调度演进**
>
> Kubernetes默认调度器以Pod为单位进行调度，适合无状态微服务。但对于AI训练、大数据、批处理等场景，需要更高级的调度能力。

**默认调度器的局限：**

| 场景 | 默认调度器问题 | 需要的能力 |
|------|---------------|------------|
| **AI训练** | 多Worker可能部分调度导致死锁 | Gang Scheduling (全部或无) |
| **批处理** | 无优先级，高优任务可能等待 | Queue + 抢占 |
| **GPU任务** | 无拓扑感知，性能次优 | GPU拓扑调度 |
| **多租户** | 无资源配额，可能被抢占 | Queue + 权重 |
| **AI Agent** | 无DAG依赖，无法编排 | Agent编排 |

## 2. K8s调度原理

### 2.1 调度流程

```
+================================================================================+
|                    K8s调度流程                                                   |
+================================================================================+
|                                                                                |
|  Pod创建请求                                                                    |
|       |                                                                        |
|       v                                                                        |
|  +-------------------+                                                         |
|  | 1. Informer监听   |  watch Pod事件，发现Pending Pod                         |
|  +---------+---------+                                                         |
|            |                                                                   |
|            v                                                                   |
|  +-------------------+                                                         |
|  | 2. 调度队列        |  按优先级、QoS排序                                      |
|  +---------+---------+                                                         |
|            |                                                                   |
|            v                                                                   |
|  +-------------------+                                                         |
|  | 3. 过滤 (Predicate)|  排除不满足条件的Node                                   |
|  |   - PodFitsResources                                                         |
|  |   - PodMatchNodeSelector                                                    |
|  |   - PodToleratesNodeTaints                                                  |
|  |   - ...                                                                      |
|  +---------+---------+                                                         |
|            |                                                                   |
|            v                                                                   |
|  +-------------------+                                                         |
|  | 4. 打分 (Priority)|  对候选Node打分排序                                      |
|  |   - LeastRequestedPriority (资源最少使用)                                    |
|  |   - BalancedResourceAllocation (资源均衡)                                    |
|  |   - NodeAffinityPriority (亲和性)                                           |
|  |   - ...                                                                      |
|  +---------+---------+                                                         |
|            |                                                                   |
|            v                                                                   |
|  +-------------------+                                                         |
|  | 5. 绑定 (Bind)     |  将Pod绑定到最高分Node                                  |
|  +-------------------+                                                         |
|                                                                                |
+================================================================================+
```

### 2.2 过滤插件 (Filter Plugins)

| 插件名称 | 功能 | 示例 |
|----------|------|------|
| **PodFitsResources** | 检查Node资源是否足够 | CPU/内存/存储 |
| **PodMatchNodeSelector** | 检查NodeSelector匹配 | `nodeSelector: disk: ssd` |
| **PodToleratesNodeTaints** | 检查污点容忍 | GPU节点的污点 |
| **NodePorts** | 检查HostPort冲突 | `hostPort: 8080` |
| **NodeAffinity** | 检查节点亲和性 | `requiredDuringScheduling` |
| **InterPodAffinity** | 检查Pod亲和性 | 与某Pod同节点 |
| **VolumeBinding** | 检查PVC绑定 | 动态存储分配 |
| **VolumeZone** | 检查存储可用区 | 跨区存储限制 |

### 2.3 打分插件 (Score Plugins)

| 插件名称 | 打分逻辑 | 权重 |
|----------|----------|------|
| **LeastRequestedPriority** | 空闲资源越多分数越高 | 1 |
| **MostRequestedPriority** | 已用资源越多分数越高 | 1 |
| **BalancedResourceAllocation** | CPU/内存使用越均衡分数越高 | 1 |
| **NodeAffinityPriority** | 匹配亲和性标签分数高 | 1 |
| **InterPodAffinityPriority** | 满足Pod亲和性分数高 | 1 |
| **ImageLocalityPriority** | 镜像已存在分数高 | 1 |
| **TaintTolerationPriority** | 容忍更多污点分数高 | 1 |

### 3.2 NodeAffinity (节点亲和性)

```yaml
# node-affinity-demo.yaml
# 场景: 优先调度到GPU节点，但非强制

apiVersion: v1
kind: Pod
metadata:
  name: ml-inference-pod
spec:
  affinity:
    nodeAffinity:
      # 硬约束: 必须满足
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/arch
            operator: In
            values:
            - amd64
      # 软约束: 优先满足
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 80  # 权重80
        preference:
          matchExpressions:
          - key: gpu-type
            operator: In
            values:
            - nvidia-a100
      - weight: 20  # 权重20
        preference:
          matchExpressions:
          - key: gpu-type
            operator: In
            values:
            - nvidia-t4
  containers:
  - name: inference
    image: python:3.11-slim
    command: ["python", "-c", "print('Inference running')"]
    resources:
      requests:
        cpu: "500m"
        memory: "512Mi"
```

**亲和性操作符：**

| 操作符 | 说明 | 示例 |
|--------|------|------|
| **In** | 标签值在列表中 | `gpu-type In [a100, t4]` |
| **NotIn** | 标签值不在列表中 | `env NotIn [prod]` |
| **Exists** | 标签存在 | `gpu-type Exists` |
| **DoesNotExist** | 标签不存在 | `gpu-type DoesNotExist` |
| **Gt** | 标签值大于指定值 | `cpu-count Gt 8` |
| **Lt** | 标签值小于指定值 | `memory-gb Lt 64` |

### 3.3 PodAffinity/AntiAffinity (Pod亲和性)

```yaml
# pod-affinity-demo.yaml
# 场景: Web服务与Cache同节点，但Web副本分散到不同节点

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      affinity:
        # Pod亲和性: 与cache同节点 (减少网络延迟)
        podAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchLabels:
                app: cache
            topologyKey: kubernetes.io/hostname  # 同节点
        # Pod反亲和性: web副本分散到不同节点 (高可用)
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchLabels:
                  app: web
              topologyKey: kubernetes.io/hostname
      containers:
      - name: web
        image: nginx:alpine
        ports:
        - containerPort: 80
---
# Cache服务
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cache
spec:
  replicas: 1
  selector:
    matchLabels:
      app: cache
  template:
    metadata:
      labels:
        app: cache
    spec:
      containers:
      - name: redis
        image: redis:7-alpine
        ports:
        - containerPort: 6379
```

### 3.4 Taint/Toleration (污点与容忍)

```yaml
# taint-toleration-demo.yaml
# 场景: GPU节点专用，只有容忍污点的Pod才能调度

# 1. 给GPU节点打污点
# kubectl taint node gpu-node-1 gpu=true:NoSchedule

# 2. 普通Pod无法调度到GPU节点
apiVersion: v1
kind: Pod
metadata:
  name: normal-pod
spec:
  containers:
  - name: app
    image: nginx:alpine
  # 无容忍，无法调度到有gpu=true:NoSchedule污点的节点

---
# 3. GPU任务Pod容忍污点
apiVersion: v1
kind: Pod
metadata:
  name: gpu-training-pod
spec:
  tolerations:
  - key: gpu
    operator: Equal
    value: "true"
    effect: NoSchedule
  nodeSelector:
    gpu-type: nvidia  # 进一步限制到GPU节点
  containers:
  - name: training
    image: pytorch/pytorch:2.0.1-cpu-py311
    command: ["python", "-c", "print('Training on GPU node')"]
```

**污点效果：**

| Effect | 说明 | 行为 |
|--------|------|------|
| **NoSchedule** | 不调度 | 新Pod无法调度，已有Pod不受影响 |
| **PreferNoSchedule** | 尽量不调度 | 软约束，资源紧张时才生效 |
| **NoExecute** | 不执行 | 新Pod无法调度，已有Pod被驱逐 |

```bash
# 污点操作命令
# 添加污点
kubectl taint node <node-name> key=value:effect

# 删除污点
kubectl taint node <node-name> key:effect-

# 查看节点污点
kubectl describe node <node-name> | grep Taints
```

### 4.5 VolcanoJob (批处理任务)

```yaml
# volcano-job-demo.yaml
# 场景: 分布式训练任务，1个Master + 3个Worker

apiVersion: batch.volcano.sh/v1alpha1
kind: Job
metadata:
  name: distributed-training
spec:
  queue: training-queue
  schedulerName: volcano
  maxRetry: 3  # 最大重试次数
  tasks:
  # Master任务
  - name: master
    replicas: 1
    policies:
    - event: TaskCompleted
      action: CompleteJob  # Master完成则整个Job完成
    template:
      spec:
        containers:
        - name: master
          image: python:3.11-slim
          command: ["python", "-c"]
          args:
          - |
            print("Master started")
            import time
            time.sleep(30)
            print("Master completed")
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
  # Worker任务
  - name: worker
    replicas: 3
    policies:
    - event: TaskFailed
      action: RetryTask  # Worker失败则重试
    template:
      spec:
        containers:
        - name: worker
          image: python:3.11-slim
          command: ["python", "-c"]
          args:
          - |
            import os
            print(f"Worker {os.environ.get('VK_TASK_INDEX', 'unknown')} started")
            import time
            time.sleep(60)
            print(f"Worker {os.environ.get('VK_TASK_INDEX', 'unknown')} completed")
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
```

## 6. 调度器扩展 (Scheduler Framework)

### 6.1 Scheduler Framework架构

```
+================================================================================+
|                    Scheduler Framework架构                                       |
+================================================================================+
|                                                                                |
|  +-------------------+                                                         |
|  |   Scheduler       |                                                         |
|  |   Framework       |                                                         |
|  +---------+---------+                                                         |
|            |                                                                   |
|            +------------------+------------------+------------------+          |
|            |                  |                  |                  |          |
|            v                  v                  v                  v          |
|  +----------------+ +----------------+ +----------------+ +----------------+  |
|  | PreFilter      | | Filter        | | PostFilter     | | PreScore       |  |
|  | (预过滤)       | | (过滤)        | | (后过滤)       | | (预打分)       |  |
|  +----------------+ +----------------+ +----------------+ +----------------+  |
|  +----------------+ +----------------+ +----------------+ +----------------+  |
|  | Score          | | NormalizeScore| | PreBind        | | Bind           |  |
|  | (打分)         | | (标准化分数)  | | (预绑定)       | | (绑定)         |  |
|  +----------------+ +----------------+ +----------------+ +----------------+  |
|                                                                                |
|  扩展点:                                                                       |
|  - 每个扩展点可以注册多个插件                                                  |
|  - 插件通过配置启用/禁用                                                       |
|  - 支持自定义插件开发                                                          |
|                                                                                |
+================================================================================+
```

### 6.2 自定义调度插件示例

```go
// custom-scheduler-plugin.go
// 自定义调度插件示例 (Go语言)

package main

import (
	"context"
	"fmt"
	
	"k8s.io/kubernetes/pkg/scheduler/framework"
)

// CustomPlugin 自定义调度插件
type CustomPlugin struct {
	handle framework.Handle
}

// Name 插件名称
func (p *CustomPlugin) Name() string {
	return "CustomPlugin"
}

// Filter 过滤阶段: 检查Node是否满足条件
func (p *CustomPlugin) Filter(
	ctx context.Context,
	state *framework.CycleState,
	pod *v1.Pod,
	nodeInfo *framework.NodeInfo,
) *framework.Status {
	// 示例: 检查Node是否有特定标签
	labels := nodeInfo.Node().Labels
	if _, ok := labels["custom-scheduler/enabled"]; !ok {
		return framework.NewStatus(framework.Unschedulable, "node not enabled for custom scheduler")
	}
	
	return framework.NewStatus(framework.Success, "")
}

// Score 打分阶段: 为Node打分
func (p *CustomPlugin) Score(
	ctx context.Context,
	state *framework.CycleState,
	pod *v1.Pod,
	nodeInfo *framework.NodeInfo,
) (int64, *framework.Status) {
	// 示例: 根据Node的自定义标签打分
	labels := nodeInfo.Node().Labels
	if priority, ok := labels["custom-scheduler/priority"]; ok {
		// 根据优先级标签打分
		if priority == "high" {
			return 100, framework.NewStatus(framework.Success, "")
		} else if priority == "medium" {
			return 50, framework.NewStatus(framework.Success, "")
		}
	}
	
	return 0, framework.NewStatus(framework.Success, "")
}

// ScoreExtensions 返回Score扩展
func (p *CustomPlugin) ScoreExtensions() framework.ScoreExtensions {
	return nil
}

// New 创建插件实例
func New(obj runtime.Object, handle framework.Handle) (framework.Plugin, error) {
	return &CustomPlugin{handle: handle}, nil
}
```

## 8. 故障排查

### 案例1: Pod一直Pending

**现象：**
Pod一直处于Pending状态，无法调度。

**排查步骤：**
```bash
# 1. 查看Pod事件
kubectl describe pod <pod-name> | grep -A 20 Events

# 2. 查看调度器日志
kubectl logs -n kube-system kube-scheduler-<node-name> | grep <pod-name>

# 3. 检查Node资源
kubectl describe node <node-name> | grep -A 10 "Allocated resources"

# 4. 检查污点
kubectl describe node <node-name> | grep Taints

# 5. 检查亲和性
kubectl get pod <pod-name> -o yaml | grep -A 20 affinity
```

**常见原因：**
- 资源不足 (CPU/内存/存储)
- NodeSelector不匹配
- 污点不容忍
- 亲和性不满足
- PVC未绑定

### 案例2: Gang Scheduling死锁

**现象：**
PodGroup中的Pod全部Pending，无法调度。

**排查步骤：**
```bash
# 1. 查看PodGroup状态
kubectl describe podgroup <pg-name>

# 2. 检查minMember和minResources
kubectl get podgroup <pg-name> -o yaml

# 3. 检查队列资源
kubectl describe queue <queue-name>

# 4. 检查Volcano调度器日志
kubectl logs -n volcano-system deployment/volcano-scheduler
```

**解决方案：**
- 减少minMember数量
- 增加队列资源配额
- 检查是否有其他PodGroup占用资源

### 案例3: GPU Pod调度失败

**现象：**
GPU Pod报错 `0/3 nodes are available: 3 Insufficient nvidia.com/gpu`

**排查步骤：**
```bash
# 1. 检查GPU节点
kubectl get nodes -l nvidia.com/gpu.present=true

# 2. 检查GPU资源
kubectl describe node <gpu-node> | grep nvidia.com/gpu

# 3. 检查GPU Operator状态
kubectl get pods -n gpu-operator

# 4. 检查Device Plugin
kubectl logs -n gpu-operator -l app=nvidia-device-plugin-daemonset
```

## 9. 高频面试题

### Q1: K8s调度流程是怎样的？

**答案要点：**
- **Informer监听** → 发现Pending Pod
- **调度队列** → 按优先级排序
- **过滤阶段** → 排除不满足条件的Node
- **打分阶段** → 对候选Node打分排序
- **绑定阶段** → 将Pod绑定到最高分Node
- **扩展点** → Scheduler Framework支持自定义插件

### Q2: NodeAffinity和NodeSelector有什么区别？

**答案要点：**
- **NodeSelector**: 简单标签匹配，只支持`In`操作
- **NodeAffinity**: 支持更丰富的表达式
  - `required`: 硬约束，必须满足
  - `preferred`: 软约束，优先满足
  - 支持`In/NotIn/Exists/DoesNotExist/Gt/Lt`操作符
- **选择建议**: 简单场景用NodeSelector，复杂场景用NodeAffinity

### Q3: 什么是Gang Scheduling？为什么需要它？

**答案要点：**
- **定义**: 将多个Pod作为一个组同时调度，要么全部成功，要么全部等待
- **解决问题**: 
  - AI训练: 多Worker需要同时启动，部分启动会死锁
  - 分布式计算: Spark/Flink需要所有Executor就绪
  - 批处理: 任务有依赖关系
- **实现**: Volcano/Yunikorn通过PodGroup CRD实现
- **对比K8s原生**: K8s原生调度器逐个调度Pod，无法保证同时性

### Q4: Volcano Queue如何实现多租户资源隔离？

**答案要点：**
- **权重分配**: 按weight比例分配资源
- **资源配额**: capability限制队列最大资源
- **保底资源**: deserved保证队列最小资源
- **资源回收**: reclaimable允许空闲资源被其他队列使用
- **抢占机制**: 高优先级队列可抢占低优先级队列资源

### Q5: GPU调度有哪些策略？

**答案要点：**
- **整卡分配**: 独占GPU，适合大模型训练
- **Time-slicing**: 时间片共享，多Pod轮流使用
- **MIG**: 硬件级GPU实例隔离 (需A100/H100)
- **拓扑感知**: NVLink/NUMA拓扑优化
- **选择建议**:
  - 训练: 整卡或MIG
  - 推理: Time-slicing或MIG
  - 多GPU通信: 拓扑感知
