## 1. 概述与架构图

### 1.1 课程目标

本模块是云原生课程的核心，详细讲解 Kubernetes 高可用集群的完整初始化过程。涵盖 3 Master 堆叠 etcd（分布式键值存储）架构、kubeadm-config.yaml 编写、kubeadm init/join 完整命令、etcd 备份恢复脚本、证书续期以及集群健康检查。完成本模块后，你将具备独立搭建生产级高可用 K8s 集群的能力。

```
  堆叠 etcd (Stacked etcd)                外部 etcd (External etcd)
  +-------------------+                   +-------------------+
  | Master-1          |                   | Master-1          |
  | +-------+-------+ |                   | +-------+-------+ |
  | | etcd  |api-svr| |                   | |api-svr|       | |
  | +-------+-------+ |                   | +-------+       | |
  +-------------------+                   +-------------------+
  | Master-2          |                   | etcd-1            |
  | +-------+-------+ |                   | etcd-2            |
  | | etcd  |api-svr| |                   | etcd-3            |
  | +-------+-------+ |                   +-------------------+
  +-------------------+
  | Master-3          |
  | +-------+-------+ |
  | | etcd  |api-svr| |
  | +-------+-------+ |
  +-------------------+

  优点: 部署简单, 节点数少              优点: etcd 独立, 更安全

```

## 2. 理论基础

### 2.1 高可用架构设计

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **堆叠 etcd** | etcd 与控制面组件运行在同一节点 | 中小规模（推荐） |
| **外部 etcd** | etcd 独立部署在专用节点 | 大规模/高安全要求 |
| **负载均衡** | HAProxy + Keepalived 提供 VIP（虚拟 IP） | 所有 HA 方案 |

### 2.2 kubeadm-config.yaml 关键配置

| 配置项 | 说明 |
|--------|------|
| `apiServer.extraArgs` | API Server 额外参数 |
| `apiServer.certSANs` | API Server 证书 SAN（主题备用名称）列表 |
| `controlPlaneEndpoint` | 控制面统一入口（VIP 或 LB，负载均衡器） |
| `etcd.local` | 堆叠 etcd 配置 |
| `networking.podSubnet` | Pod 网络 CIDR（网段） |
| `networking.serviceSubnet` | Service 网络 CIDR（网段） |
| `imageRepository` | 镜像仓库地址 |

### 2.3 证书体系

| 证书 | 签发者 | 有效期 | 用途 |
|------|--------|--------|------|
| CA | 自签名 | 10 年 | 根证书，签发所有其他证书 |
| API Server | CA | 1 年 | API Server TLS（传输层安全） |
| etcd Server | etcd CA | 1 年 | etcd 服务端认证 |
| etcd Peer | etcd CA | 1 年 | etcd 节点间认证 |
| Front Proxy | FP CA | 1 年 | 反向代理认证 |
| SA | SA Key | 1 年 | ServiceAccount JWT（令牌） |

## 3. 部署实战

### 3.1 部署 HAProxy 负载均衡器

在 192.168.1.50（或任一独立节点）上部署。

#### 3.1.1 安装 HAProxy

```bash
apt-get update && apt-get install -y haproxy
```

#### 3.1.2 配置 HAProxy

```bash
cat > /etc/haproxy/haproxy.cfg << 'EOF'
global
    log /dev/log local0
    log /dev/log local1 notice
    daemon  # 后台运行
    maxconn 2000  # 最大连接数

defaults
    log     global
    mode    tcp  # 四层 TCP 代理
    option  tcplog
    option  dontlognull
    retries 3  # 失败重试次数
    timeout connect 5000  # 连接超时 5 秒
    timeout client  50000  # 客户端超时 50 秒
    timeout server  50000  # 服务端超时 50 秒

frontend k8s-api
    bind *:6443  # 监听 K8s API 端口
    mode tcp
    option tcplog
    default_backend k8s-api-backend

backend k8s-api-backend
    mode tcp
    option tcp-check  # TCP 健康检查
    balance roundrobin  # 轮询负载均衡
    default-server inter 10s downinter 5s rise 2 fall 2 slowstart 60s maxconn 250 maxqueue 256 weight 100
    server k8s-master1 192.168.1.51:6443 check  # Master-1 后端
    server k8s-master2 192.168.1.52:6443 check  # Master-2 后端
    server k8s-master3 192.168.1.53:6443 check  # Master-3 后端
EOF

systemctl enable --now haproxy
systemctl status haproxy
```

#### 3.1.3 验证 HAProxy

```bash
# 检查监听端口
ss -tlnp | grep 6443
# 预期: LISTEN 0 128 *:6443 *:*

# 测试后端连通性
curl -k https://127.0.0.1:6443
# 预期返回 401 (因为还没有 API Server)
```

### 3.2 准备 kubeadm-config.yaml

在 Master-1（192.168.1.51）上创建配置文件。

```bash
cat > /root/kubeadm-config.yaml << 'EOF'
---
apiVersion: kubeadm.k8s.io/v1beta3  # kubeadm API 版本
kind: InitConfiguration  # 初始化配置
nodeRegistration:
  criSocket: unix:///run/containerd/containerd.sock  # CRI 套接字路径
  kubeletExtraArgs:
    cgroup-driver: "systemd"  # cgroup 驱动，须与 containerd 一致
  name: k8s-master1  # 节点名称
  taints:
    - key: "node-role.kubernetes.io/control-plane"  # 控制面污点
      effect: "NoSchedule"  # 不调度普通 Pod

---
apiVersion: kubeadm.k8s.io/v1beta3
kind: ClusterConfiguration  # 集群配置
kubernetesVersion: "v1.28.15"  # K8s 版本
controlPlaneEndpoint: "192.168.1.50:6443"  # HA 负载均衡地址
imageRepository: "192.168.1.61/registry.k8s.io"  # 镜像仓库（Harbor）
certificatesDir: "/etc/kubernetes/pki"  # 证书存储目录
clusterName: "kubernetes"  # 集群名称

apiServer:
  extraArgs:
    authorization-mode: "Node,RBAC"  # 认证模式：节点 + RBAC
    enable-admission-plugins: "NodeRestriction,PodSecurity,ServiceAccount"  # 准入控制器
    audit-log-path: "/var/log/kubernetes/audit.log"  # 审计日志路径
    audit-log-maxage: "30"  # 审计日志保留天数
    audit-log-maxbackup: "10"  # 审计日志最大备份数
    audit-log-maxsize: "100"  # 审计日志最大大小（MB）
  certSANs:  # 证书 SAN（所有可能的访问地址）
    - "192.168.1.50"  # VIP/LB
    - "192.168.1.51"  # Master-1
    - "192.168.1.52"  # Master-2
    - "192.168.1.53"  # Master-3
    - "192.168.1.54"  # Worker-1
    - "192.168.1.55"  # Worker-2
    - "127.0.0.1"  # 本地回环
    - "kubernetes"
    - "kubernetes.default"
    - "kubernetes.default.svc"
    - "kubernetes.default.svc.cluster.local"

controllerManager:
  extraArgs:
    bind-address: "0.0.0.0"  # 监听所有地址
    cluster-signing-duration: "87600h"  # 证书签名有效期（10 年）

scheduler:
  extraArgs:
    bind-address: "0.0.0.0"

etcd:
  local:
    dataDir: "/data/etcd"  # etcd 数据目录
    serverCertSANs:  # etcd 服务端证书 SAN
      - "k8s-master1"
      - "192.168.1.51"
      - "127.0.0.1"
    peerCertSANs:  # etcd 节点间通信证书 SAN
      - "k8s-master1"
      - "192.168.1.51"
      - "127.0.0.1"

networking:
  podSubnet: "10.244.0.0/16"  # Pod 网络 CIDR（与 Calico 一致）
  serviceSubnet: "10.96.0.0/12"  # Service 网络 CIDR
  dnsDomain: "cluster.local"  # 集群 DNS 域

---
apiVersion: kubeadm.k8s.io/v1beta3
kind: JoinConfiguration  # 节点加入配置
nodeRegistration:
  criSocket: unix:///run/containerd/containerd.sock
  kubeletExtraArgs:
    cgroup-driver: "systemd"
discovery:
  bootstrapToken:
    apiServerEndpoint: "192.168.1.50:6443"  # API Server 地址
    unsafeSkipCAVerification: true  # 跳过 CA 证书验证
EOF
```

### 3.8 etcd 备份恢复

#### 3.8.1 自动备份脚本

```bash
cat > /root/etcd-backup.sh << 'SCRIPT'
#!/bin/bash
# etcd 自动备份脚本
# 放置在所有 Master 节点上，通过 crontab 定时执行

set -euo pipefail

ETCDCTL_API=3  # 使用 etcd v3 API
ETCD_CERT_DIR="/etc/kubernetes/pki/etcd"  # etcd 证书目录
BACKUP_DIR="/data/etcd-backup"  # 备份存储目录
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/etcd-backup-${DATE}.db"
RETENTION_DAYS=7  # 备份保留天数

mkdir -p ${BACKUP_DIR}

# 获取本机 IP
LOCAL_IP=$(hostname -I | awk '{print $1}')
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Starting etcd backup on ${LOCAL_IP}"

# 执行快照备份
ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=${ETCD_CERT_DIR}/ca.crt \
    --cert=${ETCD_CERT_DIR}/server.crt \
    --key=${ETCD_CERT_DIR}/server.key \
    snapshot save ${BACKUP_FILE}

# 验证备份
SNAPSHOT_STATUS=$(ETCDCTL_API=3 etcdctl \
    --write-out=table \
    snapshot status ${BACKUP_FILE} 2>/dev/null)

if [ $? -eq 0 ]; then
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup successful"
    echo "${SNAPSHOT_STATUS}"
else
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup verification failed!"
    exit 1
fi

# 压缩备份
gzip ${BACKUP_FILE}
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Compressed: ${BACKUP_FILE}.gz"

# 清理过期备份
DELETED=$(find ${BACKUP_DIR} -name "etcd-backup-*.db.gz" -mtime +${RETENTION_DAYS} -delete -print)
if [ -n "${DELETED}" ]; then
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Deleted old backups: ${DELETED}"
fi

# 显示备份大小
BACKUP_SIZE=$(du -sh ${BACKUP_FILE}.gz | awk '{print $1}')
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup size: ${BACKUP_SIZE}"
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup completed: ${BACKUP_FILE}.gz"
SCRIPT

chmod +x /root/etcd-backup.sh
```

#### 3.8.2 设置定时备份

```bash
# 每小时备份一次
echo "0 * * * * root /root/etcd-backup.sh >> /var/log/etcd-backup.log 2>&1" \
    > /etc/cron.d/etcd-backup
```

#### 3.8.3 etcd 恢复演练

```bash
#!/bin/bash
# etcd 恢复脚本（在 Master-1 上执行）

ETCDCTL_API=3
ETCD_CERT_DIR="/etc/kubernetes/pki/etcd"
BACKUP_FILE="/data/etcd-backup/etcd-backup-20260524_100000.db.gz"
RESTORE_DIR="/var/lib/etcd-restore"

# 1. 停止 kubelet（在所有 Master 节点上执行）
echo "Step 1: Stopping kubelet on all masters..."
for master in 51 52 53; do
    ssh root@192.168.1.${master} "systemctl stop kubelet"
done

# 2. 停止所有静态 Pod
echo "Step 2: Stopping static pods..."
for master in 51 52 53; do
    ssh root@192.168.1.${master} "
        crictl stopp \$(crictl pods -q --label io.kubernetes.pod.name=etcd-k8s-master\$(echo ${master} | tail -c 1))
        crictl stopp \$(crictl pods -q --label io.kubernetes.pod.name=kube-apiserver-k8s-master\$(echo ${master} | tail -c 1))
    "
done

# 3. 在 Master-1 上恢复
echo "Step 3: Restoring etcd from backup..."
ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=${ETCD_CERT_DIR}/ca.crt \
    --cert=${ETCD_CERT_DIR}/server.crt \
    --key=${ETCD_CERT_DIR}/server.key \
    snapshot restore ${BACKUP_FILE} \
    --data-dir=${RESTORE_DIR}

# 4. 替换 etcd 数据目录
echo "Step 4: Replacing etcd data directory..."
mv /data/etcd /var/lib/etcd.bak.$(date +%s)
mv ${RESTORE_DIR} /data/etcd
chown -R etcd:etcd /data/etcd

# 5. 重启所有 Master 的 kubelet
echo "Step 5: Starting kubelet on all masters..."
for master in 51 52 53; do
    ssh root@192.168.1.${master} "systemctl start kubelet"
done

# 6. 验证
echo "Step 6: Verifying cluster recovery..."
sleep 30
kubectl get nodes
kubectl get pods -n kube-system
```

## 5. etcd备份与恢复策略

### 5.1 etcd数据重要性说明

etcd是Kubernetes集群的核心数据存储（基于 Raft 共识算法），保存了集群的所有状态信息：

| 数据类型 | 说明 | 丢失影响 |
|---------|------|---------|
| **资源对象** | Pod、Deployment、Service等 | 所有应用配置丢失 |
| **集群配置** | ConfigMap（配置映射）、Secret（密钥）、RBAC（角色访问控制） | 系统配置丢失 |
| **节点信息** | Node注册信息、标签污点 | 节点无法识别 |
| **网络配置** | CNI配置、Service/Endpoint | 网络通信中断 |
| **事件数据** | 集群事件、审计日志 | 故障排查困难 |

**etcd数据损坏场景：**
- 磁盘故障或文件系统损坏
- 意外删除etcd数据目录
- etcd版本升级失败
- 集群脑裂导致数据不一致
- 恶意攻击或误操作

### 5.3 定时备份脚本(etcdctl snapshot)

#### 5.3.1 基础备份脚本

```bash
cat > /root/etcd-backup.sh << 'SCRIPT'
#!/bin/bash
# etcd 自动备份脚本
# 放置在所有 Master 节点上，通过 crontab 定时执行

set -euo pipefail

ETCDCTL_API=3
ETCD_CERT_DIR="/etc/kubernetes/pki/etcd"
BACKUP_DIR="/data/etcd-backup"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/etcd-backup-${DATE}.db"
RETENTION_DAYS=7

mkdir -p ${BACKUP_DIR}

# 获取本机 IP
LOCAL_IP=$(hostname -I | awk '{print $1}')
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Starting etcd backup on ${LOCAL_IP}"

# 执行快照备份
ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=${ETCD_CERT_DIR}/ca.crt \
    --cert=${ETCD_CERT_DIR}/server.crt \
    --key=${ETCD_CERT_DIR}/server.key \
    snapshot save ${BACKUP_FILE}

# 验证备份
SNAPSHOT_STATUS=$(ETCDCTL_API=3 etcdctl \
    --write-out=table \
    snapshot status ${BACKUP_FILE} 2>/dev/null)

if [ $? -eq 0 ]; then
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup successful"
    echo "${SNAPSHOT_STATUS}"
else
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup verification failed!"
    exit 1
fi

# 压缩备份
gzip ${BACKUP_FILE}
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Compressed: ${BACKUP_FILE}.gz"

# 清理过期备份
DELETED=$(find ${BACKUP_DIR} -name "etcd-backup-*.db.gz" -mtime +${RETENTION_DAYS} -delete -print)
if [ -n "${DELETED}" ]; then
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Deleted old backups: ${DELETED}"
fi

# 显示备份大小
BACKUP_SIZE=$(du -sh ${BACKUP_FILE}.gz | awk '{print $1}')
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup size: ${BACKUP_SIZE}"
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup completed: ${BACKUP_FILE}.gz"
SCRIPT

chmod +x /root/etcd-backup.sh
```

#### 5.3.2 增强版备份脚本（含监控告警）

```bash
cat > /root/etcd-backup-enhanced.sh << 'SCRIPT'
#!/bin/bash
# etcd 增强版备份脚本 - 含监控告警功能

set -euo pipefail

ETCDCTL_API=3
ETCD_CERT_DIR="/etc/kubernetes/pki/etcd"
BACKUP_DIR="/data/etcd-backup"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/etcd-backup-${DATE}.db"
RETENTION_DAYS=7
LOG_FILE="/var/log/etcd-backup.log"
ALERT_WEBHOOK=""  # 配置告警Webhook地址

# 日志函数
log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a ${LOG_FILE}
}

# 告警函数
send_alert() {
    local message="$1"
    log "ALERT: ${message}"
    if [ -n "${ALERT_WEBHOOK}" ]; then
        curl -s -X POST -H "Content-Type: application/json" \
            -d "{\"text\":\"etcd备份告警: ${message}\"}" \
            ${ALERT_WEBHOOK} >/dev/null 2>&1 || true
    fi
}

mkdir -p ${BACKUP_DIR}

# 获取本机 IP
LOCAL_IP=$(hostname -I | awk '{print $1}')
log "Starting etcd backup on ${LOCAL_IP}"

# 检查etcd健康状态
HEALTH_OUTPUT=$(ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=${ETCD_CERT_DIR}/ca.crt \
    --cert=${ETCD_CERT_DIR}/server.crt \
    --key=${ETCD_CERT_DIR}/server.key \
    endpoint health 2>&1) || {
    send_alert "etcd健康检查失败: ${HEALTH_OUTPUT}"
    exit 1
}
log "etcd health check passed"

# 执行快照备份
if ! ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=${ETCD_CERT_DIR}/ca.crt \
    --cert=${ETCD_CERT_DIR}/server.crt \
    --key=${ETCD_CERT_DIR}/server.key \
    snapshot save ${BACKUP_FILE} 2>&1 | tee -a ${LOG_FILE}; then
    send_alert "etcd备份失败: 无法创建快照"
    exit 1
fi

# 验证备份完整性
SNAPSHOT_HASH=$(ETCDCTL_API=3 etcdctl \
    --write-out=table \
    snapshot status ${BACKUP_FILE} 2>/dev/null | tail -1 | awk '{print $NF}')

if [ -z "${SNAPSHOT_HASH}" ]; then
    send_alert "etcd备份验证失败: 无法获取快照状态"
    rm -f ${BACKUP_FILE}
    exit 1
fi

log "Backup hash: ${SNAPSHOT_HASH}"

# 压缩备份
gzip ${BACKUP_FILE}
BACKUP_FILE="${BACKUP_FILE}.gz"

# 检查备份文件大小
BACKUP_SIZE=$(stat -c%s "${BACKUP_FILE}" 2>/dev/null || stat -f%z "${BACKUP_FILE}")
if [ ${BACKUP_SIZE} -lt 1024 ]; then
    send_alert "etcd备份文件异常: 文件过小 (${BACKUP_SIZE} bytes)"
    rm -f ${BACKUP_FILE}
    exit 1
fi

log "Backup size: $(du -sh ${BACKUP_FILE} | awk '{print $1}')"

# 生成备份校验文件
sha256sum ${BACKUP_FILE} > ${BACKUP_FILE}.sha256
log "Backup checksum generated"

# 清理过期备份
DELETED=$(find ${BACKUP_DIR} -name "etcd-backup-*.db.gz" -mtime +${RETENTION_DAYS} -delete -print)
if [ -n "${DELETED}" ]; then
    log "Deleted old backups: ${DELETED}"
fi

# 统计备份信息
BACKUP_COUNT=$(find ${BACKUP_DIR} -name "etcd-backup-*.db.gz" | wc -l)
BACKUP_TOTAL_SIZE=$(du -sh ${BACKUP_DIR} | awk '{print $1}')
log "Backup completed: ${BACKUP_FILE}"
log "Total backups: ${BACKUP_COUNT}, Total size: ${BACKUP_TOTAL_SIZE}"
SCRIPT

chmod +x /root/etcd-backup-enhanced.sh
```

```bash
#!/bin/bash
# etcd 恢复脚本（在 Master-1 上执行）

ETCDCTL_API=3
ETCD_CERT_DIR="/etc/kubernetes/pki/etcd"
BACKUP_FILE="/data/etcd-backup/etcd-backup-20260524_100000.db.gz"
RESTORE_DIR="/var/lib/etcd-restore"

# 1. 停止 kubelet（在所有 Master 节点上执行）
echo "Step 1: Stopping kubelet on all masters..."
for master in 51 52 53; do
    ssh root@192.168.1.${master} "systemctl stop kubelet"
done

# 2. 停止所有静态 Pod
echo "Step 2: Stopping static pods..."
for master in 51 52 53; do
    ssh root@192.168.1.${master} "
        crictl stopp \$(crictl pods -q --label io.kubernetes.pod.name=etcd-k8s-master\$(echo ${master} | tail -c 1))
        crictl stopp \$(crictl pods -q --label io.kubernetes.pod.name=kube-apiserver-k8s-master\$(echo ${master} | tail -c 1))
    "
done

# 3. 在 Master-1 上恢复
echo "Step 3: Restoring etcd from backup..."
ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=${ETCD_CERT_DIR}/ca.crt \
    --cert=${ETCD_CERT_DIR}/server.crt \
    --key=${ETCD_CERT_DIR}/server.key \
    snapshot restore ${BACKUP_FILE} \
    --data-dir=${RESTORE_DIR}

# 4. 替换 etcd 数据目录
echo "Step 4: Replacing etcd data directory..."
mv /data/etcd /var/lib/etcd.bak.$(date +%s)
mv ${RESTORE_DIR} /data/etcd
chown -R etcd:etcd /data/etcd

# 5. 重启所有 Master 的 kubelet
echo "Step 5: Starting kubelet on all masters..."
for master in 51 52 53; do
    ssh root@192.168.1.${master} "systemctl start kubelet"
done

# 6. 验证
echo "Step 6: Verifying cluster recovery..."
sleep 30
kubectl get nodes
kubectl get pods -n kube-system
```

```bash
        if [ -n "$EXPIRY" ]; then
```

```bash
cat > /root/cert-auto-renew.sh << 'SCRIPT'
#!/bin/bash
# Kubernetes证书自动轮换脚本

set -euo pipefail

LOG_FILE="/var/log/cert-renew.log"
BACKUP_DIR="/root/pki-backup-$(date +%Y%m%d_%H%M%S)"
WARNING_DAYS=30
ALERT_WEBHOOK=""

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a ${LOG_FILE}
}

send_alert() {
    local message="$1"
    log "ALERT: ${message}"
    if [ -n "${ALERT_WEBHOOK}" ]; then
        curl -s -X POST -H "Content-Type: application/json" \
            -d "{\"text\":\"证书轮换: ${message}\"}" \
            ${ALERT_WEBHOOK} >/dev/null 2>&1 || true
    fi
}

# 检查是否需要轮换
NEED_RENEW=false
NOW=$(date +%s)

for cert in /etc/kubernetes/pki/*.crt /etc/kubernetes/pki/etcd/*.crt; do
    if [ -f "$cert" ]; then
        EXPIRY=$(openssl x509 -in "$cert" -noout -enddate 2>/dev/null | cut -d= -f2)
        if [ -n "$EXPIRY" ]; then
            EXPIRY_EPOCH=$(date -d "$EXPIRY" +%s)

            if [ $DAYS_REMAINING -lt $WARNING_DAYS ]; then
                log "Certificate $(basename $cert) expires in ${DAYS_REMAINING} days, need renewal"
                NEED_RENEW=true
            fi
        fi
    fi
done

if [ "$NEED_RENEW" = false ]; then
    log "No certificates need renewal"
    exit 0
fi

log "Starting certificate renewal process..."

# 1. 备份现有证书
log "Backing up certificates to ${BACKUP_DIR}..."
mkdir -p ${BACKUP_DIR}
cp -r /etc/kubernetes/pki/* ${BACKUP_DIR}/
cp /etc/kubernetes/*.conf ${BACKUP_DIR}/

# 2. 执行证书续期
log "Renewing certificates..."
if kubeadm certs renew all 2>&1 | tee -a ${LOG_FILE}; then
    log "Certificate renewal successful"
else
    log "ERROR: Certificate renewal failed"
    send_alert "证书续期失败，请检查日志"
    exit 1
fi

# 3. 更新kubeconfig
log "Updating kubeconfig..."
cp /etc/kubernetes/admin.conf $HOME/.kube/config
chown $(id -u):$(id -g) $HOME/.kube/config

# 4. 重启kubelet
log "Restarting kubelet..."
systemctl restart kubelet

# 5. 等待Pod重启
log "Waiting for static pods to restart..."
sleep 30

# 6. 验证
log "Verifying certificate renewal..."
kubeadm certs check-expiration | head -20

# 7. 验证集群状态
if kubectl get nodes >/dev/null 2>&1; then
    log "Cluster is healthy after certificate renewal"
    send_alert "证书续期成功，集群状态正常"
else
    log "WARNING: Cluster may have issues after renewal"
    send_alert "证书续期完成，但集群状态异常，请检查"
fi

log "Certificate renewal process completed"
SCRIPT

chmod +x /root/cert-auto-renew.sh
```

### 7.6 定期备份验证

#### 7.6.1 备份验证脚本

```bash
cat > /root/backup-validation.sh << 'SCRIPT'
#!/bin/bash
# 定期备份验证脚本

set -euo pipefail

LOG_FILE="/var/log/backup-validation.log"
ALERT_WEBHOOK=""

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a ${LOG_FILE}
}

send_alert() {
    local message="$1"
    log "ALERT: ${message}"
    if [ -n "${ALERT_WEBHOOK}" ]; then
        curl -s -X POST -H "Content-Type: application/json" \
            -d "{\"text\":\"备份验证: ${message}\"}" \
            ${ALERT_WEBHOOK} >/dev/null 2>&1 || true
    fi
}

# 1. 检查最近的备份
log "Checking recent backups..."
LATEST_BACKUP=$(velero backup get --output json | jq -r '.items | sort_by(.status.completionTimestamp) | last | .metadata.name')

if [ -z "$LATEST_BACKUP" ] || [ "$LATEST_BACKUP" = "null" ]; then
    send_alert "没有找到有效的备份"
    exit 1
fi

log "Latest backup: $LATEST_BACKUP"

# 2. 检查备份状态
BACKUP_STATUS=$(velero backup describe $LATEST_BACKUP | grep "Phase:" | awk '{print $2}')
if [ "$BACKUP_STATUS" != "Completed" ]; then
    send_alert "最新备份 $LATEST_BACKUP 状态异常: $BACKUP_STATUS"
    exit 1
fi

log "Backup status: $BACKUP_STATUS"

# 3. 检查备份年龄
BACKUP_TIME=$(velero backup describe $LATEST_BACKUP | grep "Started:" | sed 's/Started:[[:space:]]*//')
BACKUP_EPOCH=$(date -d "$BACKUP_TIME" +%s)
NOW=$(date +%s)
AGE_HOURS=$(( (NOW - BACKUP_EPOCH) / 3600 ))

if [ $AGE_HOURS -gt 25 ]; then
    send_alert "最新备份已超过25小时: $AGE_HOURS 小时"
fi

log "Backup age: $AGE_HOURS hours"

# 4. 验证备份完整性
log "Validating backup integrity..."
if ! velero backup describe $LATEST_BACKUP --details | grep -q "Backup Item Operations"; then
    log "WARNING: Backup may not contain all items"
fi

# 5. 检查存储可访问性
log "Checking backup storage accessibility..."
BACKUP_LOCATION=$(velero backup-location get --output json | jq -r '.items[0].status.phase')
if [ "$BACKUP_LOCATION" != "Available" ]; then
    send_alert "备份存储不可访问: $BACKUP_LOCATION"
    exit 1
fi

log "Backup storage is accessible"

# 6. 检查快照状态
SNAPSHOT_COUNT=$(velero backup describe $LATEST_BACKUP --details | grep -c "Snapshot" || true)
log "Snapshots in backup: $SNAPSHOT_COUNT"

log "Backup validation completed successfully"
SCRIPT

chmod +x /root/backup-validation.sh

# 每周执行一次验证
echo "0 2 * * 0 root /root/backup-validation.sh >> /var/log/backup-validation.log 2>&1" \
    > /etc/cron.d/backup-validation
```

## 10. 高频面试题

### Q1: Kubernetes 高可用架构有哪几种？各有什么优缺点？ [难度: 高]

**答案：** Kubernetes 高可用架构主要有两种：堆叠 etcd（Stacked etcd）和外部 etcd（External etcd）。堆叠 etcd 架构中，etcd 与 API Server 等控制面组件运行在同一组 Master 节点上，每个 Master 运行一个 etcd 成员，形成 3 节点或 5 节点的 etcd 集群。优点是部署简单，节点数量少（3 个 Master 即可），运维成本低；缺点是耦合度高，etcd 和 API Server 共享资源（CPU/内存/磁盘），可能互相影响，且 Master 节点故障同时影响 etcd 和 API Server。外部 etcd 架构中，etcd 独立部署在专用节点上（通常 3 或 5 个），与 Master 节点分离。优点是隔离性好，etcd 有独立资源，更安全，故障域独立；缺点是需要更多节点（3 Master + 3 etcd = 6 节点），部署和运维复杂度更高。对于中小规模集群（< 1000 节点），推荐使用堆叠 etcd 架构；对于大规模或高安全要求场景，推荐外部 etcd。无论哪种架构，都需要前置负载均衡器（HAProxy + Keepalived）提供统一的 API Server 入口。

### Q2: kubeadm init 的完整流程是什么？--upload-certs 参数的作用？ [难度: 中]

**答案：** kubeadm init 按顺序执行以下 Phase：Preflight Check（检查系统环境）、Generate Certificates（生成 CA 和所有组件证书）、Generate KubeConfig（生成 admin/kubelet/controller-manager 的 kubeconfig）、Generate Static Pod Manifests（在 /etc/kubernetes/manifests/ 生成 etcd/API Server/Scheduler/Controller Manager 的 YAML）、Upload Config（上传 kubeadm-config 和 kubelet-config 到 ConfigMap）、Bootstrap Token（创建节点加入令牌）、Upload Certs（上传控制面证书到 Secret）、Install CNI（等待 CNI 插件就绪）。`--upload-certs` 参数的作用是将控制面证书（ca.crt、sa.pub、sa.key、front-proxy-ca.crt、etcd-ca.crt）上传到集群的 `kubeadm-certs` Secret 中，并生成一个解密密钥（certificate-key）。这样其他 Master 节点在执行 `kubeadm join --control-plane` 时，可以通过这个密钥从集群中下载证书，而不需要手动从第一个 Master 复制证书文件。certificate-key 默认 2 小时过期。

### Q3: etcd 备份恢复的完整步骤是什么？有哪些注意事项？ [难度: 高]

**答案：** etcd 备份使用 `etcdctl snapshot save` 命令，该命令通过 gRPC 协议连接 etcd Server，获取当前数据状态的一致性快照。备份命令需要指定 CA 证书、服务端证书和密钥进行 TLS 认证。恢复步骤为：首先停止所有 Master 节点的 kubelet（这会停止所有静态 Pod）；然后使用 `etcdctl snapshot restore` 将快照恢复到一个新目录；接着替换原有的 etcd 数据目录（建议先备份旧目录）；最后重启 kubelet。注意事项：第一，恢复操作会改变 etcd 的集群 ID，因此必须同时在所有 etcd 成员上恢复同一个快照，不能只恢复单个成员；第二，恢复后所有 etcd 成员的数据会回退到备份时间点，备份之后的数据变更会丢失；第三，恢复前必须确认快照文件完整（使用 `etcdctl snapshot status` 验证）；第四，生产环境建议定期验证备份的可恢复性；第五，备份文件应存储到集群外部的存储系统（S3/NFS），避免与集群同时故障。

### Q4: Kubernetes 证书体系是怎样的？如何进行证书续期？ [难度: 高]

**答案：** Kubernetes 使用 PKI 体系，由根 CA 签发所有组件证书。证书分为三类：集群 CA（ca.crt/ca.key）签发 API Server 证书、kubelet 客户端证书、前端代理证书；etcd CA（etcd/ca.crt/ca.key）签发 etcd Server 和 Peer 证书；ServiceAccount 使用独立的 RSA 密钥对签发 JWT Token。K8s v1.15+ 的 kubeadm 引入了自动证书轮转：kubelet 会在客户端证书剩余有效期不足 80% 时自动向 API Server 提交 CSR，由 controller-manager 自动审批和签发。但控制面证书（API Server、etcd 等）不会自动续期，需要手动执行 `kubeadm certs renew all`。续期步骤为：执行续期命令 -> 更新 kubeconfig 文件 -> 重启 kubelet（自动重启静态 Pod）。如果集群使用外部 etcd，需要单独续期 etcd 证书。证书默认有效期 1 年，可以通过 `--cluster-signing-duration` 参数调整（最大 10 年）。可以使用 `kubeadm certs check-expiration` 查看所有证书的有效期。

### Q5: controlPlaneEndpoint 的作用是什么？为什么 HA 集群必须配置？ [难度: 中]

**答案：** controlPlaneEndpoint 是 Kubernetes 高可用集群的核心配置，指定了控制面的统一访问地址。这个地址通常指向一个负载均衡器（如 HAProxy）的 VIP 或域名。配置此参数后，kubeadm 会将此地址写入所有 kubeconfig 文件（admin.conf、kubelet.conf、controller-manager.conf、scheduler.conf）和 API Server 证书的 SAN 列表中。所有组件（包括 Worker 节点的 kubelet、其他 Master 节点、kubectl 客户端）都通过这个地址与 API Server 通信。如果不配置此参数，kubeadm 会使用第一个 Master 节点的 IP 作为 API Server 地址，导致所有请求都发送到单个 Master，无法实现负载均衡和高可用。当该 Master 故障时，整个集群将不可访问。在 HA 架构中，负载均衡器将请求分发到所有健康的 Master 节点，当某个 Master 故障时自动剔除，保证服务持续可用。

### Q6: HAProxy 在 K8s HA 架构中的作用是什么？如何配置？ [难度: 中]

**答案：** HAProxy 在 K8s HA 架构中作为四层（TCP）负载均衡器，为 API Server 提供统一的入口和流量分发。所有 K8s 组件（kubelet、kubectl、Controller Manager、Scheduler）通过 HAProxy 的 VIP（如 192.168.1.50:6443）访问 API Server，HAProxy 将请求轮询分发到后端的 3 个 Master 节点（192.168.1.51-53:6443）。配置要点：使用 TCP 模式（mode tcp），因为 API Server 使用 TLS 加密，HAProxy 不需要解密流量；配置健康检查（option tcp-check + check），定期探测后端 Master 的 6443 端口，自动剔除故障节点；设置合理的超时参数（timeout connect/client/server），避免长时间等待。HAProxy 本身也存在单点风险，可以通过 Keepalived 实现 HAProxy 的 VIP 漂移，或使用多个 HAProxy 实例。在生产环境中，HAProxy 可以部署在独立节点上，也可以使用云厂商的 LB 服务。

### Q7: kubeadm join 的工作原理是什么？Bootstrap Token 的生命周期？ [难度: 高]

**答案：** kubeadm join 是 Worker 节点（或其他 Master 节点）加入集群的过程。工作原理为：kubelet 使用 Bootstrap Token 向 API Server 发起 TLS Bootstrap 请求，提交 CSR（Certificate Signing Request）；API Server 根据 Token 关联的 RBAC 角色进行审批；审批通过后，controller-manager 签发客户端证书；kubelet 获得证书后，使用正式证书与 API Server 通信。对于控制面节点加入（`--control-plane`），还需要从集群的 `kubeadm-certs` Secret 中下载控制面证书（使用 `--certificate-key` 解密），然后生成静态 Pod Manifest 并启动控制面组件。Bootstrap Token 的生命周期：通过 `kubeadm token create` 创建，默认 24 小时过期；Token 格式为 `[a-z0-9]{6}.[a-z0-9]{16}`；Token 关联了 `system:bootstrappers:kubeadm:default-node-token` 组的 RBAC 权限；可以使用 `--ttl=0` 创建永不过期的 Token（不推荐）；过期后需要重新创建 Token 才能加入新节点。

### Q8: 如何验证 K8s 集群的高可用性？ [难度: 中]

**答案：** 验证 K8s 集群高可用性需要从多个维度测试。第一，控制面冗余测试：逐个停止 Master 节点的 kubelet，确认集群仍然可用（kubectl get nodes 返回正确结果，Pod 调度正常）。第二，负载均衡测试：检查 HAProxy 的后端状态，确认所有 Master 都在负载池中；通过 VIP 多次访问 API Server，确认请求分发到不同 Master。第三，etcd 集群测试：使用 `etcdctl endpoint health --cluster` 检查所有 etcd 成员健康；停止一个 etcd 成员，确认集群仍可读写（Raft 协议需要多数成员存活，3 节点下可容忍 1 个故障）。第四，业务连续性测试：在 Master 故障期间创建/删除/更新 Deployment，确认操作成功。第五，恢复测试：恢复故障 Master 后，确认节点自动恢复 Ready 状态，etcd 自动重新加入集群。第六，脑裂测试：模拟网络分区，确认不会出现两个集群同时可写的情况。建议将这些测试整合为自动化测试脚本，定期执行。

### Q9: kubeadm upgrade 的流程是什么？有哪些注意事项？ [难度: 高]

**答案：** kubeadm upgrade 的流程遵循严格的顺序：首先升级 kubeadm（`apt-get install kubeadm=<new-version>`），然后执行 `kubeadm upgrade plan` 查看升级计划和兼容性，接着执行 `kubeadm upgrade apply <new-version>` 升级控制面（此步骤会自动升级 etcd、API Server、Controller Manager、Scheduler 的静态 Pod），然后升级 kubelet（`apt-get install kubelet=<new-version>` 并重启 kubelet），最后升级 kubectl。注意事项：第一，K8s 不支持跨次版本升级，必须逐版本升级（如 1.26 -> 1.27 -> 1.28）；第二，升级前必须备份 etcd；第三，升级控制面时，先升级第一个 Master，确认成功后再升级其他 Master；第四，升级期间 API Server 会短暂不可用（通常 < 1 分钟），建议在维护窗口执行；第五，升级后需要验证所有组件版本一致（`kubectl get nodes`）；第六，升级可能导致 API 废弃（Deprecated API Removal），需要提前检查并迁移；第七，CNI 插件可能需要单独升级。详见第10.4节集群升级策略。

### Q10: 什么是静态 Pod？为什么 K8s 控制面组件使用静态 Pod？ [难度: 中]

**答案：** 静态 Pod 是由 kubelet 直接管理的 Pod，不受 API Server 调度。kubelet 监视指定目录（默认 /etc/kubernetes/manifests/）下的 YAML/JSON 文件，自动创建和管理对应的容器。K8s 控制面组件（etcd、API Server、Scheduler、Controller Manager）使用静态 Pod 的原因：第一，鸡生蛋问题：这些组件本身就是 K8s 的核心组件，不能依赖 API Server 来调度自己；第二，简单可靠：静态 Pod 由 kubelet 直接管理，不依赖任何外部组件，启动顺序可控；第三，自动重启：kubelet 会监控静态 Pod 的健康状态，异常退出时自动重启。在 kubeadm 部署的集群中，`kubeadm init` 会在 /etc/kubernetes/manifests/ 目录下生成这些组件的 YAML 文件，kubelet 读取后自动创建容器。API Server 中可以看到对应的 Mirror Pod（只读副本），名称格式为 `<component>-<hostname>`。修改静态 Pod 的 YAML 文件后，kubelet 会自动重建对应的 Pod。

### Q11: 如何排查 kubeadm join 失败的问题？ [难度: 中]

**答案：** kubeadm join 失败的常见原因和排查步骤：第一，Token 过期或无效：在 Master 上执行 `kubeadm token list` 检查 Token 是否存在且未过期，过期则重新创建。第二，CA 证书哈希不匹配：检查 `--discovery-token-ca-cert-hash` 是否正确，可以在 Master 上通过 `openssl x509 -pubkey -in /etc/kubernetes/pki/ca.crt | openssl rsa -pubin -outform der 2>/dev/null | openssl dgst -sha256 -hex | sed 's/^.* //'` 获取正确哈希。第三，网络不通：检查 Worker 到 VIP（192.168.1.50:6443）的连通性，确认防火墙和 HAProxy 配置正确。第四，control-plane 证书密钥过期：`--certificate-key` 默认 2 小时过期，需要重新生成。第五，kubelet 配置不一致：检查 containerd 的 SystemdCgroup 配置是否与 kubelet 的 cgroup-driver 一致。第六，资源不足：检查节点 CPU/内存/磁盘是否满足最低要求。使用 `kubeadm join --v=5` 可以获取详细的调试日志。

### Q12: 如何配置 Kubernetes API Server 的审计日志？ [难度: 中]

**答案：** API Server 审计日志记录所有 API 请求，用于安全审计和故障排查。配置步骤为：首先创建审计策略文件（audit-policy.yaml），定义哪些请求需要记录以及记录级别（None/Metadata/Request/RequestResponse）。然后在 kubeadm-config.yaml 的 apiServer.extraArgs 中添加 `--audit-log-path`（日志路径）、`--audit-policy-file`（策略文件路径）、`--audit-log-maxage`（保留天数）、`--audit-log-maxbackup`（备份数量）、`--audit-log-maxsize`（单文件最大大小）。审计策略示例：对 Secrets 的读写操作记录 RequestResponse 级别（包含请求和响应体），对 ConfigMap 的写操作记录 Request 级别，对 Pod 的读操作记录 Metadata 级别，其他请求不记录。审计日志是 JSON 格式，每行一个审计事件，包含 stage（请求阶段）、verb（操作类型）、user（用户）、object（资源对象）、responseStatus（响应状态）等字段。生产环境中建议将审计日志集中收集到 ELK 或 Loki，并设置告警规则。


### 案例 7: 证书过期导致API不可访问

**现象：**
```bash
# kubectl 命令失败
kubectl get nodes
# Unable to connect to the server: x509: certificate has expired or is not yet valid

# API Server 日志显示证书错误
crictl logs <apiserver-container-id>
# "http: TLS handshake error from 192.168.1.51:12345: remote error: tls: bad certificate"
```

**排查步骤：**
1. 检查证书有效期：`kubeadm certs check-expiration`
2. 检查系统时间：`date`（时间不同步会导致证书验证失败）
3. 检查 API Server 证书：`openssl x509 -in /etc/kubernetes/pki/apiserver.crt -noout -dates`

**解决方案：**
```bash
# 1. 确认证书已过期
kubeadm certs check-expiration

# 2. 备份现有证书
mkdir -p /root/pki-backup-$(date +%Y%m%d)
cp -r /etc/kubernetes/pki/* /root/pki-backup-$(date +%Y%m%d)/

# 3. 续期所有证书（在所有Master节点上执行）
kubeadm certs renew all

# 4. 更新kubeconfig
cp /etc/kubernetes/admin.conf $HOME/.kube/config
chown $(id -u):$(id -g) $HOME/.kube/config

# 5. 重启kubelet（自动重启静态Pod）
systemctl restart kubelet

# 6. 等待Pod重启
sleep 30

# 7. 验证
kubectl get nodes
kubeadm certs check-expiration

# 8. 如果是多Master集群，同步证书到其他Master
for master in 192.168.1.52 192.168.1.53; do
    scp /etc/kubernetes/pki/*.crt root@$master:/etc/kubernetes/pki/
    scp /etc/kubernetes/pki/*.key root@$master:/etc/kubernetes/pki/
    scp /etc/kubernetes/admin.conf root@$master:/etc/kubernetes/
    ssh root@$master "systemctl restart kubelet"
done
```

**预防措施：**
```bash
# 设置证书到期监控（提前30天告警）
cat > /root/cert-expiry-alert.sh << 'EOF'
#!/bin/bash
WARNING_DAYS=30
NOW=$(date +%s)

for cert in /etc/kubernetes/pki/*.crt; do
    if [ -f "$cert" ]; then
        EXPIRY=$(openssl x509 -in "$cert" -noout -enddate 2>/dev/null | cut -d= -f2)
        if [ -n "$EXPIRY" ]; then
            EXPIRY_EPOCH=$(date -d "$EXPIRY" +%s)
            DAYS_REMAINING=$(( (EXPIRY_EPOCH - NOW) / 86400 ))
            if [ $DAYS_REMAINING -lt $WARNING_DAYS ]; then
                echo "ALERT: Certificate $(basename $cert) expires in $DAYS_REMAINING days"
            fi
        fi
    fi
done
EOF
chmod +x /root/cert-expiry-alert.sh

# 添加到crontab每天检查
echo "0 9 * * * root /root/cert-expiry-alert.sh" > /etc/cron.d/cert-alert
```

## 12. 生产环境建议

### 12.1 高可用最佳实践

1. **负载均衡**：使用 Keepalived + HAProxy 提供 VIP，避免 LB 单点故障
2. **etcd**：使用 SSD 存储 etcd 数据，定期监控磁盘延迟（< 10ms）
3. **节点规划**：Master 节点至少 4C8G，Worker 节点根据工作负载规划
4. **网络规划**：Pod 子网和 Service 子网不与物理网络重叠
5. **时间同步**：所有节点必须配置 NTP/chrony，时间偏差 < 500ms

### 12.2 安全加固

1. **RBAC**：遵循最小权限原则，定期审计 RBAC 规则
2. **审计日志**：启用 API Server 审计日志，集中收集和分析
3. **证书管理**：设置证书到期告警，提前 30 天续期
4. **网络策略**：部署 Calico NetworkPolicy，默认拒绝所有流量
5. **Secret 加密**：启用 etcd 静态加密（EncryptionConfiguration）

### 12.3 运维管理

1. **监控**：部署 Prometheus + Grafana 监控集群核心指标
2. **日志**：部署 Loki + Promtail 收集集群日志
3. **备份**：etcd 每小时备份，保留 7 天，异地存储
4. **升级**：跟随 K8s LTS 版本，提前在测试环境验证
5. **文档**：记录集群拓扑、网络规划、升级历史

### 12.4 集群升级策略

#### 12.4.1 升级前检查(API废弃警告)

**升级前检查清单：**

```bash
# 1. 检查当前版本
kubectl version

# 2. 检查可升级版本
kubeadm upgrade plan
```

#### 12.4.2 控制平面滚动升级

**升级架构图：**

```
+-------------------------------------------------------------+
|                  控制平面滚动升级流程                        |
+-------------------------------------------------------------+
|                                                             |
|  阶段1: Master-1                                            |
|  +-------------------+     +-------------------+           |
|  |  升级kubeadm      | --> |  升级控制面组件    |           |
|  |  kubeadm upgrade  |     |  API Server       |           |
|  |  apply            |     |  Controller       |           |
|  +-------------------+     |  Scheduler        |           |
|            |               +-------------------+           |
|            v                        |                       |
|  +-------------------+              v                       |

|  +-------------------+     +-------------------+           |

|  阶段3: Master-3 (重复阶段1流程)                            |

+-------------------------------------------------------------+
```

**控制平面升级步骤：**

```bash
#!/bin/bash
# 控制平面滚动升级脚本

MASTER_NODES=("192.168.1.51" "192.168.1.52" "192.168.1.53")
TARGET_VERSION="1.29.0"

# 1. 升级第一个Master节点
echo "Step 1: Upgrading first master node..."
ssh root@${MASTER_NODES[0]} "
    # 升级kubeadm
    apt-get update
    apt-get install -y kubeadm=${TARGET_VERSION}-00
    
    # 验证升级计划
    kubeadm upgrade plan
    
    # 执行升级（非交互模式）
    kubeadm upgrade apply v${TARGET_VERSION} --yes
    
    # 升级kubelet和kubectl
    apt-get install -y kubelet=${TARGET_VERSION}-00 kubectl=${TARGET_VERSION}-00
    systemctl restart kubelet
"

# 等待第一个Master就绪
sleep 30
kubectl wait --for=condition=Ready node/${MASTER_NODES[0]} --timeout=120s

# 2. 升级其他Master节点
for i in 1 2; do
    echo "Step $((i+1)): Upgrading master node ${MASTER_NODES[$i]}..."
    ssh root@${MASTER_NODES[$i]} "
        # 升级kubeadm
        apt-get update
        apt-get install -y kubeadm=${TARGET_VERSION}-00
        
        # 升级控制面（不升级etcd，因为已经在第一个Master升级）
        kubeadm upgrade node
        
        # 升级kubelet和kubectl
        apt-get install -y kubelet=${TARGET_VERSION}-00 kubectl=${TARGET_VERSION}-00
        systemctl restart kubelet
    "
    
    # 等待节点就绪
    sleep 30
    kubectl wait --for=condition=Ready node/${MASTER_NODES[$i]} --timeout=120s
done

echo "Control plane upgrade completed!"
```

#### 12.4.3 Worker节点分批升级

**Worker节点升级策略：**

```bash
#!/bin/bash
# Worker节点分批升级脚本

WORKER_NODES=("192.168.1.54" "192.168.1.55" "192.168.1.56" "192.168.1.57")
TARGET_VERSION="1.29.0"
BATCH_SIZE=2  # 每批升级的节点数

# 分批升级Worker节点
for ((i=0; i<${#WORKER_NODES[@]}; i+=BATCH_SIZE)); do
    BATCH=(${WORKER_NODES[@]:i:BATCH_SIZE})
    echo "Upgrading batch: ${BATCH[@]}"
    
    for node in "${BATCH[@]}"; do
        echo "  - Draining node: $node"
        kubectl drain $node --ignore-daemonsets --delete-emptydir-data --force
        
        echo "  - Upgrading node: $node"
        ssh root@$node "
            apt-get update
            apt-get install -y kubeadm=${TARGET_VERSION}-00
            kubeadm upgrade node
            apt-get install -y kubelet=${TARGET_VERSION}-00 kubectl=${TARGET_VERSION}-00
            systemctl restart kubelet
        "
        
        echo "  - Uncordon node: $node"
        kubectl uncordon $node
    done
    
    # 等待当前批次稳定
    echo "Waiting for batch to stabilize..."
    sleep 60
    
    # 验证当前批次
    for node in "${BATCH[@]}"; do
        kubectl wait --for=condition=Ready node/$node --timeout=120s
    done
done

echo "Worker nodes upgrade completed!"
```

#### 12.4.4 升级失败回滚方案

**回滚策略：**

```bash
#!/bin/bash
# 集群升级回滚脚本

BACKUP_DIR="/root/k8s-backup-$(date +%Y%m%d)"
PREVIOUS_VERSION="1.28.15"

# 1. 停止kubelet（所有节点）
echo "Step 1: Stopping kubelet on all nodes..."
for node in $(kubectl get nodes -o jsonpath='{.items[*].metadata.name}'); do
    ssh root@$node "systemctl stop kubelet"
done

# 2. 从etcd备份恢复
echo "Step 2: Restoring etcd from backup..."
ETCDCTL_API=3 etcdctl snapshot restore ${BACKUP_DIR}/etcd-pre-upgrade.db \
    --data-dir=/var/lib/etcd-restored

# 在所有Master节点上替换etcd数据
for master in 192.168.1.51 192.168.1.52 192.168.1.53; do
    ssh root@$master "
        mv /var/lib/etcd /var/lib/etcd.upgrade-failed
        mv /var/lib/etcd-restored /var/lib/etcd
        chown -R etcd:etcd /var/lib/etcd
    "
done

# 3. 降级kubeadm、kubelet、kubectl
echo "Step 3: Downgrading packages..."
for node in $(kubectl get nodes -o jsonpath='{.items[*].metadata.name}'); do
    ssh root@$node "
        apt-get install -y kubeadm=${PREVIOUS_VERSION}-00
        apt-get install -y kubelet=${PREVIOUS_VERSION}-00
        apt-get install -y kubectl=${PREVIOUS_VERSION}-00
    "
done

# 4. 重启kubelet
echo "Step 4: Restarting kubelet..."
for node in $(kubectl get nodes -o jsonpath='{.items[*].metadata.name}'); do
    ssh root@$node "systemctl start kubelet"
done

# 5. 验证回滚
echo "Step 5: Verifying rollback..."
sleep 60
kubectl get nodes
kubectl version

echo "Rollback completed!"
```
