## 一、理论核心

### 1.1 公司业务架构全景

公司是一家AI智能文档与大模型应用服务商，核心产品覆盖"识别-理解-生成-交互"全链路。运维工程师需要理解各产品线的技术栈和运维重点，才能快速接手工作。

| 产品模块 | 核心技术栈 | 运维重点 | 常见故障模式 |
|---------|-----------|---------|------------|
| **AI-OCR** | 图像识别模型、OpenCV、GPU推理、异步队列 | 高并发图像处理稳定性、模型版本管理、GPU资源调度 | 识别准确率下降、GPU显存溢出、队列堆积 |
| **大模型平台** | vLLM/TensorRT-LLM、K8s、向量数据库、知识库 | 推理服务高可用、Token成本控制、模型热切换 | 推理延迟飙升、Token消耗异常、模型加载失败 |
| **智能文档Agent(IDRP)** | 文档解析引擎、工作流引擎、RAG检索、合同比对 | 文档处理流水线稳定性、数据合规、版本一致性 | 文档解析超时、RAG检索失效、比对结果不一致 |
| **知识管理Agent(AI-KM)** | 智能客服引擎、多轮对话、知识图谱、培训系统 | 对话服务可用性、知识库同步、用户会话管理 | 对话中断、知识库更新延迟、会话状态丢失 |
| **智能问数(AI-SQL)** | NL2SQL模型、数据库代理、数据看板、权限控制 | 数据库查询性能、SQL生成准确性、数据安全 | 查询超时、SQL语法错误、敏感数据泄露 |
| **AI语音外呼** | ASR语音识别、TTS语音合成、NLP对话、RTC通信 | 通话质量、低延迟网络、合规录音、AI质检 | 通话卡顿、语音识别率低、合规风险 |

**关键要点**：
- 公司产品的共同特点是"AI模型+传统服务"的混合架构，运维需同时关注模型推理层和应用服务层
- OCR和语音外呼是公司的收入核心，其稳定性直接影响客户SLA和收入
- 大模型平台和Agent是公司的技术壁垒，其性能和创新决定市场竞争力
- 所有产品都涉及客户敏感数据（文档、对话、语音），数据安全和合规是红线

## 二、实操演练

### 任务1：AI-OCR服务高并发保障与GPU资源管理

**任务目标**：掌握OCR推理服务的运维要点，包括GPU节点管理、图像处理队列监控、识别准确率异常排查。

**操作步骤**：

```bash
# 步骤1：查看OCR推理服务状态（GKE中）
kubectl get pods -n ocr-production -l app=ocr-inference
# 预期输出：Pod列表，显示READY状态、重启次数、运行时间

# 步骤2：检查GPU节点状态和显存使用
kubectl top nodes --show-capacity | grep nvidia.com/gpu
# 预期输出：各GPU节点的CPU、内存、GPU利用率

# 查看NVIDIA显存详情（进入GPU节点）
kubectl exec -it ocr-inference-xxx -n ocr-production -- nvidia-smi
# 预期输出：GPU型号、显存使用、温度、进程列表

# 步骤3：监控图像处理队列长度（Redis队列）
kubectl exec -it redis-ocr-queue-0 -n ocr-production -- redis-cli LLEN ocr:pending:images
# 预期输出：待处理图像数量（>1000表示队列堆积）

# 步骤4：查看OCR识别准确率指标（Prometheus查询）
# 在Grafana中查询：
# ocr_accuracy_rate{model="invoice-v3"}
# ocr_latency_p99{service="ocr-api"}
# ocr_throughput_images_per_second

# 步骤5：排查识别准确率下降
# 1. 检查模型版本是否变更
kubectl get deployment ocr-inference -n ocr-production -o yaml | grep image
# 预期输出：当前使用的模型镜像版本

# 2. 查看模型加载日志
kubectl logs -l app=ocr-inference -n ocr-production --tail=100 | grep -i "model load\|accuracy\|error"
# 预期输出：模型加载状态、准确率校准信息

# 3. 检查输入图像质量（是否有过小、模糊、格式异常的图像）
kubectl logs -l app=ocr-preprocessor -n ocr-production --tail=50 | grep -i "invalid\|corrupt\|format"
# 预期输出：异常图像处理记录

# 步骤6：GPU OOM紧急处理
# 1. 查看OOM事件
kubectl get events -n ocr-production --field-selector reason=OOMKilled --sort-by='.lastTimestamp'
# 预期输出：OOMKilled事件列表

# 2. 降低并发处理数（减少同时处理的图像数量）
kubectl patch deployment ocr-inference -n ocr-production -p '{"spec":{"template":{"spec":{"containers":[{"name":"ocr","env":[{"name":"MAX_CONCURRENT_IMAGES","value":"5"}]}]}}}}'
# 预期输出：deployment patched

# 3. 扩容GPU节点（如使用Cluster Autoscaler）
kubectl scale deployment ocr-inference -n ocr-production --replicas=5
# 预期输出：deployment scaled
```

**注意事项**：
- OCR服务对GPU显存敏感，建议设置显存使用阈值告警（>85%触发告警）
- 图像预处理（尺寸调整、格式转换）应在CPU节点完成，仅推理阶段使用GPU
- 模型版本更新需先在小流量环境验证准确率，确认无误后再全量发布
- 队列堆积时优先扩容消费者（推理Pod），而非增加队列长度

### 任务2：大模型推理平台日常运维

**任务目标**：掌握大模型推理平台的日常运维操作，包括模型切换、Token监控、成本控制和故障排查。

**操作步骤**：

```bash
# 步骤1：查看当前运行的模型和版本
kubectl get deployment -n llm-production -o custom-columns=NAME:.metadata.name,MODEL:.spec.template.spec.containers[0].env[?(@.name=="MODEL_ID")].value,REPLICAS:.spec.replicas
# 预期输出：各推理服务的模型ID和副本数

# 步骤2：查看Token消耗实时数据
# Prometheus查询：
# sum(rate(vllm:generation_tokens_total[5m])) by (model)
# sum(rate(vllm:prompt_tokens_total[5m])) by (tenant)

# 步骤3：模型热切换（不中断服务）
# 方法：使用K8s滚动更新，逐步替换Pod
kubectl set image deployment/llm-inference -n llm-production \
    llm=vllm/vllm-openai:v0.6.1 \
    --record
# 预期输出：deployment image updated

# 监控滚动更新进度
kubectl rollout status deployment/llm-inference -n llm-production
# 预期输出：deployment successfully rolled out

# 步骤4：查看推理延迟分布
# Grafana查询：
# histogram_quantile(0.99, sum(rate(vllm:time_to_first_token_seconds_bucket[5m])) by (le))
# histogram_quantile(0.95, sum(rate(vllm:time_per_output_token_seconds_bucket[5m])) by (le))

# 步骤5：成本异常排查（Token消耗突增）
# 1. 查看按租户分组的Token消耗
kubectl exec -it prometheus-0 -n monitoring -- curl -s 'http://localhost:9090/api/v1/query?query=sum(rate(vllm:generation_tokens_total[1h])) by (tenant)' | jq
# 预期输出：各租户的Token消耗速率

# 2. 查看异常租户的请求日志
kubectl logs -l app=llm-gateway -n llm-production --tail=1000 | grep "tenant=异常租户ID" | grep -i "prompt\|token"
# 预期输出：异常租户的具体请求内容

# 步骤6：推理服务完全故障时的应急处理
# 1. 切换到备用模型（轻量级模型兜底）
kubectl patch deployment llm-inference -n llm-production -p '{"spec":{"template":{"spec":{"containers":[{"name":"llm","env":[{"name":"MODEL_ID","value":"gemma-3-4b-it"}]}]}}}}'
# 预期输出：deployment patched（切换到小模型，保证服务可用）

# 2. 通知客户降级（通过Slack/飞书）
curl -X POST https://hooks.slack.com/services/xxx \
    -H 'Content-Type: application/json' \
    -d '{"text":"【告警】大模型推理服务降级通知：当前使用轻量级模型，复杂任务精度可能下降，预计30分钟内恢复"}'
```

**注意事项**：
- 模型热切换需确保新旧模型API兼容（输入输出格式一致）
- Token消耗突增可能是Prompt注入攻击或循环调用，需设置单租户Token上限
- 备用模型（Fallback Model）策略是保障SLA的关键，建议常备一个轻量级模型
- 推理服务故障时，优先保证"服务可用"，再追求"模型精度"

### 任务3：智能文档处理Agent（IDRP）流水线监控

**任务目标**：掌握文档处理流水线的运维，包括文档解析、RAG检索、合同比对等环节的监控和故障处理。

**操作步骤**：

```bash
# 步骤1：查看文档处理流水线各阶段状态
kubectl get pods -n idrp-production
# 预期输出：各阶段Pod状态（文档上传→解析→向量化→存储→检索→生成）

# 步骤2：监控文档解析队列
kubectl exec -it rabbitmq-0 -n idrp-production -- rabbitmqctl list_queues name messages_ready messages_unacknowledged
# 预期输出：各队列的待处理消息数和未确认消息数

# 步骤3：查看RAG检索性能
# Prometheus查询：
# idrp_rag_retrieval_latency_seconds{quantile="0.99"}
# idrp_rag_recall_rate{kb="合同知识库"}

# 步骤4：排查合同比对结果不一致
# 1. 查看比对服务日志
kubectl logs -l app=idrp-compare -n idrp-production --tail=200 | grep -i "mismatch\|diff\|error"
# 预期输出：比对差异的具体信息

# 2. 检查向量数据库同步状态
kubectl exec -it vector-db-0 -n idrp-production -- curl -s http://localhost:6333/collections/contract_kb | jq '.result.points_count'
# 预期输出：向量数据库中的文档数量

# 3. 对比源文档和向量库文档数量
kubectl exec -it postgres-idrp-0 -n idrp-production -- psql -U idrp -c "SELECT COUNT(*) FROM documents WHERE status='processed';"
# 预期输出：已处理文档数量（应与向量库数量一致）

# 步骤5：文档处理超时处理
# 1. 查看超时文档
kubectl exec -it postgres-idrp-0 -n idrp-production -- psql -U idrp -c "SELECT id, filename, status, created_at FROM documents WHERE status='processing' AND created_at < NOW() - INTERVAL '30 minutes';"
# 预期输出：超时的文档列表

# 2. 重置超时文档状态（重新处理）
kubectl exec -it postgres-idrp-0 -n idrp-production -- psql -U idrp -c "UPDATE documents SET status='pending' WHERE status='processing' AND created_at < NOW() - INTERVAL '30 minutes';"
# 预期输出：UPDATE影响的行数

# 3. 检查解析服务是否卡住
kubectl logs -l app=idrp-parser -n idrp-production --tail=50 | grep -i "timeout\|stuck\|kill"
# 预期输出：解析服务的超时或卡死记录
```

**注意事项**：
- 文档处理是长事务流程，需设置每个阶段的超时时间和重试机制
- 向量数据库（如Qdrant/Milvus）的同步状态是关键检查点，文档数和向量数不一致会导致检索失效
- 合同比对涉及法律合规，比对结果必须可追溯，所有差异需记录审计日志
- 大文档（>100MB）解析容易超时，建议设置文档大小限制或分块处理

### 任务4：AI语音外呼服务运维与通话质量保障

**任务目标**：掌握语音外呼服务的运维要点，包括通话质量监控、ASR/TTS服务管理、合规录音存储。

**操作步骤**：

```bash
# 步骤1：查看语音外呼服务整体状态
kubectl get pods -n voice-production
# 预期输出：各组件状态（ASR、TTS、NLP对话、RTC网关、呼叫控制）

# 步骤2：监控通话质量指标（MOS评分、延迟、丢包率）
# Prometheus查询：
# voice_mos_score{service="ai-call"}
# voice_latency_ms{direction="roundtrip"}
# voice_packet_loss_rate{service="rtc"}

# 步骤3：查看ASR识别准确率
kubectl logs -l app=asr-service -n voice-production --tail=100 | grep -i "accuracy\|wer\|cer"
# 预期输出：ASR的准确率、词错误率（WER）、字错误率（CER）

# 步骤4：排查通话卡顿问题
# 1. 检查RTC网关的网络质量
kubectl exec -it rtc-gateway-0 -n voice-production -- curl -s http://localhost:8080/metrics | grep -i "jitter\|latency\|loss"
# 预期输出：网络抖动、延迟、丢包指标

# 2. 检查TTS合成延迟
kubectl logs -l app=tts-service -n voice-production --tail=50 | grep -i "synthesis_time\|latency"
# 预期输出：TTS合成耗时

# 3. 检查NLP对话响应时间
kubectl logs -l app=nlp-dialogue -n voice-production --tail=50 | grep -i "response_time\|inference"
# 预期输出：NLP模型推理耗时

# 步骤5：合规录音存储检查
# 1. 检查录音文件是否成功上传到存储桶
kubectl exec -it voice-recorder-0 -n voice-production -- gsutil ls gs://voice-recordings-prod/$(date +%Y-%m-%d)/ | wc -l
# 预期输出：当日录音文件数量

# 2. 检查录音元数据是否写入数据库
kubectl exec -it postgres-voice-0 -n voice-production -- psql -U voice -c "SELECT COUNT(*) FROM call_recordings WHERE call_date=CURRENT_DATE;"
# 预期输出：当日通话记录数（应与录音文件数一致）

# 步骤6：外呼任务失败排查
# 1. 查看呼叫控制服务日志
kubectl logs -l app=call-controller -n voice-production --tail=200 | grep -i "failed\|reject\|busy\|no_answer"
# 预期输出：呼叫失败原因统计

# 2. 检查运营商线路状态
kubectl exec -it sip-trunk-0 -n voice-production -- sngrep -r | grep -i "403\|404\|486\|503"
# 预期输出：SIP响应码（403=Forbidden, 486=Busy, 503=Service Unavailable）
```

**注意事项**：
- 语音外呼受运营商线路质量影响大，需监控SIP Trunk状态和响应码
- 通话录音涉及合规要求（如金融行业的双录），必须确保录音完整存储且不可篡改
- ASR识别准确率受口音、噪音影响，需定期更新模型和优化音频预处理
- 外呼频率需遵守合规要求（如每日最多3次），避免被运营商封号

### 任务5：入职第一周实战——处理一次完整故障

**任务目标**：模拟入职后处理一次跨产品的生产故障，建立完整的故障处理流程。

**操作步骤**：

```bash
# 场景：客户反馈OCR识别准确率突然下降，同时大模型推理延迟飙升

# 步骤1：接收告警（PagerDuty/OpsGenie）
# 告警内容：
# [P1] ocr_accuracy_rate < 90% for 5 minutes
# [P1] llm_p99_latency > 10s for 3 minutes

# 步骤2：确认影响范围
# 1. 查看受影响客户
kubectl logs -l app=api-gateway -n production --since=10m | grep -i "error\|timeout" | jq -r '.tenant_id' | sort | uniq -c | sort -rn | head -10
# 预期输出：受影响租户列表和错误次数

# 2. 查看SLA影响
# Grafana仪表盘：客户SLA面板，显示各租户的服务可用性

# 步骤3：快速定位根因
# 假设：GPU节点故障导致OCR和LLM共享的GPU资源不足

# 1. 查看GPU节点状态
kubectl get nodes -l accelerator=nvidia-t4
# 预期输出：节点状态（NotReady/Ready）

# 2. 查看节点事件
kubectl describe node gpu-node-01 | grep -A 10 Events
# 预期输出：节点事件（如KernelDeadlock、MemoryPressure）

# 3. 查看GPU驱动状态
kubectl exec -it nvidia-driver-installer-xxx -- nvidia-smi
# 预期输出：GPU驱动和硬件状态

# 步骤4：应急处理
# 1. 将故障节点上的Pod驱逐到其他节点
kubectl drain gpu-node-01 --ignore-daemonsets --delete-emptydir-data
# 预期输出：节点已排空，Pod已迁移

# 2. 验证服务恢复
kubectl get pods -n ocr-production -w
kubectl get pods -n llm-production -w
# 预期输出：Pod重新调度并变为Running

# 3. 验证指标恢复
# Grafana查看：ocr_accuracy_rate 和 llm_p99_latency 是否恢复正常

# 步骤5：通知相关方
# 1. 在Slack/飞书发布故障公告
curl -X POST https://hooks.slack.com/services/xxx \
```

## 三、面试真题

### 基础 高频 - Q1: 你入职一家AI公司后，第一周会怎么做？

> **参考答案**：
> 1. **Day 1-2**：获取账号权限，熟悉组织架构，查看生产环境架构图，了解核心服务清单和依赖关系
> 2. **Day 3**：熟悉监控大盘（CloudWatch/Grafana）和告警规则，了解值班流程和升级机制
> 3. **Day 4**：查看最近一周的故障记录和变更记录，了解历史问题和经验教训
> 4. **Day 5**：跟随值班人员处理一次告警或故障，实战学习并记录操作手册
> 5. **持续**：制作个人"运维地图"，记录每个服务的负责人、技术栈、关键指标、常见故障
>
> **延伸知识**：入职第一周不要急于动手改东西，先建立对系统的整体认知，避免"新人效应"导致的误操作

---

### 中等 高频 - Q2: AI-OCR服务识别准确率突然下降，你会怎么排查？

> **参考答案**：
> 1. **检查模型版本**：确认最近是否有模型更新，回滚到上一个版本验证是否恢复
> 2. **检查GPU状态**：nvidia-smi查看GPU是否正常工作，显存是否溢出，驱动是否加载
> 3. **检查输入质量**：查看预处理日志，确认是否有大量异常图像（过小、模糊、格式错误）
> 4. **检查队列状态**：Redis/RabbitMQ队列是否堆积，消费者是否足够
> 5. **检查依赖服务**：图像存储（S3/GCS）是否可达，预处理服务是否正常
> 6. **对比历史数据**：查看相同时间段的历史准确率，确认是否是周期性波动
>
> **延伸知识**：OCR准确率下降的常见原因：模型版本问题（40%）、输入质量问题（30%）、GPU资源问题（20%）、依赖服务问题（10%）

---

### 中等 高频 - Q3: 大模型推理服务的Token消耗突然翻倍，你怎么处理？

> **参考答案**：
> 1. **定位异常租户**：Prometheus按租户分组查询Token消耗，找出贡献最大的租户
> 2. **分析请求内容**：查看该租户的请求日志，确认是否是Prompt注入攻击或循环调用
> 3. **设置临时限制**：对该租户设置Token上限（Rate Limiting），防止继续消耗
> 4. **检查模型配置**：确认是否有模型参数变更（如temperature升高导致输出变长）
> 5. **通知客户**：如果是正常业务增长，协商升级套餐；如果是异常调用，协助排查
> 6. **长期措施**：设置单租户Token告警阈值（如小时消耗超过日均3倍触发告警）
>
> **延伸知识**：Token消耗异常增长的常见原因：业务增长（50%）、Prompt优化不当（20%）、恶意调用（15%）、模型参数变更（10%）、其他（5%）

---

### 困难 中频 - Q4: 公司的AI语音外呼服务出现通话卡顿，你会怎么排查？

> **参考答案**：
> 1. **网络质量检查**：RTC网关的 jitter、latency、packet_loss 指标，确认是否是网络问题
> 2. **ASR服务检查**：查看ASR识别延迟和准确率，确认是否是语音识别环节卡顿
> 3. **TTS服务检查**：查看TTS合成延迟，确认是否是语音合成环节卡顿
> 4. **NLP对话检查**：查看NLP模型推理延迟，确认是否是对话理解环节卡顿
> 5. **运营商线路检查**：SIP Trunk响应码（403/486/503），确认是否是线路问题
> 6. **端到端追踪**：使用Cloud Trace/Jaeger查看完整通话链路的延迟分布
>
> **延伸知识**：语音外呼卡顿的常见原因：网络抖动（35%）、ASR延迟（25%）、NLP推理慢（20%）、TTS合成慢（15%）、线路质量差（5%）

---

### 中等 中频 - Q5: 智能文档处理Agent的RAG检索失效，你会怎么排查？

> **参考答案**：
> 1. **检查向量数据库状态**：确认向量数据库（Qdrant/Milvus）是否正常运行，集合是否存在
> 2. **检查文档同步状态**：对比源数据库文档数和向量库向量数，确认是否同步
> 3. **检查嵌入模型**：确认文档嵌入模型是否正常运行，向量维度是否一致
> 4. **检查检索参数**：确认top_k、相似度阈值等参数是否合理
> 5. **检查索引状态**：向量数据库的索引是否构建完成，是否过期
> 6. **手动测试**：用已知文档内容手动执行检索，验证是否能返回正确结果
>
> **延伸知识**：RAG检索失效的常见原因：向量库未同步（40%）、嵌入模型变更（25%）、索引损坏（15%）、参数配置错误（10%）、其他（10%）

---

### 困难 低频 - Q6: 公司的多个AI产品共享GPU集群，如何设计资源隔离和调度策略？

> **参考答案**：
> 1. **节点池隔离**：按产品划分GPU节点池（ocr-pool、llm-pool、voice-pool），避免资源争抢
> 2. **Namespace隔离**：每个产品独立Namespace，ResourceQuota限制GPU、CPU、内存配额
> 3. **优先级调度**：核心收入产品（OCR、语音外呼）设置高PriorityClass，保证资源优先分配
> 4. **时间片调度**：非实时产品（文档处理）安排在低峰期运行，错峰使用GPU
> 5. **动态伸缩**：基于队列长度和Token吞吐量自动伸缩各产品的推理实例数
> 6. **成本分摊**：按GPU使用时长和Token消耗量，生成各产品的月度成本报表
>
> **延伸知识**：GPU资源争抢的典型场景：大模型推理占用大量显存，导致OCR服务无法调度；解决方案是设置GPU节点的taints和tolerations，强制产品隔离

### 基础 中频 - Q7: 入职后如何快速建立对复杂系统的认知？

> **参考答案**：
> 1. **画架构图**：从用户请求入口开始，追踪请求经过的每个服务，画出完整的调用链
> 2. **做服务清单**：记录每个服务的名称、技术栈、负责人、关键指标、常见故障
> 3. **看监控大盘**：每天花15分钟看Grafana仪表盘，建立对正常状态的直觉
> 4. **读故障记录**：查看最近3个月的P1/P2故障复盘，了解系统的薄弱环节
> 5. **跟值班**：跟随资深运维处理告警，学习排查思路和工具使用
> 6. **写Runbook**：每处理一次故障，更新运维手册，形成知识沉淀
>
> **延伸知识**：建议制作"运维地图"（Mind Map），将系统架构、服务依赖、监控链接、联系人信息整合在一张图上，入职第一周完成

### 中等 低频 - Q8: 公司的AI产品涉及客户敏感数据，运维上如何保障数据安全？

> **参考答案**：
> 1. **访问控制**：IAM最小权限原则，运维人员只分配必要的只读权限，敏感操作需审批
> 2. **数据加密**：传输中TLS加密，存储中AES-256加密，密钥由KMS/Secret Manager管理
> 3. **审计日志**：所有数据访问操作记录CloudTrail/Cloud Audit Logs，保留至少1年
> 4. **网络隔离**：生产环境VPC与办公网络隔离，敏感数据服务部署在私有子网
> 5. **数据脱敏**：测试环境使用脱敏数据，禁止将生产数据导出到本地
> 6. **合规检查**：定期扫描S3/GCS存储桶的公开访问设置，Config规则监控资源配置合规性
> 7. **应急响应**：数据泄露事件响应预案，24小时内通知受影响客户和监管部门
>
> **延伸知识**：AI公司的数据安全特别关注点：模型训练数据是否包含PII（个人身份信息）、推理日志是否记录敏感输入、向量数据库是否泄露客户知识库内容
