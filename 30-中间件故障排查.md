# 模块30：中间件故障排查

---

## 一、理论核心

### 1.1 中间件故障分层诊断模型

中间件故障排查遵循"客户端到网络到服务端到存储"的四层链路思路。生产环境中中间件故障往往表现为应用层异常，但根因可能在网络、配置或资源层面。

| 层级 | 排查对象 | 核心命令 | 典型故障 |
|------|----------|----------|----------|
| 客户端层 | 连接池、驱动版本、超时配置 | `netstat -anp` | 连接泄漏、驱动bug |
| 网络层 | 连通性、延迟、带宽、防火墙 | `ping` / `telnet` | 网络分区、DNS抖动 |
| 服务端层 | 进程状态、资源使用、日志 | `systemctl status` | 进程崩溃、内存不足 |
| 存储层 | 磁盘I/O、文件句柄、数据文件 | `iostat -x` | 磁盘满、I/O瓶颈 |

**关键要点**：
- 中间件故障80%是连接问题，先确认网络连通再深入排查
- 监控中间件核心指标（QPS、延迟、连接数、资源使用率）是快速定位的基础
- 中间件集群模式下，优先排查集群状态而非单节点

---

### 1.2 连接池与超时机制

应用与中间件交互的核心是连接池，连接池配置不当是生产环境最常见的故障根因。

| 参数 | 作用 | 配置过小后果 | 配置过大后果 |
|------|------|-------------|-------------|
| 最大连接数 | 限制并发连接上限 | 请求排队超时 | 中间件资源耗尽 |
| 连接超时 | 建立连接的最大等待时间 | 网络抖动时频繁失败 | 线程长期阻塞 |
| 读取超时 | 等待响应的最大时间 | 慢查询时失败 | 级联阻塞扩散 |
| 空闲检测 | 回收空闲连接的周期 | 连接泄漏累积 | 增加中间件负担 |

**关键要点**：
- 连接池大小公式：核心数 * 2 + 有效磁盘数（适用于大多数中间件）
- 超时设置应分层：连接超时 < 读取超时 < 业务超时
- 必须配置连接泄漏检测（如HikariCP的leakDetectionThreshold）

---

### 1.3 中间件集群高可用机制

生产环境中间件通常以集群模式部署，理解各中间件的故障转移机制是排查的前提。

| 中间件 | 集群模式 | 故障检测机制 | 故障转移方式 |
|--------|----------|--------------|--------------|
| MySQL | 主从复制 / MGR | heartbeat | Orchestrator自动切换 |
| Redis | 主从 + Sentinel | Sentinel PING | Sentinel选举新主节点 |
| Elasticsearch | 多节点集群 | Zen Discovery | 自动选举主节点，分片重新分配 |
| Kafka | 多Broker + KRaft | Controller心跳 | 自动选举新Controller，分区重平衡 |
| RabbitMQ | 镜像队列 | Erlang心跳 | 队列镜像自动切换 |
| RocketMQ | 多Master + 多Slave | Broker心跳 | NameServer路由切换 |
| Nacos | 多节点 + Raft | Raft心跳 | 自动Leader选举 |
| Nginx | 多实例 + Keepalived | VRRP心跳 | VIP自动漂移 |
| Hadoop | NameNode HA | ZKFC心跳 | 自动切换Active NameNode |

**关键要点**：
- 集群模式下单节点故障不应影响整体服务，若出现整体不可用需检查集群状态
- 脑裂（Split-Brain）是集群模式下的特殊故障，需配置quorum机制避免
- 故障转移期间可能出现短暂不可用，需评估业务容忍度

---

### 1.4 主流技术栈产物与部署方式

运维排查中间件故障时，需理解上下游应用的产物形态和部署方式，才能准确判断故障边界。

| 技术栈 | 语言/框架 | 产物形态 | 部署方式 | 运维关注点 |
|--------|-----------|----------|----------|------------|
| React前端 | JavaScript | 打包后的静态文件夹（build/） | Nginx托管静态文件 | 静态资源缓存、CDN回源、路由配置 |
| Vue前端 | JavaScript | dist目录下的HTML/CSS/JS | Nginx托管静态文件 | 同React，注意history路由需配try_files |
| Spring Boot | Java | 可执行Jar包 | java -jar启动或Docker容器 | JVM内存参数、GC策略、Heap Dump |
| Django | Python | 项目源码包 + requirements.txt | gunicorn/uwsgi + Nginx | Python版本、虚拟环境、依赖冲突 |
| Gin | Go | 编译好的二进制文件 | 直接运行或Docker容器 | 交叉编译、静态链接、无运行时依赖 |

**关键要点**：
- 前端产物是纯静态文件，故障排查方向是Nginx配置和CDN，不涉及进程崩溃
- Java产物是Jar包，依赖JVM运行时，内存和GC是排查重点
- Go产物是二进制文件，无运行时依赖，排查最简单，关注二进制本身和系统环境
- Python产物是源码包，依赖环境一致性，虚拟环境隔离是常见排障点

---

### 1.5 大数据组件基础

大数据组件是数据管道和AI训练的基础设施，运维需要理解各组件的架构角色和常见故障模式。

| 组件 | 架构角色 | 核心进程 | 常见故障 | 监控指标 |
|------|----------|----------|----------|----------|
| Hadoop HDFS | 分布式文件存储 | NameNode（元数据）+ DataNode（数据块） | NameNode HA切换、DataNode磁盘满、块丢失 | 存储容量、块数量、DataNode存活数 |
| YARN | 资源调度管理 | ResourceManager + NodeManager | Container分配失败、AM未释放资源 | 可用vcore/内存、运行/等待容器数 |
| Spark | 内存计算引擎 | Driver + Executor | Executor OOM、Shuffle阶段数据倾斜、Stage卡住 | Executor存活数、Shuffle读写量、Stage耗时 |
| Flink | 流式计算引擎 | JobManager + TaskManager | Checkpoint失败导致重启、反压（Backpressure）、Kafka消费延迟 | Checkpoint大小/耗时、反压指标、消费Lag |
| ClickHouse | 列式OLAP数据库 | Server节点（分片+副本） | MergeTree合并I/O高、分布式JOIN慢、ZooKeeper元数据不一致 | 查询QPS、P99延迟、MergeTree合并进度 |
| Hive | SQL on Hadoop | HiveServer2 + Metastore + HDFS | Metastore连接池耗尽、小文件过多导致MapReduce任务数爆炸 | 查询耗时、Tez/Spark引擎任务数 |

**关键要点**：
- HDFS NameNode是单点，生产环境必须HA（Active+Standby+ZooKeeper）
- Spark的Shuffle是性能瓶颈，数据倾斜时少数Executor处理大量数据导致OOM
- Flink的Checkpoint是 Exactly-Once 语义的保证，失败后从最近Checkpoint恢复
- ClickHouse适合OLAP查询（列式存储，聚合快），不适合OLTP（单行更新慢）
- 大数据集群的共性故障：磁盘满、内存不足、网络分区、ZooKeeper异常

## 二、实操演练

### 任务1：MySQL连接超时与慢查询排查

**任务目标**：定位MySQL连接超时根因，优化慢查询

**操作步骤**：

```bash
# 步骤1：检查MySQL进程状态
systemctl status mysqld
# 预期输出：Active: active (running)

# 步骤2：检查当前连接数
mysql -e "SHOW STATUS LIKE 'Threads_connected';"
# 预期输出：当前连接数
mysql -e "SHOW VARIABLES LIKE 'max_connections';"
# 预期输出：最大连接数限制

# 步骤3：检查慢查询日志
mysql -e "SHOW VARIABLES LIKE 'slow_query_log%';"
# 预期输出：确认慢查询日志已开启
cat /var/lib/mysql/slow.log | tail -20
# 预期输出：显示执行时间超过long_query_time的SQL

# 步骤4：分析慢查询
mysql -e "SELECT * FROM mysql.slow_log ORDER BY start_time DESC LIMIT 10;"
# 预期输出：显示最近10条慢查询的SQL、执行时间、扫描行数

# 步骤5：检查锁等待
mysql -e "SHOW ENGINE INNODB STATUS;" | grep -A20 "LATEST DETECTED DEADLOCK"
# 预期输出：显示死锁信息

# 步骤6：优化建议
mysql -e "EXPLAIN SELECT * FROM orders WHERE user_id = 123;"
# 预期输出：显示查询执行计划，确认是否使用索引

# 步骤7：验证优化效果
mysql -e "SHOW GLOBAL STATUS LIKE 'Slow_queries';"
# 预期输出：慢查询计数不再增长
```

**注意事项**：
- 连接超时可能是连接池耗尽，也可能是网络问题，需分层排查
- 慢查询优化优先添加索引，其次优化SQL写法，最后考虑分库分表
- 生产环境修改max_connections前需评估服务器内存容量

### 任务2：Redis内存溢出与主从切换排查

**任务目标**：定位Redis内存问题，排查主从切换失败

**操作步骤**：

```bash
# 步骤1：检查Redis进程和内存使用
redis-cli INFO memory
# 预期输出：used_memory、used_memory_rss、maxmemory等指标

# 步骤2：检查内存淘汰策略
redis-cli CONFIG GET maxmemory-policy
# 预期输出：noeviction / allkeys-lru / volatile-lru等

# 步骤3：检查大Key
redis-cli --bigkeys
# 预期输出：显示占用内存最大的Key

# 步骤4：检查主从复制状态
redis-cli INFO replication
# 预期输出：role、master_link_status、master_last_io_seconds_ago

# 步骤5：检查Sentinel状态（Sentinel模式）
redis-cli -p 26379 INFO sentinel
# 预期输出：sentinel_masters、status

# 步骤6：检查Cluster节点状态（Cluster模式）
redis-cli CLUSTER NODES
# 预期输出：所有节点ID、IP、端口、角色、连接状态

# 步骤7：检查持久化状态
redis-cli INFO persistence
# 预期输出：rdb_last_bgsave_status、aof_last_write_status

# 步骤8：验证修复
redis-cli PING
# 预期输出：PONG
```

**注意事项**：
- Redis内存溢出时若策略为noeviction，写操作会直接失败
- 主从切换失败常见原因：Sentinel配置错误、网络分区、从节点数据滞后
- Cluster模式下节点宕机会导致部分slot不可用，需确认集群是否完整

### 任务3：Elasticsearch集群健康度排查

**任务目标**：定位ES集群状态异常（黄/红）根因

**操作步骤**：

```bash
# 步骤1：检查集群健康状态
curl -s http://<es-node>:9200/_cluster/health | jq .
# 预期输出：status为green/yellow/red，active_shards、unassigned_shards

# 步骤2：检查节点列表
curl -s http://<es-node>:9200/_cat/nodes?v
# 预期输出：所有节点IP、角色、负载、内存使用率

# 步骤3：检查分片分配
curl -s http://<es-node>:9200/_cat/shards?v | grep UNASSIGNED
# 预期输出：显示未分配的分片

# 步骤4：检查集群任务队列
curl -s http://<es-node>:9200/_cluster/pending_tasks
# 预期输出：显示pending任务列表，判断集群是否繁忙

# 步骤5：检查索引状态
curl -s http://<es-node>:9200/_cat/indices?v | grep -E "red|yellow"
# 预期输出：显示非green状态的索引

# 步骤6：检查磁盘水位
curl -s http://<es-node>:9200/_cat/allocation?v
# 预期输出：各节点磁盘使用率，是否超过85%（high watermark）

# 步骤7：强制重新分配（仅yellow状态）
curl -X POST http://<es-node>:9200/_cluster/reroute?retry_failed=true
# 预期输出：{"acknowledged":true}

# 步骤8：验证修复
curl -s http://<es-node>:9200/_cluster/health | jq .status
# 预期输出："green"
```

**注意事项**：
- 集群red状态表示存在未分配主分片，数据可能丢失
- 磁盘超过high watermark（默认85%）会阻止分片分配到新节点
- 节点离线后重新加入集群，分片会自动重新分配，无需手动干预

### 任务4：Kafka消息堆积与消费延迟排查

**任务目标**：定位Kafka消息堆积根因，恢复消费正常

**操作步骤**：

```bash
# 步骤1：检查Broker状态
kafka-broker-api-versions.sh --bootstrap-server <broker>:9092
# 预期输出：所有Broker的API版本信息

# 步骤2：检查Topic分区状态
kafka-topics.sh --bootstrap-server <broker>:9092 --describe --topic <topic-name>
# 预期输出：分区数、副本分布、ISR列表、Leader信息

# 步骤3：检查消费者组状态
kafka-consumer-groups.sh --bootstrap-server <broker>:9092 --describe --group <group-id>
# 预期输出：各分区的CURRENT-OFFSET、LOG-END-OFFSET、LAG

# 步骤4：检查消息堆积量
kafka-consumer-groups.sh --bootstrap-server <broker>:9092 --describe --group <group-id> | awk '{sum+=$5} END {print "Total LAG:", sum}'
# 预期输出：总堆积消息数

# 步骤5：检查消费者日志
# 查看消费者应用日志，确认是否有消费异常、反序列化错误、Rebalance频繁

# 步骤6：检查Broker磁盘
ssh <broker-ip> df -h
# 预期输出：确认Kafka日志目录磁盘未超过85%

# 步骤7：扩容分区（临时方案）
kafka-topics.sh --bootstrap-server <broker>:9092 --alter --topic <topic-name> --partitions <new-count>
# 预期输出：分区扩容成功

# 步骤8：验证修复
kafka-consumer-groups.sh --bootstrap-server <broker>:9092 --describe --group <group-id>
# 预期输出：LAG逐渐降低至0
```

**注意事项**：
- 消息堆积常见原因：消费者处理速度慢、消费者数量不足、消费者频繁Rebalance
- Rebalance频繁通常由消费超时（session.timeout.ms）设置不当导致
- 分区扩容只能增加分区数，不能减少，且不会重新分配已有数据

### 任务5：Nginx高并发场景502/504排查

**任务目标**：定位Nginx反向代理超时和连接异常根因

**操作步骤**：

```bash
# 步骤1：检查Nginx进程状态
systemctl status nginx
# 预期输出：Active: active (running)

# 步骤2：检查错误日志
tail -50 /var/log/nginx/error.log
# 预期输出：upstream timed out、connection refused等错误

# 步骤3：检查连接数
ss -ant | grep :80 | wc -l
# 预期输出：当前80端口连接数

# 步骤4：检查Nginx配置
nginx -T | grep -E "worker_connections|upstream|proxy_connect_timeout|proxy_read_timeout"
# 预期输出：worker_connections、超时时间、后端地址

# 步骤5：检查后端服务健康
curl -s -o /dev/null -w "%{http_code}" http://<backend-ip>:<port>/health
# 预期输出：200
# 如果非200：后端服务异常

# 步骤6：检查后端连接超时
# 临时调大proxy_read_timeout验证是否为后端处理慢
grep proxy_read_timeout /etc/nginx/nginx.conf
# 预期输出：默认60秒

# 步骤7：检查系统连接限制
cat /proc/sys/net/core/somaxconn
# 预期输出：128或更大值
ulimit -n
# 预期输出：文件句柄限制，需大于worker_connections * worker_processes

# 步骤8：验证修复
nginx -t && nginx -s reload
# 预期输出：syntax is ok，配置重载成功
```

**注意事项**：
- 502表示后端无响应或拒绝连接，504表示后端处理超时
- Nginx的worker_connections受系统文件句柄限制（ulimit -n）约束
- upstream的keepalive连接可减少TCP握手开销，但需配置合理数量

## 三、面试真题

### 基础 高频 - Q1: MySQL出现连接超时，如何排查？

> **参考答案**：
> 1. **检查连接数**：执行`SHOW STATUS LIKE 'Threads_connected'`和`SHOW VARIABLES LIKE 'max_connections'`，确认是否达到最大连接数限制
> 2. **检查连接池**：确认应用连接池配置（最大连接数、连接超时时间）是否合理，是否存在连接泄漏
> 3. **检查网络**：在应用服务器执行`telnet <mysql-host> 3306`，确认网络连通性
> 4. **检查慢查询**：查看慢查询日志，确认是否有长事务占用连接不释放
> 5. **检查防火墙**：确认MySQL端口未被防火墙阻挡或连接数限制
>
> **延伸知识**：连接池大小建议公式为（核心数 * 2 + 有效磁盘数），单库连接数不宜超过1000

---

### 基础 高频 - Q2: Redis出现内存不足，如何处理？

> **参考答案**：
> 1. **检查内存使用**：执行`INFO memory`查看used_memory和maxmemory，确认是否达到上限
> 2. **检查淘汰策略**：执行`CONFIG GET maxmemory-policy`，确认是否为noeviction（不淘汰直接失败）
> 3. **检查大Key**：执行`--bigkeys`或`MEMORY USAGE <key>`定位占用内存大的Key
> 4. **优化方案**：设置合理的淘汰策略（allkeys-lru）、拆分大Key、设置Key过期时间
> 5. **扩容方案**：若数据量确实超过单节点容量，考虑Redis Cluster分片
>
> **延伸知识**：Redis内存碎片率（mem_fragmentation_ratio）大于1.5时需考虑重启或启用内存碎片整理

---

### 中等 高频 - Q3: Elasticsearch集群状态变为yellow/red，如何排查？

> **参考答案**：
> 1. **检查集群健康**：执行`/_cluster/health`，确认status和unassigned_shards数量
> 2. **检查节点状态**：执行`/_cat/nodes`，确认是否有节点离线
> 3. **检查磁盘水位**：执行`/_cat/allocation`，确认各节点磁盘是否超过85%（high watermark）
> 4. **yellow状态处理**：通常是有副本分片未分配，执行`/_cluster/reroute?retry_failed=true`重试分配
> 5. **red状态处理**：存在主分片未分配，需检查节点日志确认分片损坏情况，必要时从快照恢复
>
> **延伸知识**：ES集群发现机制基于Zen Discovery，最小主节点数（discovery.zen.minimum_master_nodes）需设置为N/2+1以避免脑裂

---

### 中等 高频 - Q4: Kafka消息堆积严重，如何排查和解决？

> **参考答案**：
> 1. **确认堆积量**：执行`kafka-consumer-groups.sh --describe`，查看各分区的LAG值
> 2. **检查消费者状态**：确认消费者应用是否正常运行，是否有消费异常日志
> 3. **检查消费速度**：对比CURRENT-OFFSET和LOG-END-OFFSET的增长速度，判断是生产过快还是消费过慢
> 4. **常见原因**：消费者数量不足、消费者处理逻辑慢、消费者频繁Rebalance、分区数过少
> 5. **解决方案**：增加消费者实例、优化消费逻辑、调整Rebalance参数、扩容Topic分区数
>
> **延伸知识**：消费者数量不应超过分区数，否则多余的消费者无法分配分区处于空闲状态

---

### 中等 中频 - Q5: Nginx出现502/504错误，如何排查？

> **参考答案**：
> 1. **检查错误日志**：查看`/var/log/nginx/error.log`，区分502（Bad Gateway）和504（Gateway Timeout）
> 2. **502排查**：后端服务未启动、后端端口不通、后端拒绝连接，用curl直接测试后端健康接口
> 3. **504排查**：后端处理超时，检查`proxy_read_timeout`设置，确认后端业务逻辑是否耗时过长
> 4. **检查连接限制**：确认`worker_connections`和系统`ulimit -n`是否足够支撑并发
> 5. **检查upstream配置**：确认后端地址正确，keepalive连接配置合理
>
> **延伸知识**：502通常是后端瞬间不可用（如重启），504通常是后端处理慢，两者根因不同

### 中等 中频 - Q6: RabbitMQ消息消费异常，如何排查？

> **参考答案**：
> 1. **检查队列状态**：管理界面或`rabbitmqctl list_queues`查看队列消息数、消费者数
> 2. **检查消费者连接**：`rabbitmqctl list_connections`确认消费者连接是否正常
> 3. **检查消息是否被Ack**：若消费者未发送Ack，消息会重新入队导致重复消费
> 4. **检查死信队列**：若配置了死信交换机，查看死信队列中是否有消息堆积
> 5. **检查内存和磁盘告警**：RabbitMQ在内存或磁盘超过阈值时会阻塞生产者
>
> **延伸知识**：RabbitMQ的镜像队列模式下，所有镜像节点都宕机时队列不可用，需关注镜像策略配置

### 困难 低频 - Q7: Hadoop NameNode HA切换失败，如何排查？

> **参考答案**：
> 1. **检查ZKFC状态**：执行`hdfs zkfc -formatZK`确认ZooKeeper连接正常，ZKFC进程运行中
> 2. **检查NameNode状态**：执行`hdfs haadmin -getServiceState nn1/nn2`，确认当前Active和Standby节点
> 3. **检查JournalNode**：确认JournalNode集群正常运行，EditLog同步无延迟
> 4. **检查ZooKeeper**：执行`zkCli.sh ls /hadoop-ha`，确认ZK上的选举状态正常
> 5. **手动切换**：执行`hdfs haadmin -failover --forcefence --forceactive nn1 nn2`强制切换
>
> **延伸知识**：NameNode HA依赖QJM（Quorum Journal Manager）或NFS共享存储，QJM更常用且无需额外存储

### 困难 中频 - Q8: RocketMQ消息发送失败或消费延迟，如何排查？

> **参考答案**：
> 1. **检查NameServer状态**：确认NameServer集群全部运行，路由信息正常
> 2. **检查Broker状态**：执行`mqadmin clusterList`，确认Broker是否全部注册到NameServer
> 3. **检查Topic路由**：执行`mqadmin topicRoute -t <topic>`，确认队列分布正常
> 4. **检查消息堆积**：执行`mqadmin consumerProgress -g <group>`，查看消费进度和堆积量
> 5. **检查Broker磁盘**：Broker磁盘超过85%会触发流控，拒绝新消息写入
>
> **延伸知识**：RocketMQ的Broker分Master和Slave，Slave仅同步数据不接收写入，Master宕机后需手动切换

### 困难 中频 - Q9: Nacos服务注册发现异常，如何排查？

> **参考答案**：
> 1. **检查Nacos集群状态**：登录Nacos控制台，查看集群节点列表和Leader节点
> 2. **检查服务注册**：控制台查看服务列表，确认服务是否成功注册
> 3. **检查心跳机制**：Nacos客户端每5秒发送心跳，服务端15秒无心跳标记为不健康，30秒剔除
> 4. **检查网络分区**：Nacos集群节点间网络不通会导致Raft选举失败，集群不可用
> 5. **检查配置推送**：订阅者列表确认客户端是否成功订阅，配置变更是否推送成功
>
> **延伸知识**：Nacos 2.x使用gRPC长连接替代1.x的HTTP短轮询，性能和实时性大幅提升

### 困难 高频 - Q10: 讲一个你处理过的中间件生产故障案例（STAR-R法）

> **参考答案**（以Kafka消息堆积导致业务延迟为例）：
>
> **Situation**：电商订单系统，Kafka Topic承载订单消息，日常QPS 5000，大促峰值QPS 50000。大促期间订单处理延迟从秒级增加到分钟级，用户投诉下单后长时间未收到确认
>
> **Task**：作为SRE工程师，定位消息堆积根因，恢复订单处理时效
>
> **Action**：
> 1. 执行`kafka-consumer-groups.sh`发现订单消费者组LAG达到200万条，且持续增长
> 2. 检查消费者应用日志，发现频繁Rebalance，消费线程不断重启
> 3. 根因分析：消费者数量从10台扩容到30台，但Topic只有12个分区，超出分区数的消费者无法分配分区，触发Rebalance
> 4. 紧急扩容Topic分区到48个，消费者重新平衡后LAG开始下降
> 5. 同时优化消费者处理逻辑，将同步数据库操作改为批量异步写入
>
> **Result**：消息堆积在30分钟内消化完毕，订单处理延迟恢复到秒级。大促期间未再出现堆积
>
> **Reflection**：消费者数量不应超过分区数，扩容消费者前必须先扩容分区。改进措施包括建立消费者-分区配比规范、大促前压测验证扩容方案、监控LAG告警阈值设置
>
> **延伸知识**：Kafka分区扩容只能增加不能减少，且不会重新分配已有数据，需提前规划分区数
