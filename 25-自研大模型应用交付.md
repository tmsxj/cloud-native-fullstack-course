# 模块25：自研大模型应用交付

---

## 一、理论核心

### 1.1 大模型应用交付架构

大模型应用交付不是简单的模型部署，而是涵盖模型推理服务、智能体编排、多租户隔离、成本治理的完整工程体系。运维工程师的核心职责是确保推理服务的高可用、弹性伸缩、可观测性和安全合规。

| 对比维度 | 传统微服务交付 | 大模型应用交付 | 运维关注点变化 |
|----------|-------------|--------------|--------------|
| **核心资源** | CPU/内存 | GPU显存/算力 | GPU资源调度、显存碎片管理 |
| **弹性指标** | QPS/CPU利用率 | Token吞吐量/队列长度 | 自定义HPA指标（Token/秒） |
| **冷启动** | 秒级 | 分钟级（模型加载） | 预热策略、模型缓存、常驻实例 |
| **版本管理** | 代码版本 | 模型权重+代码版本 | 模型版本注册、灰度发布、A/B测试 |
| **成本结构** | 按节点计费 | 按Token计费+GPU节点计费 | Token消耗监控、成本分摊 |
| **故障模式** | 服务崩溃 | 模型幻觉/推理超时/GPUOOM | 输出校验、超时熔断、显存监控 |

**关键要点**：
- 大模型推理服务的核心是 GPU 资源管理，显存碎片和利用率直接影响成本和性能
- 模型加载时间（冷启动）通常需要 30 秒到 5 分钟，必须通过预热池或常驻实例解决
- 推理服务的弹性伸缩不能只看 CPU，必须基于 Token 吞吐量、队列长度、GPU 利用率等自定义指标
- 模型版本和代码版本需要独立管理，支持模型热切换（不重启服务更换模型）

### 1.2 智能体（Agent）生命周期管理

智能体是大模型应用的交互形态，其生命周期管理涵盖开发、测试、发布、运行、下线五个阶段。运维团队需要建立标准化的 Agent 交付流水线。

| 生命周期阶段 | 运维动作 | 关键检查点 | 常用工具 |
|-------------|---------|-----------|---------|
| **开发** | 提供测试环境、配置模板 | 模型配置验证、Prompt安全检查 | kubectl dry-run、config validator |
| **测试** | 部署到测试命名空间、压测 | 功能正确性、延迟达标、Token消耗合理 | k6、locust、自定义压测脚本 |
| **发布** | 金丝雀发布、流量切换 | 错误率<1%、P99延迟<5s、无异常日志 | Argo Rollouts、Flagger |
| **运行** | 监控、告警、弹性伸缩 | GPU利用率、Token吞吐量、对话成功率 | Prometheus、Grafana、HPA |
| **下线** | 流量排空、资源清理 | 无活跃对话、数据备份完成 | kubectl drain、Velero |

**关键要点**：
- Agent 配置使用 Kubernetes CRD（Custom Resource Definition）定义，通过 Operator 模式实现自动化生命周期管理
- 金丝雀发布阶段必须监控三个核心指标：错误率、延迟、Token 消耗成本，任一指标异常自动回滚
- 智能体的优雅关闭需要等待活跃对话完成（terminationGracePeriodSeconds 建议设为 120-300 秒）
- 版本回滚不仅是代码回滚，还包括模型权重版本和配置版本的同步回滚

### 1.3 多租户隔离与资源配额

企业级大模型平台必须支持多租户隔离，确保不同团队/项目的模型服务互不干扰，同时实现成本分摊和配额管控。

| 隔离维度 | 实现方式 | 配置示例 |
|---------|---------|---------|
| **命名空间隔离** | 每个租户独立 Namespace | `tenant-a`、`tenant-b` |
| **网络隔离** | NetworkPolicy 限制跨租户通信 | 只允许同 Namespace Pod 互通 |
| **资源配额** | ResourceQuota 限制 CPU/内存/GPU/Pod 数 | 租户 A 最多 4 GPU、20 Pod |
| **存储隔离** | PVC 绑定到租户 Namespace | 模型缓存、日志存储独立 |
| **权限隔离** | RBAC 限制租户只能操作自己的资源 | Role + RoleBinding |
| **成本分摊** | 按 Namespace 标签聚合监控指标 | `tenant=tenant-a` 的 Token 消耗 |

**关键要点**：
- GPU 节点池建议按租户或按模型类型（大模型/小模型）划分，避免显存碎片
- ResourceQuota 必须同时限制 requests 和 limits，防止租户申请过多资源导致集群资源耗尽
- 模型缓存（如 HuggingFace 模型文件）使用 ReadWriteMany PVC 共享，减少重复下载
- 成本分摊通过 Prometheus 的 `agent_cost_dollars` 指标按租户标签聚合，生成月度成本报表

### 任务2：配置基于 Token 吞吐量的 HPA 弹性伸缩

**任务目标**：为推理服务配置自定义指标 HPA，基于 Token 吞吐量（tokens/second）和 GPU 利用率自动伸缩副本数。

**操作步骤**：

```bash
# 步骤1：部署Prometheus Adapter（将自定义指标暴露给K8s Metrics API）
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

helm install prometheus-adapter prometheus-community/prometheus-adapter \
    --namespace monitoring \
    --set prometheus.url=http://prometheus.monitoring.svc.cluster.local \
    --set prometheus.port=9090

# 步骤2：配置自定义指标规则（Token吞吐量和GPU利用率）
cat > adapter-config.yaml << 'YAMLEOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: adapter-config
  namespace: monitoring
data:
  config.yaml: |
    rules:
      # Token吞吐量指标（每秒生成的Token数）
      - seriesQuery: 'vllm:generation_tokens_total{namespace!="",pod!=""}'
        resources:
          overrides:
            namespace: {resource: "namespace"}
            pod: {resource: "pod"}
        name:
          matches: "^(.*)_total"
          as: "${1}_per_second"
        metricsQuery: 'sum(rate(<<.Series>>{<<.LabelMatchers>>}[2m])) by (<<.GroupBy>>)'
      
      # GPU显存利用率
      - seriesQuery: 'nvidia_gpu_memory_used_bytes{namespace!="",pod!=""}'
        resources:
          overrides:
            namespace: {resource: "namespace"}
            pod: {resource: "pod"}
        name:
          matches: "^(.*)_bytes"
          as: "${1}_utilization"
        metricsQuery: 'sum(<<.Series>>{<<.LabelMatchers>>}) by (<<.GroupBy>>) / sum(nvidia_gpu_memory_total_bytes{<<.LabelMatchers>>}) by (<<.GroupBy>>)'
YAMLEOF

kubectl apply -f adapter-config.yaml
kubectl rollout restart deployment prometheus-adapter -n monitoring

# 步骤3：验证自定义指标可用
kubectl get --raw "/apis/custom.metrics.k8s.io/v1beta1/namespaces/llm-production/pods/*/generation_tokens_per_second" | jq .
# 预期输出：包含指标名称和当前值的JSON

# 步骤4：创建基于自定义指标的HPA
cat > inference-hpa.yaml << 'YAMLEOF'
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: gemma-inference-hpa
  namespace: llm-production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: gemma-inference
  minReplicas: 2
  maxReplicas: 10
  metrics:
    # 基于Token吞吐量伸缩（目标：每个Pod处理1000 tokens/秒）
    - type: Pods
      pods:
        metric:
          name: generation_tokens_per_second
        target:
          type: AverageValue
          averageValue: "1000"
    # 基于GPU显存利用率伸缩（目标：70%）
    - type: Pods
      pods:
        metric:
          name: gpu_memory_used_utilization
        target:
          type: AverageValue
          averageValue: "0.7"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        - type: Percent
          value: 100
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60
YAMLEOF

kubectl apply -f inference-hpa.yaml
# 预期输出：horizontalpodautoscaler.autoscaling/gemma-inference-hpa created

# 步骤5：验证HPA状态
kubectl get hpa gemma-inference-hpa -n llm-production -w
# 预期输出：显示当前副本数、目标指标值和实际指标值

# 步骤6：压测验证弹性伸缩
# 使用locust或k6进行压测，观察HPA是否自动扩容
kubectl run load-test --rm -i --restart=Never \
    --image=busybox \
    -n llm-production \
    -- sh -c 'while true; do wget -qO- http://gemma-inference-svc:8000/v1/models; done'
```

**注意事项**：
- Prometheus Adapter 的 metricsQuery 必须正确匹配 Prometheus 中的指标名称和标签
- scaleDown 的 stabilizationWindowSeconds 设为 300 秒，防止流量波动导致频繁缩容
- GPU 节点扩容比 CPU 节点慢（通常需要 3-5 分钟），建议设置 minReplicas=2 保证基础容量
- 自定义指标 HPA 要求 Prometheus 和 Adapter 正常运行，需配置相应的监控告警

### 任务3：智能体金丝雀发布与 A/B 测试

**任务目标**：使用 Argo Rollouts 实现智能体服务的金丝雀发布，配置自动流量切换和指标验证。

**操作步骤**：

```bash
# 步骤1：安装Argo Rollouts
kubectl create namespace argo-rollouts
kubectl apply -n argo-rollouts -f https://github.com/argoproj/argo-rollouts/releases/latest/download/install.yaml

# 步骤2：创建Argo Rollout资源（替代传统Deployment）
cat > agent-canary-rollout.yaml << 'YAMLEOF'
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: customer-service-agent
  namespace: llm-production
  labels:
    app: customer-service-agent
spec:
  replicas: 4
  strategy:
    canary:
      canaryService: customer-service-agent-canary
      stableService: customer-service-agent-stable
      trafficRouting:
        managedRoutes:
          - name: primary
        ingress:
          className: gce
          name: customer-service-ingress
      steps:
        # 阶段1：5%流量，持续10分钟
        - setWeight: 5
        - pause: {duration: 10m}
        # 阶段2：25%流量，持续20分钟
        - setWeight: 25
        - pause: {duration: 20m}
        # 阶段3：50%流量，持续30分钟
        - setWeight: 50
        - pause: {duration: 30m}
        # 阶段4：100%流量
        - setWeight: 100
      analysis:
        templates:
          - templateName: agent-success-rate
        args:
          - name: service-name
            value: customer-service-agent-canary
  selector:
    matchLabels:
      app: customer-service-agent
  template:
    metadata:
      labels:
        app: customer-service-agent
    spec:
      containers:
        - name: agent
          image: asia-east1-docker.pkg.dev/my-project-id/llm-repo/customer-service-agent:v2.0.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "500m"
              memory: "1Gi"
            limits:
              cpu: "2000m"
              memory: "4Gi"
          env:
            - name: MODEL_VERSION
              value: "v2.0.0"
            - name: PROMPT_TEMPLATE
              value: "customer-service-v2"
---
apiVersion: v1
kind: Service
metadata:
  name: customer-service-agent-canary
  namespace: llm-production
spec:
  selector:
    app: customer-service-agent
  ports:
    - port: 8080
      targetPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: customer-service-agent-stable
  namespace: llm-production
spec:
  selector:
    app: customer-service-agent
  ports:
    - port: 8080
      targetPort: 8080
YAMLEOF

kubectl apply -f agent-canary-rollout.yaml
# 预期输出：rollout.argoproj.io/customer-service-agent created

# 步骤3：创建AnalysisTemplate（自动指标验证）
cat > agent-analysis.yaml << 'YAMLEOF'
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: agent-success-rate
  namespace: llm-production
spec:
  args:
    - name: service-name
  metrics:
    # 验证错误率<1%
    - name: error-rate
      interval: 1m
      count: 5
      successCondition: result[0] < 0.01
      provider:
        prometheus:
          address: http://prometheus.monitoring.svc.cluster.local:9090
          query: |
            sum(rate(agent_request_failure_total{service="{{args.service-name}}"}[2m]))
            /
            sum(rate(agent_requests_total{service="{{args.service-name}}"}[2m]))
    
    # 验证P99延迟<5秒
    - name: latency-p99
      interval: 1m
      count: 5
      successCondition: result[0] < 5000
      provider:
        prometheus:
          address: http://prometheus.monitoring.svc.cluster.local:9090
          query: |
            histogram_quantile(0.99,
              sum(rate(agent_response_latency_seconds_bucket{service="{{args.service-name}}"}[2m])) by (le)
            )
    
    # 验证Token消耗成本增长<20%
    - name: token-cost
      interval: 1m
      count: 5
      successCondition: result[0] < 1.2
      provider:
        prometheus:
          address: http://prometheus.monitoring.svc.cluster.local:9090
          query: |
            sum(rate(agent_tokens_total{service="{{args.service-name}}"}[2m]))
            /
            sum(rate(agent_tokens_total{service="customer-service-agent-stable"}[2m]))
YAMLEOF

kubectl apply -f agent-analysis.yaml
# 预期输出：analysistemplate.argoproj.io/agent-success-rate created

# 步骤4：触发金丝雀发布（更新镜像版本）
kubectl argo rollouts set image customer-service-agent \
    agent=asia-east1-docker.pkg.dev/my-project-id/llm-repo/customer-service-agent:v2.1.0 \
    -n llm-production
# 预期输出：rollout "customer-service-agent" image updated

# 步骤5：监控发布进度
kubectl argo rollouts get rollout customer-service-agent -n llm-production -w
# 预期输出：显示当前阶段、副本数、流量比例和分析结果

# 步骤6：如需手动回滚
kubectl argo rollouts undo customer-service-agent -n llm-production
# 预期输出：rollout 'customer-service-agent' undone
```

**注意事项**：
- Argo Rollouts 的 canary 策略支持自动和手动暂停，关键版本建议手动确认后再继续下一阶段
- AnalysisTemplate 中的 successCondition 必须合理设置，过于严格会导致正常发布被误判为失败
- 金丝雀发布期间需要同时监控新旧版本的指标对比，确保新版本性能不低于旧版本
- GKE Ingress 的流量拆分需要配置 managedRoutes，确保流量按比例分配到 canary 和 stable Service

### 任务4：多租户资源配额与成本分摊

**任务目标**：为两个租户（team-a 和 team-b）创建独立的 Namespace、ResourceQuota、NetworkPolicy，并配置按租户的成本监控。

**操作步骤**：

```bash
# 步骤1：创建租户Namespace和基础资源
cat > tenant-setup.yaml << 'YAMLEOF'
# 租户A配置
apiVersion: v1
kind: Namespace
metadata:
  name: tenant-a
  labels:
    tenant: tenant-a
    cost-center: engineering
---
apiVersion: v1
kind: ResourceQuota
metadata:
  name: tenant-a-quota
  namespace: tenant-a
```

### 1.4 行业痛点与运维挑战

自研大模型应用交付在 2025-2026 年已进入深水区，企业在落地过程中面临一系列共性的运维难题。

| 痛点类别 | 具体问题 | 影响范围 | 运维应对策略 |
|---------|---------|---------|------------|
| **GPU显存管理** | 模型加载后显存占用高，多模型并发时显存碎片严重，频繁触发 OOM | 推理服务崩溃、用户体验中断 | PagedAttention、显存预留策略（0.85）、模型量化（INT8/INT4） |
| **推理延迟不稳定** | 首Token延迟（TTFT）波动大，长上下文场景延迟指数级增长 | 用户流失、客服场景无法落地 | 连续批处理（Continuous Batching）、KV Cache预热、预热池 |
| **成本失控** | GPU节点费用+Token消耗双重计费，团队缺乏成本意识，月度账单超预算 3-5 倍 | 项目被砍、资源紧缩 | Token消耗监控、按租户成本分摊、Spot实例、自动缩容 |
| **冷启动慢** | 模型加载需 30 秒到 5 分钟，缩容后重新扩容用户等待时间过长 | 用户体验差、无法缩容节省成本 | minReplicas=2常驻、模型缓存共享、渐进式加载 |
| **版本管理混乱** | 模型权重、Prompt模板、代码版本各自为政，回滚时版本不一致 | 故障恢复困难、数据不一致 | 模型版本注册表（MLflow）、统一版本标签、原子化发布 |
| **输出不可控** | 模型幻觉、敏感信息泄露、Prompt注入攻击 | 合规风险、品牌损失 | 输出校验网关、内容安全过滤、输入审计日志 |
| **可观测性缺失** | 缺乏Token吞吐量、GPU利用率、推理延迟的专项监控，故障定位靠猜 | MTTR长、运维效率低 | vLLM指标采集、DCGM Exporter、自定义Grafana仪表盘 |
| **弹性伸缩失效** | HPA基于CPU伸缩，GPU利用率已100%但HPA未触发扩容 | 推理队列堆积、超时激增 | 自定义指标HPA（Token/秒）、KEDA事件驱动、Cluster Autoscaler |

**关键要点**：
- 行业调研显示，80% 的企业在大模型落地第一年都经历过至少一次 GPU OOM 导致的生产事故
- 成本失控是最普遍的痛点，某电商客户年 Token 成本从 800 万降到 380 万的案例说明优化空间巨大
- 可观测性是大模型运维的短板，多数团队仅监控 CPU/内存，忽略了 Token 吞吐量和 GPU 显存碎片

### 1.5 主流运维工具选型

大模型推理部署框架和运维工具在 2025-2026 年快速迭代，运维工程师需要理解各工具的定位和适用场景。

| 工具/框架 | 核心定位 | 性能亮点 | 适用场景 | 运维复杂度 |
|----------|---------|---------|---------|-----------|
| **vLLM** | 高吞吐生产级推理框架 | PagedAttention 减少显存碎片，Continuous Batching 提升吞吐 3-5 倍 | 生产环境主力推理，支持多模型并发 | 中 |
| **TensorRT-LLM** | NVIDIA 官方高性能推理引擎 | 量化推理（INT8/INT4）性能提升 2-4 倍，显存占用减少 50-75% | 追求极致性能，NVIDIA GPU 环境 | 高 |
| **SGLang** | 企业级高性能推理引擎 | RadixAttention 技术吞吐 5 倍于 vLLM，结构化输出快 10 倍 | 高并发 API 服务、结构化输出场景 | 中 |
| **TGI (Text Generation Inference)** | HuggingFace 官方推理服务 | 与 HuggingFace 生态深度集成，支持流式输出 | 快速原型验证、HF 模型一键部署 | 低 |
| **Ollama** | 轻量级本地化工具 | 一键本地运行，无需容器编排 | 开发测试、个人本地部署 | 低 |
| **KServe** | K8s 原生模型服务框架 | 标准化 InferenceService CRD，支持 A/B 测试、金丝雀发布 | K8s 环境模型服务编排 | 中 |
| **BentoML** | 模型服务统一框架 | 支持多框架模型（PyTorch/TensorFlow/ONNX），内置批处理 | 多模型统一服务层 | 中 |
| **MLflow** | 模型生命周期管理 | 模型版本注册、实验追踪、模型打包 | 模型版本管理、实验对比 | 低 |
| **OpenLLMetry** | LLM 可观测性工具 | 自动采集 Token 消耗、延迟、成本，支持 20+ LLM 提供商 | LLM 应用性能监控和成本追踪 | 低 |
| **Langfuse** | LLM 应用可观测平台 | 对话追踪、Prompt 版本管理、成本分析、用户反馈收集 | 对话类应用全链路追踪 | 低 |

**选型决策树**：
- **追求吞吐量和稳定性** → vLLM（生产环境首选，社区活跃，文档完善）
- **追求极致性能且用 NVIDIA GPU** → TensorRT-LLM（需额外编译，性能最优）
- **高并发结构化输出（JSON/XML）** → SGLang（RadixAttention 对结构化输出优化显著）
- **快速验证和原型** → TGI 或 Ollama（开箱即用，无需复杂配置）
- **K8s 原生编排** → KServe + vLLM（标准化 CRD，支持 GitOps）

## 三、面试真题

### 基础 高频 - Q1: 大模型推理服务和传统微服务在运维上有什么区别？

> **参考答案**：
> 1. **资源维度不同**：传统服务关注 CPU/内存，推理服务核心关注 GPU 显存和算力，需要专门的 GPU 调度策略
> 2. **弹性指标不同**：传统服务用 QPS/CPU 做 HPA，推理服务需要用 Token 吞吐量、队列长度、GPU 利用率等自定义指标
> 3. **冷启动问题**：模型加载需要 30 秒到 5 分钟，必须通过预热池或常驻实例解决，不能像传统服务缩到零
> 4. **成本结构不同**：推理服务成本 = GPU 节点费用 + Token 消耗费用，需要双重成本监控
> 5. **故障模式不同**：除了服务崩溃，还有模型幻觉、推理超时、GPU OOM 等特殊故障

---

### 中等 高频 - Q2: 如何实现大模型推理服务的弹性伸缩？

> **参考答案**：
> 1. **自定义指标采集**：vLLM 暴露 Token 吞吐量、队列长度等指标，通过 Prometheus Adapter 暴露给 K8s Metrics API
> 2. **HPA 配置**：基于 Token/秒 和 GPU 利用率设置双指标伸缩，minReplicas=2 保证基础容量
> 3. **节点自动扩容**：Cluster Autoscaler 监控 GPU 节点池，资源不足时自动创建新 GPU 节点
> 4. **预热策略**：设置 minReplicas=2 避免冷启动，或使用 KEDA 的 ScaledObject 实现事件驱动预热
> 5. **缩容保护**：scaleDown 的 stabilizationWindowSeconds 设为 300 秒，防止流量波动导致频繁缩容
>
> **延伸知识**：GPU 节点扩容比 CPU 慢 3-5 分钟，建议配合预测性扩容（基于历史流量模式提前扩容）

---

### 中等 高频 - Q3: 智能体的金丝雀发布和传统应用的蓝绿发布有什么区别？

> **参考答案**：
> 1. **验证指标不同**：传统应用验证错误率和延迟，智能体还需要验证 Token 消耗成本、输出质量、对话成功率
> 2. **发布粒度不同**：传统应用按副本比例切流量，智能体可以按用户特征分流（如 beta 用户 100% 到新版本）
> 3. **回滚复杂度不同**：智能体回滚需要同步回滚模型权重版本、Prompt 模板和代码版本
> 4. **工具依赖**：智能体金丝雀发布需要 Argo Rollouts + AnalysisTemplate 自动验证多维度指标
> 5. **优雅关闭**：智能体需要等待活跃对话完成（terminationGracePeriodSeconds 120-300 秒），传统应用通常 30 秒即可

---

### 困难 中频 - Q4: 如何设计企业级大模型平台的多租户隔离方案？

> **参考答案**：
> 1. **命名空间隔离**：每个租户独立 Namespace，通过 ResourceQuota 限制 CPU/内存/GPU/Pod 数量
> 2. **网络隔离**：NetworkPolicy 限制跨租户通信，只允许同 Namespace Pod 互通和 DNS 查询
> 3. **权限隔离**：RBAC 配置租户管理员只能操作自己的 Namespace，禁止跨租户操作
> 4. **资源隔离**：GPU 节点池按租户或模型类型划分，避免显存碎片和资源争抢
> 5. **成本分摊**：Prometheus 按租户标签聚合 Token 消耗、GPU 使用时长等指标，生成月度成本报表
> 6. **模型缓存共享**：ReadWriteMany PVC 共享模型文件，减少重复下载，但需确保缓存数据不涉及租户隐私

### 中等 中频 - Q5: 大模型推理服务的可观测性需要采集哪些核心指标？

> **参考答案**：
> 1. **推理性能指标**：Token 吞吐量（tokens/秒）、首 Token 延迟（TTFT）、每 Token 生成时间（TPOT）、队列长度
> 2. **资源指标**：GPU 利用率、GPU 显存使用量/总量、CPU 使用率、内存使用率
> 3. **业务指标**：请求成功率、错误率、对话轮数、知识库查询次数、工具调用次数
> 4. **成本指标**：Token 消耗总量、累计成本（agent_cost_dollars）、按租户/模型维度的成本分摊
> 5. **日志和追踪**：结构化推理日志（输入/输出/延迟/Token 数）、分布式链路追踪（请求全链路）
>
> **延伸知识**：vLLM 框架默认暴露 /metrics 端点，包含大部分推理性能指标；GPU 指标需要 NVIDIA DCGM Exporter

### 困难 低频 - Q6: 如何优化大模型推理服务的 GPU 显存利用率？

> **参考答案**：
> 1. **PagedAttention**：使用 vLLM 的 PagedAttention 技术，将 KV Cache 分页管理，减少显存碎片
> 2. **量化推理**：使用 INT8/INT4 量化模型，显存占用减少 50-75%，推理速度提升 2-4 倍
> 3. **Tensor 并行**：多 GPU 张量并行（--tensor-parallel-size），支持更大模型和更长上下文
> 4. **显存预留策略**：设置 --gpu-memory-utilization=0.85，预留 15% 显存给临时计算和碎片
> 5. **动态批处理**：启用 Continuous Batching，动态合并多个请求的 KV Cache，提升吞吐量
> 6. **模型共享**：多个推理 Pod 共享同一份模型文件（ReadWriteMany PVC），减少重复加载

### 基础 中频 - Q7: 大模型应用交付的 CI/CD 流水线和传统应用有什么不同？

> **参考答案**：

> 3. **镜像构建**：推理镜像通常较大（5-20GB），需要优化层缓存，模型文件通过 PVC 挂载而非打包进镜像
> 4. **部署策略**：支持模型热切换（不重启服务更换模型），以及金丝雀发布按流量比例切分
> 5. **回滚机制**：需要同时回滚代码、模型权重和配置（Prompt 模板、温度参数等）
>
> **延伸知识**：GitOps 工具（ArgoCD/Flux）可以直接管理模型配置 CRD，实现声明式模型部署

### 困难 高频 - Q9: 生产环境大模型推理服务 GPU OOM 故障如何排查和解决？

> **参考答案**：
> 1. **现象确认**：Pod 状态为 OOMKilled（Exit Code 137），kubectl describe pod 显示 `Last State: Terminated, Reason: OOMKilled`
> 2. **根因定位**：
>    - 检查模型参数：`--max-model-len` 设置过大导致 KV Cache 占用过多显存
>    - 检查并发请求数：同时处理的请求过多，累计 KV Cache 超出显存容量
>    - 检查显存碎片：多模型并发或频繁加载/卸载模型导致显存碎片
>    - 检查 GPU 显存监控：nvidia-smi 或 DCGM Exporter 查看显存使用趋势
> 3. **紧急处理**：
>    - 降低 `--max-model-len` 到安全值（如 4096）
>    - 降低 `--gpu-memory-utilization` 到 0.75，预留更多显存缓冲
>    - 限制并发请求数（`--max-num-seqs`），避免同时处理过多请求
>    - 重启推理 Pod，清理显存碎片
> 4. **长期优化**：
>    - 启用 PagedAttention（vLLM 默认开启），减少 KV Cache 碎片
>    - 模型量化（INT8/INT4），显存占用减少 50-75%
>    - 升级 GPU（如 T4 → L4 → A100），获取更大显存
>    - 多 GPU 张量并行（`--tensor-parallel-size=2`），分散显存压力
>
> **延伸知识**：80% 的 GPU OOM 故障是由于 `--max-model-len` 设置过大或并发请求数未限制导致的

### 中等 高频 - Q10: 推理服务冷启动慢（模型加载 3-5 分钟）如何优化？

> **参考答案**：
> 1. **常驻实例策略**：设置 `minReplicas=2`，保持至少 2 个 Pod 常驻，避免缩容到零
> 2. **模型缓存共享**：使用 ReadWriteMany PVC 挂载模型文件，多个 Pod 共享同一份缓存，避免重复下载
> 3. **渐进式加载**：使用 vLLM 的 `--load-format=auto` 和 `--dtype=half`，减少模型加载时的内存拷贝
> 4. **预热池**：在流量低峰期预先加载模型到 GPU，高峰期直接接收请求
> 5. **模型切片**：将大模型切分为多个小文件并行加载，减少单文件加载时间
> 6. **SSD 缓存**：使用本地 NVMe SSD 作为模型缓存层，加载速度比网络存储快 5-10 倍
>
> **延伸知识**：冷启动问题在 Serverless 场景（如 Cloud Run）尤为突出，建议生产环境避免将推理服务缩到零

### 困难 中频 - Q11: 大模型推理服务成本从月均 50 万降到 20 万的优化路径是什么？

> **参考答案**：
> 1. **模型量化**：INT8 量化减少 50% 显存占用，同样 GPU 可部署更多模型实例，节点数减少 40%
> 2. **Spot 实例**：GPU 节点使用 Spot/抢占式实例，成本降低 60-70%，配合 Pod 反亲和性保证高可用
> 3. **弹性缩容**：配置基于 Token 吞吐量的 HPA，低峰期自动缩容，避免资源闲置
> 4. **批处理优化**：启用 Continuous Batching，单 GPU 吞吐量提升 3-5 倍，相同 QPS 所需 GPU 减少
> 5. **模型选型**：用 7B 模型替代 70B 模型处理 80% 的常规请求，仅对复杂请求调用大模型
> 6. **缓存策略**：对高频查询结果设置 Redis 缓存，减少重复推理，Token 消耗降低 30%
> 7. **成本监控**：OpenCost + Prometheus 按租户/模型维度监控，识别高成本请求并优化
>
> **延伸知识**：某电商客户通过上述组合策略，年 Token 成本从 800 万降到 380 万，推理延迟还下降了 20%

### 中等 中频 - Q12: vLLM、TensorRT-LLM、SGLang 三大推理框架如何选型？

> **参考答案**：
> 1. **vLLM**：生产环境首选，PagedAttention + Continuous Batching 成熟稳定，社区活跃，文档完善，适合多模型并发场景
> 2. **TensorRT-LLM**：NVIDIA 官方出品，量化推理性能最优（INT8/INT4），但需额外编译，运维复杂度高，适合追求极致性能的场景
> 3. **SGLang**：新兴高性能框架，RadixAttention 技术吞吐 5 倍于 vLLM，结构化输出（JSON/XML）快 10 倍，适合高并发 API 服务
> 4. **选型原则**：
>    - 快速上线、团队经验少 → vLLM
>    - NVIDIA GPU、性能敏感、有编译能力 → TensorRT-LLM
>    - 高并发、结构化输出、愿意尝试新技术 → SGLang
>    - K8s 原生编排 → KServe + vLLM
>
> **延伸知识**：2026 年实测显示，vLLM 和 TensorRT-LLM 撑起了 80% 以上的生产环境推理部署
