# 模块17：平台工程与多租户

## 1. 概述与架构图

### 1.1 什么是平台工程 (Platform Engineering)

平台工程是一门设计和构建工具链与工作流的学科，为云原生时代的软件工程组织提供自助服务(Self-Service)能力。它通过产品化的内部开发者平台(IDP, Internal Developer Platform)减少开发者的认知负担，让开发者能够专注于业务代码而非基础设施。

**核心价值：**

| 价值点 | 说明 |
|--------|------|
| **自助服务** | 开发者无需运维介入即可获取所需资源 |
| **标准化** | 通过"黄金路径"确保最佳实践的一致性 |
| **效率提升** | 减少重复工作，加速应用交付 |
| **认知减负** | 开发者只需关注业务逻辑 |
| **治理合规** | 内置安全、成本、合规策略 |

### 1.2 平台工程 vs DevOps

```
+================================================================================+
|                    从 DevOps 到平台工程的演进                                   |
+================================================================================+
|                                                                                |
|   DevOps 模式 (2010-2020)              平台工程模式 (2020+)                    |
|   +------------------------+           +------------------------+              |
|   | "You Build It,         |           | "You Build It,         |              |
|   |  You Run It"           |           |  Platform Runs It"     |              |
|   |                        |           |                        |              |
|   | 每个团队自建工具链      |           | 统一平台提供标准化服务  |              |
|   | 高认知负担              |           | 低认知负担              |              |
|   | 重复造轮子              |           | 复用平台能力            |              |
|   +------------------------+           +------------------------+              |
|                                                                                |
|   演进驱动力:                                                                  |
|   - K8s复杂度增加，开发者难以掌握全部细节                                       |
|   - 多团队需要一致的基础设施体验                                                |
|   - 运维团队转型为平台团队，服务内部客户                                        |
|                                                                                |
+================================================================================+
```

**关键区别：**

| 维度 | DevOps | 平台工程 |
|------|--------|----------|
| **关注点** | 开发与运维协作 | 开发者体验与自助服务 |
| **交付物** | 流水线与自动化 | 内部开发者平台(IDP) |
| **用户** | 开发与运维团队 | 开发者(内部客户) |
| **核心能力** | CI/CD、监控 | 自助服务、黄金路径 |
| **组织角色** | 嵌入式SRE | 平台产品经理+工程师 |

## 2. 核心概念

### 2.1 多租户模型 (Namespace as a Service，命名空间即服务)

```
+----------------------------------------------------------+
|                Namespace as a Service 模型                |
+----------------------------------------------------------+
|                                                          |
|  软多租户 (Soft Multi-tenancy)                            |
|  +--------------------------------------------------+   |
|  |  - 共享Worker Nodes                               |   |
|  |  - 通过Namespace隔离                              |   |
|  |  - 适合可信团队(同一组织)                         |   |
|  |  - 资源开销低                                     |   |
|  +--------------------------------------------------+   |
|                                                          |
|  硬多租户 (Hard Multi-tenancy)                            |
|  +--------------------------------------------------+   |
|  |  - 物理隔离(独立集群/节点池)                      |   |
|  |  - 强安全边界                                     |   |
|  |  - 适合不可信租户(SaaS场景)                       |   |
|  |  - 资源开销高                                     |   |
|  +--------------------------------------------------+   |
|                                                          |
|  混合多租户 (Hybrid)                                      |
|  +--------------------------------------------------+   |
|  |  - 敏感工作负载独占节点池                         |   |
|  |  - 普通工作负载共享节点                           |   |
|  |  - 平衡安全与成本                                 |   |
|  +--------------------------------------------------+   |
|                                                          |
+----------------------------------------------------------+
```

**多租户隔离维度：**

| 隔离维度 | 实现机制 | 说明 |
|----------|----------|------|
| **计算隔离** | ResourceQuota/LimitRange | 限制CPU/内存使用 |
| **网络隔离** | NetworkPolicy | 控制Pod间流量 |
| **存储隔离** | StorageClass + PVC | 独立存储空间 |
| **权限隔离** | RBAC | 控制API访问 |
| **安全隔离** | PodSecurityAdmission | 强制执行安全策略 |

### 2.3 黄金路径 (Golden Path)

黄金路径是平台团队为常见用例提供的标准化、受支持的开发路径(Golden Path)。

```
+----------------------------------------------------------+
|                    黄金路径示例                             |
+----------------------------------------------------------+
|                                                          |
|  场景: 部署一个Spring Boot微服务                          |
|                                                          |
|  传统方式 (认知负担高)          黄金路径 (认知负担低)      |
|  +----------------------+      +----------------------+  |
|  | 1. 创建Dockerfile    |      | 1. 选择模板          |  |
|  | 2. 配置CI/CD         |      | 2. 填写参数          |  |
|  | 3. 编写K8s YAML      |      | 3. 一键部署          |  |
|  | 4. 配置监控告警      |      |                      |  |
|  | 5. 配置日志收集      |      | 自动包含:            |  |
|  | 6. 配置网络策略      |      | - Dockerfile         |  |
|  | 7. ...               |      | - CI/CD流水线        |  |
|  +----------------------+      | - K8s manifests      |  |
|                                | - 监控配置           |  |
|                                | - 日志配置           |  |
|                                | - 安全策略           |  |
|                                +----------------------+  |
|                                                          |
+----------------------------------------------------------+
```

### 2.4 开发者体验 (DX, Developer Experience)

开发者体验(DX)衡量开发者使用平台完成工作的效率和满意度。

| DX维度 | 指标 | 优化方向 |
|--------|------|----------|
| **时间效率** | 从代码到生产的时间 | 自动化、模板化 |
| **认知负担** | 需要了解的K8s概念数量 | 抽象、简化 |
| **反馈速度** | 构建/测试/部署反馈 | 缓存、并行化 |
| **故障恢复** | 平均恢复时间(MTTR) | 可观测性、自动化 |
| **文档质量** | 文档完整度和准确性 | 文档即代码 |



#### 4.1.2 ResourceQuota资源配额

```yaml
# resource-quota.yaml
apiVersion: v1
kind: ResourceQuota  # 资源配额，限制命名空间资源上限
metadata:
  name: team-alpha-quota
  namespace: team-alpha-prod
spec:
  hard:
    # 计算资源
    requests.cpu: "10"  # CPU请求上限
    requests.memory: 20Gi  # 内存请求上限
    limits.cpu: "20"
    limits.memory: 40Gi
    
    # 存储资源
    requests.storage: 100Gi
    persistentvolumeclaims: "10"
    
    # 对象数量限制
    pods: "50"
    services: "20"
    services.loadbalancers: "2"
    services.nodeports: "5"
    secrets: "50"
    configmaps: "50"
    
    # 特殊资源
    count/deployments.apps: "30"
    count/statefulsets.apps: "10"
    count/ingresses.networking.k8s.io: "10"
```

```bash
# 应用ResourceQuota
kubectl apply -f resource-quota.yaml

# 查看配额使用情况
kubectl describe resourcequota team-alpha-quota -n team-alpha-prod

# 查看所有Namespace配额
kubectl get resourcequota --all-namespaces
```

#### 4.1.3 LimitRange默认限制

```yaml
# limit-range.yaml
apiVersion: v1
kind: LimitRange  # 默认资源限制，自动为Pod设置资源上下限
metadata:
  name: team-alpha-limits
  namespace: team-alpha-prod
spec:
  limits:
  # 容器默认限制
  - default:
      cpu: "500m"  # 默认CPU限制
      memory: "512Mi"  # 默认内存限制
    defaultRequest:
      cpu: "100m"  # 默认CPU请求
      memory: "128Mi"  # 默认内存请求
    max:
      cpu: "2"
      memory: "4Gi"
    min:
      cpu: "50m"
      memory: "64Mi"
    type: Container
    
  # Pod级别限制
  - max:
      cpu: "4"
      memory: "8Gi"
    min:
      cpu: "50m"
      memory: "64Mi"
    type: Pod
    
  # PVC存储限制
  - max:
      storage: 50Gi
    min:
      storage: 1Gi
    default:
      storage: 10Gi
    defaultRequest:
      storage: 10Gi
    type: PersistentVolumeClaim
```

```bash
# 应用LimitRange
kubectl apply -f limit-range.yaml

# 查看LimitRange
kubectl describe limitrange team-alpha-limits -n team-alpha-prod
```

#### 4.1.4 NetworkPolicy网络隔离

```yaml
# default-deny-all.yaml
# 默认拒绝所有入站和出站流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy  # 网络策略，控制Pod间流量
metadata:
  name: default-deny-all  # 默认拒绝所有流量
  namespace: team-alpha-prod
spec:
  podSelector: {}  # 选择所有Pod
  policyTypes:
  - Ingress
  - Egress
---
# allow-same-namespace.yaml
# 允许同一Namespace内的Pod通信
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-same-namespace
  namespace: team-alpha-prod
spec:
  podSelector: {}
  ingress:
  - from:
    - podSelector: {}
  egress:
  - to:
    - podSelector: {}
---
# allow-dns.yaml
# 允许DNS查询
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: team-alpha-prod
spec:
  podSelector: {}
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
---
# allow-ingress-frontend.yaml
# 允许从Ingress Controller访问前端服务
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-ingress-frontend
  namespace: team-alpha-prod
spec:
  podSelector:
    matchLabels:
      app: frontend
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: ingress-nginx
    ports:
    - protocol: TCP
      port: 8080
```

```bash
# 应用网络策略
kubectl apply -f default-deny-all.yaml
kubectl apply -f allow-same-namespace.yaml
kubectl apply -f allow-dns.yaml
kubectl apply -f allow-ingress-frontend.yaml

# 查看网络策略
kubectl get networkpolicy -n team-alpha-prod
```

```yaml
# 配置管理
- apiGroups: [""]
  resources: ["configmaps", "secrets"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  
# 服务发现
- apiGroups: [""]
  resources: ["services", "endpoints"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  
# Ingress管理
- apiGroups: ["networking.k8s.io"]
  resources: ["ingresses"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  
# 只读访问ResourceQuota和LimitRange
- apiGroups: [""]
  resources: ["resourcequotas", "limitranges"]

# 只读角色

  name: viewer
  namespace: team-alpha-prod

- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["get", "list", "watch"]

# 集群角色 - Namespace管理员(跨Namespace创建权限)

  name: namespace-admin
rules:
- apiGroups: [""]
  resources: ["namespaces"]

- apiGroups: [""]
  resources: ["resourcequotas", "limitranges"]
  verbs: ["*"]
```

## 6. 故障排查案例

### 案例1: ResourceQuota超限导致Pod无法创建

**现象：**
开发者报告无法创建新的Deployment，Pod处于Pending状态。

**排查过程：**
```bash
# 1. 查看Pod状态
kubectl get pods -n team-alpha-prod
# 输出: pod/my-app-xxx   0/1   Pending   0   10m

# 2. 查看Pod事件
kubectl describe pod my-app-xxx -n team-alpha-prod
# 输出: Warning  FailedScheduling  ...  0/3 nodes are available: 
#       3 Insufficient cpu, 3 Insufficient memory.

# 3. 检查ResourceQuota使用情况
kubectl describe resourcequota team-alpha-quota -n team-alpha-prod
# 输出:
#   limits.cpu:     18/20
#   limits.memory:  38Gi/40Gi
#   pods:           48/50

# 4. 检查Namespace内所有资源
kubectl get all -n team-alpha-prod
```

**根因：**
Namespace的ResourceQuota已接近上限，新Pod无法调度。

**解决方案：**
```bash
# 方案1: 清理无用资源
kubectl delete pods --field-selector=status.phase=Failed -n team-alpha-prod
kubectl delete pods --field-selector=status.phase=Succeeded -n team-alpha-prod

# 方案2: 申请增加配额
cat > quota-increase.yaml << 'EOF'
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-alpha-quota
  namespace: team-alpha-prod
spec:
  hard:
    limits.cpu: "30"      # 从20增加到30
    limits.memory: 60Gi   # 从40Gi增加到60Gi
    pods: "80"            # 从50增加到80
EOF
kubectl apply -f quota-increase.yaml

# 方案3: 优化资源请求
# 检查是否有Pod配置了过高的资源请求
kubectl top pods -n team-alpha-prod
```

### 案例2: NetworkPolicy误配置导致服务不可访问

**现象：**
前端服务无法访问后端API，连接超时。

**排查过程：**
```bash
# 1. 测试Pod间连通性
kubectl exec -it frontend-pod -n team-alpha-prod -- curl backend-service:8080
# 输出: curl: (28) Connection timed out

# 2. 检查Service状态
kubectl get svc backend-service -n team-alpha-prod
# 输出: backend-service   ClusterIP   10.96.123.45   <none>   8080/TCP

# 3. 检查NetworkPolicy
kubectl get networkpolicy -n team-alpha-prod
# 输出:
# NAME              POD-SELECTOR   AGE
# default-deny-all  <none>         2d
# allow-frontend    app=frontend   1d

# 4. 查看详细策略
kubectl describe networkpolicy default-deny-all -n team-alpha-prod
# 输出: 拒绝所有入站和出站流量

kubectl describe networkpolicy allow-frontend -n team-alpha-prod
# 输出: 只允许frontend标签的Pod入站
```

**根因：**
default-deny-all策略阻止了所有流量，但缺少允许后端服务接收请求的策略。

**解决方案：**
```yaml
# allow-backend.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-backend-ingress
  namespace: team-alpha-prod
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

```bash
kubectl apply -f allow-backend.yaml
```

### 案例3: RBAC权限不足导致CI/CD失败

**现象：**
CI/CD流水线在部署阶段失败，报错权限不足。

**排查过程：**
```bash
# 1. 查看CI/CD使用的ServiceAccount
kubectl get serviceaccount -n team-alpha-prod | grep ci
# 输出: ci-sa

# 2. 检查ServiceAccount绑定的角色
kubectl get rolebinding -n team-alpha-prod
# 输出: ci-sa-binding

kubectl describe rolebinding ci-sa-binding -n team-alpha-prod
# 输出: 绑定到Role: developer

# 3. 检查Role权限
kubectl describe role developer -n team-alpha-prod
# 输出: 缺少ingresses的create权限

# 4. 验证权限
kubectl auth can-i create ingresses --as=system:serviceaccount:team-alpha-prod:ci-sa -n team-alpha-prod
# 输出: no
```

**根因：**
CI/CD使用的ServiceAccount缺少创建Ingress的权限。

**解决方案：**
```yaml
# ci-role-patch.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: ci-deployer
  namespace: team-alpha-prod
rules:
# 继承developer角色权限
- apiGroups: ["apps"]
  resources: ["deployments", "statefulsets", "replicasets"]
  verbs: ["*"]
- apiGroups: [""]
  resources: ["pods", "services", "configmaps", "secrets"]
  verbs: ["*"]
# 增加Ingress管理权限
- apiGroups: ["networking.k8s.io"]
  resources: ["ingresses"]
  verbs: ["*"]
# 增加HPA权限
- apiGroups: ["autoscaling"]
  resources: ["horizontalpodautoscalers"]
  verbs: ["*"]
```

```bash
kubectl apply -f ci-role-patch.yaml
kubectl create rolebinding ci-sa-deployer-binding \
  --role=ci-deployer \
  --serviceaccount=team-alpha-prod:ci-sa \
  -n team-alpha-prod
```

## 7. 高频面试题

### Q1: 什么是平台工程？与DevOps有什么区别？

**答案要点：**
- 平台工程是设计和构建自助服务能力的学科，通过内部开发者平台(IDP)减少开发者认知负担
- DevOps关注开发与运维的协作文化，平台工程关注开发者体验和产品化平台
- 平台工程是DevOps的演进，解决K8s复杂度增加带来的新挑战
- 平台团队作为内部服务提供商，开发者作为内部客户

### Q2: Kubernetes多租户有哪些实现模式？各自的优缺点是什么？

**答案要点：**
- **软多租户(Soft Multi-tenancy)**：通过Namespace隔离，共享Worker节点
  - 优点：资源利用率高，成本低
  - 缺点：安全边界较弱，适合可信团队
- **硬多租户(Hard Multi-tenancy)**：物理隔离，独立集群或节点池
  - 优点：强安全边界，适合SaaS场景
  - 缺点：资源开销高，管理复杂
- **混合多租户(Hybrid)**：敏感负载独占节点，普通负载共享
  - 优点：平衡安全与成本
  - 缺点：调度复杂度增加

### Q3: 如何设计一个安全的Namespace多租户方案？

**答案要点：**
- **资源隔离**：ResourceQuota限制资源使用，LimitRange设置默认值
- **网络隔离**：NetworkPolicy默认拒绝，显式允许必要流量
- **权限隔离**：RBAC最小权限原则，区分人类用户和ServiceAccount
- **安全策略**：PodSecurityAdmission强制执行安全标准
- **审计监控**：启用审计日志，监控异常行为

### Q4: 什么是"黄金路径"(Golden Path)？如何在平台工程中实现？

**答案要点：**
- 黄金路径是平台团队为常见用例提供的标准化、受支持的开发路径
- 通过软件模板(Software Templates)实现，如Backstage Scaffolder
- 包含：项目脚手架、CI/CD配置、K8s manifests、监控告警、安全策略
- 目标是减少开发者认知负担，确保最佳实践的一致性
- 需要平台团队持续维护和更新

### Q5: 如何管理多租户环境下的临时权限提升？

**答案要点：**
- 使用RoleBinding/ClusterRoleBinding的annotations记录权限有效期
- 定期审查和清理过期权限
- 使用Just-in-Time (JIT) 权限管理系统
- 记录权限提升的原因和审批人
- 结合审计日志监控特权操作
- 考虑使用工具如Pomerium、Teleport实现临时访问

## 8. 生产环境建议

### 8.1 平台工程成熟度模型

| 成熟度 | 特征 | 关键指标 |
|--------|------|----------|
| **Level 1** | 基础自动化 | 手动脚本，无自助服务 |
| **Level 2** | 部分自助服务 | 部分模板，文档化流程 |
| **Level 3** | 完整IDP | 统一门户，黄金路径 |
| **Level 4** | 智能平台 | AI辅助，自动优化 |
| **Level 5** | 生态平台 | 开发者社区，内部开源 |
