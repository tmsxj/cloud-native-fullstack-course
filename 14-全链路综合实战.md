### 1.3 请求链路追踪

```
用户请求: POST /api/orders (创建订单)

Ingress (Nginx) -> API Gateway (Spring Cloud Gateway)
  -> Order Service (创建订单)
    -> User Service (查询用户信息) [Nacos 服务发现]
    -> Inventory Service (扣减库存) [Kafka 异步消息]
    -> Payment Service (发起支付) [Kafka 异步消息]
      -> 支付结果回调
    -> Order Service (更新订单状态)

Trace 链路:
[Ingress] -> [API Gateway] -> [Order Service] -> [User Service]
                                       -> [Inventory Service] -> [Kafka] -> [Payment Service]

每个 Span 包含:
- TraceID: 全局唯一
- SpanID: 当前操作 ID
- ParentSpanID: 父操作 ID
- Duration: 操作耗时
- Tags: HTTP method, URL, status code, service name
- Logs: 错误信息、业务日志
```

## 2. 理论基础

### 2.1 微服务全链路核心原则

| 原则 | 说明 |
|------|------|
| 服务拆分 | 按业务领域拆分，每个服务独立部署、独立数据库 |
| 声明式配置 | 所有基础设施和应用配置以 YAML 形式存储在 Git 中 |
| 自动化交付 | CI/CD 全自动化，从代码提交到生产部署无需人工干预 |
| 可观测性 | Trace + Metrics + Logs 三大支柱覆盖全链路 |
| 弹性设计 | 超时、重试、熔断、降级、限流 |
| 数据一致性 | 基于消息队列的最终一致性 |
| 安全纵深 | mTLS（双向 TLS 认证） + RBAC（基于角色的访问控制） + NetworkPolicy（网络策略） + Secret 管理 |

### 2.2 各模块协作关系

```
模块协作矩阵:

                    Ingress/Istio    Nacos       Kafka      OTel       ArgoCD     Tekton
Ingress/Istio        -            服务发现     -         Trace采集   应用交付    -
Nacos              服务注册          -         -         -         -         -
Kafka              事件驱动        服务发现      -         Trace采集   -         -
OTel               Trace生成      配置中心     Trace采集    -         -         -
ArgoCD             应用暴露        配置管理      -         -          -        CI产出
Tekton              -              -         -         -         Git变更     -
```

# --- 阶段二：运行时镜像 ---
FROM eclipse-temurin:17-jre-alpine  # 运行阶段: JRE 精简镜像
LABEL maintainer="mall-team"

# 创建非 root 用户
RUN addgroup -S app && adduser -S app -G app
USER app  # 非 root 用户运行

WORKDIR /app

# 从构建阶段复制 JAR
COPY --from=builder /build/target/*.jar app.jar

# OTel Java Agent（离线环境需提前导入到 Harbor）
ENV OTEL_JAVA_AGENT_PATH=/app/opentelemetry-javaagent.jar
# 如果 OTel Agent 已预置在基础镜像中则无需额外 COPY
# COPY --from=otel /app/opentelemetry-javaagent.jar /app/

#### 3.4.3 离线构建方案

**方案一：有网机器构建 + 导入（推荐）**

```bash
#!/bin/bash
# ===== 在有网机器上执行 =====

SERVICES=(user-service order-service inventory-service payment-service api-gateway)
REGISTRY="192.168.1.61:80"
VERSION="v1.0.0"

for svc in "${SERVICES[@]}"; do
  echo ">>> 构建 ${svc} ..."
  cd /path/to/${svc}  # 各服务源码目录
  docker build -t ${REGISTRY}/mall/${svc}:${VERSION} .
  docker push ${REGISTRY}/mall/${svc}:${VERSION}
  echo ">>> ${svc} 推送完成"
done

# 导出所有镜像为 tar（传输到离线环境备用）
docker save \
  ${REGISTRY}/mall/user-service:${VERSION} \
  ${REGISTRY}/mall/order-service:${VERSION} \
  ${REGISTRY}/mall/inventory-service:${VERSION} \
  ${REGISTRY}/mall/payment-service:${VERSION} \
  ${REGISTRY}/mall/api-gateway:${VERSION} \
  -o /tmp/mall-services-${VERSION}.tar

# 传输到离线集群
scp /tmp/mall-services-${VERSION}.tar root@<Master节点>:/tmp/

# ===== 在离线集群 Master 节点上执行 =====
# docker load -i /tmp/mall-services-${VERSION}.tar
# 注意：如果已通过方案一直接推送到 Harbor，此步骤可跳过

**方案二：Kaniko（无守护进程镜像构建） 集群内构建（无需 Docker daemon）**

```bash
# 创建 Kaniko Task（Tekton 内使用 Kaniko 在 Pod 内构建镜像）
cat <<'EOF' | kubectl apply -f -
apiVersion: tekton.dev/v1beta1  # Tekton API 版本
kind: Task  # Tekton 任务
metadata:
  name: kaniko-build-push
  namespace: tekton-pipelines
spec:
  params:
  - name: image
    type: string
  - name: dockerfile
    type: string
    default: Dockerfile
  - name: context
    type: string
    default: .
  workspaces:
  - name: source
  - name: dockerconfig
    mountPath: /kaniko/.docker  # Docker 配置挂载路径
  steps:
  - name: build-and-push
    image: 192.168.1.61:80/kaniko-project/executor:v1.19.0  # 镜像地址(Harbor)
    args:
    - --dockerfile=$(params.dockerfile)
    - --context=$(workspaces.source.path)/$(params.context)
    - --destination=$(params.image)
    - --skip-tls-verify
    # 离线 Harbor 使用 HTTP，跳过 TLS 验证
    - --insecure
    - --insecure-pull
    securityContext:
      runAsUser: 0
EOF

# 注意：Kaniko 方案要求 Maven 依赖已缓存在集群内
# 建议在有网环境预先构建好镜像，通过方案一推送到 Harbor
# Kaniko 更适合后续迭代更新时使用
```

| **CI/CD** | ArgoCD | argocd | 200m | 1C | 256Mi | 1Gi | 1 | 200m | 256Mi |
| | Tekton (按需) | tekton-pipelines | 500m | 2C | 512Mi | 2Gi | 0* | 0 | 0 |
| **Ingress** | Nginx Ingress | ingress-nginx | 500m | 2C | 256Mi | 1Gi | 2 | 1C | 512Mi |
| | | | | | | | | | |
| **总计** | | | | | | | | **~6.3C** | **~12.7Gi** |

> **说明：**
> - 微服务默认 **1 副本**（非 2 副本），以节省资源。如需高可用可按需扩容。
> - Tekton PipelineRun（流水线执行实例） 仅在触发时消耗资源，平时不占用。
> - Istio Sidecar 每个微服务 Pod 注入一个，每个约 100m CPU / 128Mi 内存。
> - **总 Request 约 6.3C / 12.7Gi，远低于集群可用 19C/38G，资源充裕。**
> - 如果需要全部微服务 2 副本，总 Request 约 7.5C / 15.3Gi，仍然在安全范围内。

**备份脚本：**

```bash
#!/bin/bash
# mysql-backup.sh - MySQL 备份脚本

BACKUP_DIR="/backup/mysql"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=7

# 创建备份目录
mkdir -p ${BACKUP_DIR}/{full,incremental}

# 全量备份（每日 02:00 执行）
full_backup() {
  echo "[$(date)] 开始全量备份..."
  
  kubectl exec -it deploy/mysql-master -n middleware -- \
    mysqldump -u root -p'Nacos@2024' --all-databases \
    --single-transaction --routines --triggers \
    > ${BACKUP_DIR}/full/full_backup_${DATE}.sql
  
  # 压缩备份文件
  gzip ${BACKUP_DIR}/full/full_backup_${DATE}.sql
  
  echo "[$(date)] 全量备份完成: full_backup_${DATE}.sql.gz"
}

# 增量备份（每小时执行，基于 binlog）
incremental_backup() {
  echo "[$(date)] 开始增量备份..."
  
  # 刷新 binlog
  kubectl exec -it deploy/mysql-master -n middleware -- \
    mysql -u root -p'Nacos@2024' -e "FLUSH LOGS;"
  
  # 复制 binlog 文件
  kubectl cp middleware/mysql-master:/var/lib/mysql/mysql-bin. ${BACKUP_DIR}/incremental/
  
  echo "[$(date)] 增量备份完成"
}

# 清理过期备份
cleanup() {
  echo "[$(date)] 清理 ${RETENTION_DAYS} 天前的备份..."
  find ${BACKUP_DIR}/full -name "*.gz" -mtime +${RETENTION_DAYS} -delete
  find ${BACKUP_DIR}/incremental -name "mysql-bin.*" -mtime +${RETENTION_DAYS} -delete
}

# 主函数
case "$1" in
  full)
    full_backup
    ;;
  incremental)
    incremental_backup
    ;;
  cleanup)
    cleanup
    ;;
  *)
    echo "用法: $0 {full|incremental|cleanup}"
    exit 1
    ;;
esac
```

### 6.6.3 RTO/RPO 验证与灾难恢复手册

**RTO/RPO 定义：**

| 指标 | 定义 | 目标值 | 测量方法 |
|------|------|--------|----------|
| RTO | 恢复时间目标（从故障到服务恢复的时间） | < 5 分钟 | 故障注入到服务可用的时间 |
| RPO | 恢复点目标（可接受的数据丢失时间） | < 1 分钟 | 最后一次备份到故障发生的时间 |

**RTO/RPO 验证脚本：**

```bash
#!/bin/bash
# rto-rpo-verification.sh - RTO/RPO 验证脚本

echo "============================================"
echo "  RTO/RPO 验证测试"
echo "============================================"

# 配置
TEST_NAMESPACE="mall"
TEST_SERVICE="order-service"
TEST_ENDPOINT="http://192.168.1.54:30080/api/health"

# 1. 记录测试开始时间和数据状态
echo "[1/5] 记录初始状态..."
START_TIME=$(date +%s%N)
START_DATA=$(curl -s ${TEST_ENDPOINT} -H "Host: api.mall.local" 2>/dev/null)
echo "  初始数据: $START_DATA"

# 2. 模拟数据写入（用于 RPO 验证）
echo "[2/5] 写入测试数据..."
TEST_ORDER=$(curl -s -X POST http://192.168.1.54:30080/api/orders \
  -H "Host: api.mall.local" \
  -H "Content-Type: application/json" \
  -d '{"userId":"RTO_TEST","items":[{"sku":"TEST-001","qty":1}],"totalAmount":99.99}')
ORDER_ID=$(echo $TEST_ORDER | jq -r '.orderId')
echo "  创建测试订单: $ORDER_ID"

# 记录数据写入时间
DATA_WRITE_TIME=$(date +%s%N)

# 3. 执行故障注入
echo "[3/5] 执行故障注入..."
FAILURE_START_TIME=$(date +%s%N)

# 示例：删除服务 Pod（可根据需要修改故障类型）
kubectl delete pod -n ${TEST_NAMESPACE} -l app=${TEST_SERVICE} --force --grace-period=0

# 4. 测量恢复时间（RTO）
echo "[4/5] 测量恢复时间..."
RECOVERY_START_TIME=$(date +%s%N)

# 等待服务恢复
RETRY_COUNT=0
MAX_RETRY=300  # 最大等待 5 分钟

while [ $RETRY_COUNT -lt $MAX_RETRY ]; do
  HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" ${TEST_ENDPOINT} -H "Host: api.mall.local" 2>/dev/null)
  
  if [ "$HTTP_CODE" == "200" ]; then
    RECOVERY_END_TIME=$(date +%s%N)
    break
  fi
  
  sleep 1
  RETRY_COUNT=$((RETRY_COUNT + 1))
  echo -n "."
done

echo ""

if [ $RETRY_COUNT -eq $MAX_RETRY ]; then
  echo "[错误] 服务未在预期时间内恢复"
  exit 1
fi

# 5. 验证数据恢复（RPO）
echo "[5/5] 验证数据恢复..."
RECOVERED_DATA=$(curl -s "http://192.168.1.54:30080/api/orders/${ORDER_ID}" \
  -H "Host: api.mall.local" 2>/dev/null)

if echo "$RECOVERED_DATA" | grep -q "$ORDER_ID"; then
  echo "  [成功] 测试订单数据已恢复"
  DATA_LOST="否"
else
  echo "  [警告] 测试订单数据丢失"
  DATA_LOST="是"
fi

# 计算 RTO（毫秒）
RTO_MS=$(( (RECOVERY_END_TIME - FAILURE_START_TIME) / 1000000 ))
RTO_SEC=$(( RTO_MS / 1000 ))

# 计算 RPO（毫秒）
RPO_MS=$(( (FAILURE_START_TIME - DATA_WRITE_TIME) / 1000000 ))
RPO_SEC=$(( RPO_MS / 1000 ))

# 输出结果
echo ""
echo "============================================"
echo "  RTO/RPO 验证结果"
echo "============================================"
echo "  RTO (恢复时间): ${RTO_SEC} 秒 (${RTO_MS} 毫秒)"
echo "  RPO (数据丢失): ${RPO_SEC} 秒 (${RPO_MS} 毫秒)"
echo "  数据是否丢失: ${DATA_LOST}"
echo "============================================"

# 验证是否满足目标
if [ $RTO_SEC -lt 300 ]; then
  echo "  [通过] RTO < 5 分钟目标"
else
  echo "  [未通过] RTO 超过 5 分钟目标"
fi

if [ "$DATA_LOST" == "否" ]; then
  echo "  [通过] RPO = 0 (无数据丢失)"
else
  echo "  [警告] 存在数据丢失"
fi
```

## 7. 混沌工程实践

### 7.1 混沌工程概念与原则

**什么是混沌工程？**

混沌工程（Chaos Engineering）是通过在分布式系统上有意注入故障，来验证系统韧性和发现潜在问题的实践方法。其核心思想是"通过破坏来构建更强大的系统"。

**混沌工程五大原则：**

| 原则 | 说明 | 实践示例 |
|------|------|----------|
| 建立稳态假设 | 定义系统正常行为的可测量指标 | 订单服务 P99 延迟 < 500ms，错误率 < 1% |
| 引入真实变量 | 模拟真实世界的故障场景 | 网络延迟、Pod 崩溃、节点宕机 |
| 在生产环境运行 | 在真实环境验证（需控制风险） | 仅影响 mall 命名空间，避开核心时段 |
| 自动化持续运行 | 将混沌实验集成到 CI/CD | 每次发布前自动执行故障演练 |
| 最小化爆炸半径 | 控制故障影响范围 | 使用 Namespace 隔离、PDB 保护 |

**混沌工程架构图：**

```
+================================================================+
|                      混沌工程架构                               |
+================================================================+
|                                                                 |
|   +----------------+     +----------------+     +------------+ |
|   |  混沌实验编排   |---->|  故障注入引擎   |---->|  目标系统   | |
|   |  (Chaos Mesh)  |     |  (CRD/Operator)|     |  (微服务)   | |
|   +----------------+     +----------------+     +------+-----+ |
|          ^                                              |       |
|          |                                              |       |
|          |     +----------------+     +------------+    |       |
|          +-----|  稳态指标监控   |<----|  可观测性    |<---+       |
|                |  (Prometheus)  |     |  (Grafana)  |            |
|                +----------------+     +------------+            |
|                                                                 |
|   实验流程：                                                     |
|   1. 定义稳态指标（SLO）                                          |
|   2. 设计故障场景（YAML）                                          |
|   3. 执行故障注入                                                 |
|   4. 监控指标变化                                                 |
|   5. 自动回滚/恢复                                                |
|   6. 生成实验报告                                                 |
|                                                                 |
+================================================================+
```

### 7.7 混沌工程中止条件与自动回滚

**中止条件（Abort Conditions）** 是混沌工程的安全机制，当系统指标超过预设阈值时自动中止实验，防止故障扩大。

**中止条件定义：**

```yaml
# chaos-abort-conditions.yaml
# 定义混沌实验中止条件

apiVersion: v1  # API 版本
kind: ConfigMap  # K8s 配置映射
metadata:
  name: chaos-abort-conditions
  namespace: chaos-mesh
data:
  # 全局中止条件
  global: |
    # 错误率阈值
    error_rate_threshold: 0.05  # 5% 错误率
    # 延迟阈值
    latency_p99_threshold: 3000  # P99 > 3s
    # 可用性阈值
    availability_threshold: 0.95  # 可用性 < 95%
    # 资源阈值
    cpu_usage_threshold: 0.90  # CPU > 90%
    memory_usage_threshold: 0.90  # 内存 > 90%
  
  # 服务级中止条件
  services: |
    order-service:
      error_rate_threshold: 0.02  # 订单服务更严格，2%
      latency_p99_threshold: 2000  # P99 > 2s
    payment-service:
      error_rate_threshold: 0.01  # 支付服务最严格，1%
      latency_p99_threshold: 1500  # P99 > 1.5s
```

**Prometheus 告警规则作为中止条件：**

```yaml
# prometheus-chaos-alerts.yaml
apiVersion: monitoring.coreos.com/v1  # API 版本
kind: PrometheusRule
metadata:
  name: chaos-abort-rules
  namespace: monitoring
spec:
  groups:
  - name: chaos-abort
    rules:
    # 错误率告警
    - alert: ChaosExperimentHighErrorRate
      expr: |
        sum(rate(http_requests_total{namespace="mall",status=~"5.."}[1m]))
        /
        sum(rate(http_requests_total{namespace="mall"}[1m])) > 0.05
      for: 30s
      labels:
        severity: critical
        chaos_action: abort
      annotations:
        summary: "混沌实验错误率过高"
        description: "错误率 {{ $value | humanizePercentage }} 超过阈值 5%"
    
    # 延迟告警
    - alert: ChaosExperimentHighLatency
      expr: |
        histogram_quantile(0.99,
          sum(rate(http_request_duration_seconds_bucket{namespace="mall"}[1m])) by (le)
        ) > 3
      for: 30s
      labels:
        severity: critical
        chaos_action: abort
      annotations:
        summary: "混沌实验延迟过高"
        description: "P99 延迟 {{ $value }}s 超过阈值 3s"
    
    # 可用性告警
    - alert: ChaosExperimentLowAvailability
      expr: |
        sum(kube_deployment_status_replicas_available{namespace="mall"})
        /
        sum(kube_deployment_spec_replicas{namespace="mall"}) < 0.5
      for: 30s
      labels:
        severity: critical
        chaos_action: abort
      annotations:
        summary: "混沌实验可用性过低"
        description: "可用副本比例 {{ $value | humanizePercentage }} 低于 50%"
```

**自动回滚机制：**

```yaml
# chaos-auto-rollback.yaml
apiVersion: chaos-mesh.org/v1alpha1  # API 版本
kind: Workflow
metadata:
  name: chaos-with-rollback
  namespace: chaos-mesh
spec:
  entry: entry
  templates:
  - name: entry
    templateType: Serial
    serial:
      tasks:
      - inject-fault
      - monitor-and-rollback
  
  # 故障注入任务
  - name: inject-fault
    templateType: NetworkChaos
    networkChaos:
      selector:
        namespaces:
        - mall
        labelSelectors:
          app: inventory-service
      direction: to
      action: delay
      delay:
        latency: "500ms"
      duration: "5m"
  
  # 监控与回滚任务
  - name: monitor-and-rollback
    templateType: Task
    task:
      container:
        name: monitor
        image: curlimages/curl:latest
        command:
        - /bin/sh
        - -c
        - |
          #!/bin/sh
          set -e
          
          PROMETHEUS_URL="http://prometheus.monitoring.svc:9090"
          THRESHOLD_ERROR_RATE=0.05
          THRESHOLD_LATENCY=3000
          
          echo ">>> 开始监控混沌实验..."
          
          for i in $(seq 1 30); do
            sleep 10
            
            # 查询错误率
            ERROR_RATE=$(curl -s "${PROMETHEUS_URL}/api/v1/query?query=sum(rate(http_requests_total{namespace=\"mall\",status=~\"5..\"}[1m]))/sum(rate(http_requests_total{namespace=\"mall\"}[1m]))" | jq -r '.data.result[0].value[1]')
            
            # 查询 P99 延迟
            LATENCY=$(curl -s "${PROMETHEUS_URL}/api/v1/query?query=histogram_quantile(0.99,sum(rate(http_request_duration_seconds_bucket{namespace=\"mall\"}[1m]))by(le))" | jq -r '.data.result[0].value[1]')
            
            echo "检查 $i/30: 错误率=${ERROR_RATE}, P99延迟=${LATENCY}s"
            
            # 检查是否超过阈值
            if [ "$(echo "${ERROR_RATE} > ${THRESHOLD_ERROR_RATE}" | bc)" -eq 1 ]; then
              echo "!!! 错误率超过阈值，触发回滚"
              # 删除所有混沌实验
              kubectl delete networkchaos,PodChaos,IOChaos -n chaos-mesh --all
              exit 1
            fi
            
            if [ "$(echo "${LATENCY} > ${THRESHOLD_LATENCY}" | bc)" -eq 1 ]; then
              echo "!!! 延迟超过阈值，触发回滚"
              kubectl delete networkchaos,PodChaos,IOChaos -n chaos-mesh --all
              exit 1
            fi
          done
          
          echo ">>> 混沌实验完成，指标正常"
```

### 7.8 Cluster Autoscaler 概念

**Cluster Autoscaler (CA)** 是 Kubernetes 的节点自动扩缩容组件，根据 Pod 调度需求和节点资源使用情况自动调整集群节点数量。

**核心工作原理：**

```
+================================================================+
|              Cluster Autoscaler 工作原理                         |
+================================================================+
|                                                                 |
|  扩容触发条件:                                                   |
|  +----------------------------------------------------------+  |
|  |  存在 Pending 状态的 Pod（因资源不足无法调度）             |  |
|  |  → CA 请求云 Provider 创建新节点                          |  |
|  |  → 新节点加入集群                                         |  |
|  |  → Pod 被调度到新节点                                     |  |
|  +----------------------------------------------------------+  |
|                                                                 |
|  缩容触发条件:                                                   |
|  +----------------------------------------------------------+  |
|  |  节点资源利用率低（默认 < 50%）                           |  |
|  |  → CA 尝试驱逐节点上的 Pod                                |  |
|  |  → 成功驱逐后删除节点                                     |  |
|  |  → 云 Provider 释放节点资源                               |  |
|  +----------------------------------------------------------+  |
|                                                                 |
|  缩容安全检查:                                                   |
|  +----------------------------------------------------------+  |
|  |  1. 节点上所有 Pod 都能调度到其他节点                      |  |
|  |  2. 没有 PodDisruptionBudget 阻止驱逐                      |  |
|  |  3. 节点不是 kube-system 关键 Pod 所在节点                 |  |
|  |  4. 节点有 "cluster-autoscaler-kubernetes-io/safe-to-evict" |  |
|  +----------------------------------------------------------+  |
|                                                                 |
+================================================================+
```

**云 Provider 集成：**

| 云平台 | Provider | 特点 |
|--------|----------|------|
| AWS | cluster-autoscaler-aws | 支持 ASG，按需/Spot 实例 |
| GCP | cluster-autoscaler-gce | 原生集成，MIG 支持 |
| Azure | cluster-autoscaler-azure | 支持 VMSS |
| Alibaba Cloud | cluster-autoscaler-alibaba | 支持 ESS 弹性伸缩 |
| OpenStack | cluster-autoscaler-openstack | 私有云支持 |

**Cluster Autoscaler 配置示例（云环境）：**

```yaml
# cluster-autoscaler-deployment.yaml（仅作概念参考）
# 注意：离线/裸金属环境无法使用此功能

apiVersion: apps/v1  # API 版本
kind: Deployment  # K8s 部署
metadata:
  name: cluster-autoscaler
  namespace: kube-system
spec:
  selector:
    matchLabels:
      app: cluster-autoscaler
  template:
    spec:
      serviceAccountName: cluster-autoscaler
      containers:
      - image: k8s.gcr.io/autoscaling/cluster-autoscaler:v1.8.0
        name: cluster-autoscaler
        command:
        - ./cluster-autoscaler
        - --scale-down-unneeded-time=10m  # 节点空闲 10 分钟后缩容
        - --scale-down-delay-after-add=10m  # 扩容后等待 10 分钟才缩容
        - --scale-down-delay-after-failure=3m
        - --scale-down-unready-time=20m
        - --balance-similar-node-groups
        - --expander=priority  # 扩容优先级策略
        - --min-size=3  # 最小节点数
        - --max-size=10  # 最大节点数
        env:
        # 云 Provider 配置（以 AWS 为例）
        - name: AWS_REGION
          value: us-west-2
        resources:
          limits:
            cpu: 100m  # CPU 100m
            memory: 300Mi  # 内存 300Mi
          requests:
            cpu: 100m  # CPU 100m
            memory: 300Mi  # 内存 300Mi
```

**为什么离线/裸金属环境不适用：**

```
+================================================================+
|           Cluster Autoscaler 环境适用性分析                      |
+================================================================+
|                                                                 |
|  云环境（适用）:                                                 |
|  +----------------------------------------------------------+  |
|  |  - 云 Provider API 可用                                   |  |
|  |  - 支持动态创建/删除虚拟机                                |  |
|  |  - 节点可自动加入/离开集群                                |  |
|  |  - 按需付费，扩容成本可控                                 |  |
|  +----------------------------------------------------------+  |
|                                                                 |
|  离线/裸金属环境（不适用）:                                       |
|  +----------------------------------------------------------+  |
|  |  - 无云 Provider API                                      |  |
|  |  - 物理服务器无法动态创建                                 |  |
|  |  - 节点数量固定，无法自动增减                             |  |
|  |  - 硬件采购周期长，无法即时扩容                           |  |
|  +----------------------------------------------------------+  |
|                                                                 |
|  替代方案:                                                       |
|  +----------------------------------------------------------+  |
|  |  1. 手动扩容：提前规划容量，手动添加节点                   |  |
|  |  2. 资源优化：使用 HPA/VPA 优化 Pod 资源使用               |  |
|  |  3. 超卖策略：合理设置 requests/limits 提高资源利用率       |  |
|  |  4. 混合部署：非关键服务与关键服务混合部署                 |  |
|  +----------------------------------------------------------+  |
|                                                                 |
+================================================================+
```

**CKA 考点关联：**

| 考点 | 说明 |
|------|------|
| **节点管理** | 理解节点扩缩容机制 |
| **调度** | 理解 Pending Pod 触发扩容的原理 |
| **PDB** | 理解 PDB 如何影响缩容决策 |
| **资源管理** | 理解 requests/limits 与节点容量的关系 |
| **云原生架构** | 理解云环境与裸金属环境的差异 |

**高频面试题：**

1. **Q: Cluster Autoscaler 如何决定是否扩容？**
   - A: 当存在因资源不足而 Pending 的 Pod 时，CA 会请求云 Provider 创建新节点。CA 定期检查集群状态，发现 Pending Pod 后触发扩容流程。

2. **Q: 为什么裸金属环境无法使用 Cluster Autoscaler？**
   - A: Cluster Autoscaler 依赖云 Provider API 动态创建/删除节点。裸金属环境的物理服务器无法通过 API 即时创建，节点数量固定，因此不适用。

3. **Q: Cluster Autoscaler 与 HPA 有什么区别？**
   - A: HPA 自动调整 Pod 副本数（水平扩缩容），Cluster Autoscaler 自动调整节点数量。HPA 解决 Pod 级别的扩缩容，CA 解决节点级别的扩缩容。两者可以配合使用。

      userId: userId,
      items: [{ sku: sku, qty: 1, price: 99.99 }],
      totalAmount: 99.99,

        })

            name="/api/users/[id]"
        ) as response:

                response.failure(f"Unexpected status: {response.status_code}")

        """查询库存（权重 20%）"""
        sku = random.choice(self.skus)

        """创建订单（权重 30%）"""

            "userId": random.choice(self.user_ids),
            "items": [{
                "sku": random.choice(self.skus),

**瓶颈分析流程：**

```
+================================================================+
|                   系统瓶颈识别流程                              |
+================================================================+
|                                                                 |
|   1. 数据收集                                                    |

|            v                   v                   v            |
|   +--------+-------------------+-------------------+-------+   |
|   |                    数据分析层                          |   |
|   +--------+-------------------+-------------------+-------+   |
|            |                   |                   |            |
|            v                   v                   v            |
|   +--------v-------+  +--------v-------+  +--------v-------+   |
|   | 应用层瓶颈     |  | 中间件瓶颈     |  | 基础设施瓶颈   |   |
|   | - 慢查询       |  | - Kafka 积压   |  | - CPU 饱和     |   |
|   | - 锁竞争       |  | - Redis 热点   |  | - 内存不足     |   |
|   | - GC 频繁      |  | - MySQL 连接池 |  | - 网络 IO      |   |
|   +----------------+  +----------------+  +----------------+   |
|                                                                 |
+================================================================+
```

**常见瓶颈及诊断方法：**

| 瓶颈类型 | 症状 | 诊断命令 | 解决方案 |
|----------|------|----------|----------|
| CPU 饱和 | CPU > 80%, 响应变慢 | `kubectl top pod` | 扩容 Pod/优化代码 |
| 内存不足 | OOMKilled, GC 频繁 | `kubectl describe pod` | 增加内存限制/排查泄漏 |
| 数据库连接池耗尽 | 连接超时错误 | `SHOW PROCESSLIST;` | 增加连接池大小/优化 SQL |
| Kafka 积压 | Consumer Lag 增加 | `kafka-consumer-groups.sh` | 增加消费者/优化消费逻辑 |
| 锁竞争 | 响应时间突增 | Trace 分析 | 优化并发控制/减少锁粒度 |
| 网络 IO | 延迟高但 CPU 低 | `iftop`/`nicstat` | 优化序列化/压缩数据 |

**瓶颈定位脚本：**

```bash
#!/bin/bash
# bottleneck-analysis.sh

echo "============================================"
echo "  系统瓶颈分析脚本"
echo "============================================"

NAMESPACE="mall"

# 1. 应用层瓶颈分析
echo ""
echo "[1/5] 应用层瓶颈分析..."

# 检查 JVM GC
for pod in $(kubectl get pods -n $NAMESPACE -l app=order-service -o name); do
  echo "  $pod GC 统计:"

## 8. 高频面试题（综合）

### Q1: 请描述一个完整的电商订单创建的全链路流程。（难度：困难）

**答案：** 用户下单的全链路流程：1）用户通过浏览器/APP 发送 POST /api/orders 请求；2）DNS 解析将 api.mall.local 解析到 Ingress Controller 的 NodePort（192.168.1.54:30080）；3）Nginx Ingress Controller 根据 Host 和 Path 路由规则将请求转发到 order-service；4）order-service 接收请求后，通过 Nacos 服务发现获取 user-service 地址，调用用户服务查询用户信息（验证用户状态、地址等）；5）order-service 通过 Istio VirtualService（虚拟服务） 路由到 inventory-service，发送库存扣减请求；6）inventory-service 扣减库存成功后，order-service 向 Kafka 的 order-events Topic 发送订单创建事件（Exactly-Once 语义）；7）payment-service 消费 order-events Topic，发起支付；8）支付完成后，payment-service 向 payment-events Topic 发送支付结果事件；9）order-service 消费支付结果事件，更新订单状态为"已支付"；10）整个过程中，OTel Java Agent（OpenTelemetry 探针） 自动采集 Trace、Metrics 和 Logs，发送到 OTel Collector，最终存储到 Tempo（Trace）、Prometheus（Metrics）和 Loki（Logs）。全链路可在 Grafana 中通过 TraceID（全链路追踪 ID） 关联查看。

### Q2: 如何设计一个高可用的微服务架构？（难度：困难）

**答案：** 高可用微服务架构设计从多个层面考虑：**接入层**：Nginx Ingress Controller 至少 2 副本 + PDB，配置限流和熔断；**服务网格**：Istio Sidecar 自动注入，mTLS 加密通信，VirtualService 实现金丝雀发布和故障注入；**服务注册**：Nacos 集群 3 节点 + MySQL 主从，服务注册使用临时实例 + 心跳机制；**消息队列**：Kafka KRaft 3 Broker + 3 Controller，Topic replication.factor=3，min.insync.replicas=2；**数据层**：MySQL 主从复制 + 读写分离，Redis Sentinel 高可用；**可观测性**：Prometheus 2 副本 + Grafana，OTel Collector 2 副本，Tempo + Loki 持久化存储；**CI/CD**：Tekton Pipeline 自动化构建，ArgoCD GitOps（Git 驱动运维） 自动化部署；**容灾**：多可用区部署，Pod 反亲和性，PDB 保障最小可用副本数。资源规划：1 Master（2C4G）+ 5 Worker（4C8G），总资源 22C44G（Worker 可用 20C40G）。关键指标：SLA >= 99.9%，RTO < 5 分钟，RPO < 1 分钟。

### Q3: 如何实现从代码提交到生产部署的全自动化？（难度：困难）

**答案：** 全自动化 CI/CD 链路设计：1）**代码提交**：开发者推送代码到 Gitea（http://192.168.1.61:3000），触发 Gitea Webhook（回调钩子）；2）**CI 流水线（Tekton）**：EventListener（事件监听器） 接收 Webhook -> TriggerBinding（触发参数绑定） 提取参数（commit SHA、分支名、仓库名）-> TriggerTemplate（触发模板） 创建 PipelineRun -> Pipeline 执行：Git Clone -> 单元测试（Maven）-> 构建镜像（Kaniko）-> 推送到 Harbor -> 更新 Helm values Git 仓库中的 image.tag；3）**CD 交付（ArgoCD）**：ArgoCD Watch Helm values Git 仓库 -> 检测到 values.yaml 变更 -> 自动 Sync 新配置到 K8s -> 验证应用健康状态；4）**可观测性**：OTel Agent 自动注入，Trace/Metrics/Logs 自动采集；5）**通知**：Pipeline 成功/失败发送钉钉通知。关键设计点：CI 和 CD 通过 Git 解耦，不直接交互；Helm values 文件作为 CI 和 CD 的桥梁；ArgoCD 的 Self-Heal（自动修复） 确保集群状态与 Git 一致；回滚通过 Git Revert 实现，ArgoCD 自动同步回滚后的配置。

### Q4: 如何排查一个"用户下单后一直显示处理中"的问题？（难度：困难）

**答案：** 系统化排查步骤：1）**确认请求是否到达**：检查 Ingress 访问日志（`kubectl logs -n ingress-nginx`），确认请求是否被正确路由到 order-service；2）**检查 order-service 日志**：`kubectl logs deploy/order-service -n mall`，查看是否有异常日志；3）**检查 Trace**：在 Grafana Tempo 中搜索 order-service 的 Trace，查看请求在哪个环节耗时或失败；4）**检查 Kafka 消息**：查看 order-events Topic 是否有消息积压（`kafka-consumer-groups.sh --describe`），payment-service 消费者是否正常消费；5）**检查 payment-service**：查看 payment-service 日志和 Pod 状态，是否正常运行；6）**检查 Nacos 服务发现**：确认 payment-service 在 Nacos 中注册且健康；7）**检查 Istio 策略**：确认 AuthorizationPolicy（授权策略） 是否阻止了 order-service 到 payment-service 的调用；8）**检查数据库**：确认订单状态是否已写入数据库（可能显示问题而非处理问题）。根据排查结果定位具体环节并修复。

### Q5: 如何设计微服务的限流降级策略？（难度：中等）

**答案：** 限流降级策略分多层面设计：**接入层限流**：Nginx Ingress Controller 配置 `limit-rps=200`（每秒请求数限制）和 `limit-connections=100`（并发连接数限制），防止流量洪峰压垮后端。**网关限流**：Spring Cloud Gateway 配置 Redis 限流（RequestRateLimiter GatewayFilter），按用户/IP/接口维度限流。**服务间限流**：Sentinel 配置 QPS 限流规则（FlowRule），如 order-service 调用 user-service 限制为 500 QPS。**熔断降级**：Sentinel 配置熔断规则（DegradeRule），当错误率 > 50% 或慢调用比例 > 80% 时触发熔断，直接返回降级响应（如"系统繁忙，请稍后再试"）。**Kafka 限流**：配置 Producer 的 `linger.ms` 和 `batch.size` 控制发送速率，Consumer 的 `max.poll.records` 控制消费速率。**降级策略**：核心链路（下单）优先保障，非核心功能（推荐、评价）降级；使用 Sentinel 的 @SentinelResource fallback 方法返回默认值。

### Q6: 如何保证 Kafka 消息的端到端一致性？（难度：困难）

**答案：** 端到端消息一致性需要生产者、Broker、消费者三方配合。**生产者端**：启用 `enable.idempotence=true`（幂等生产）+ `acks=all`（所有 ISR 确认）+ `transactional.id`（事务），确保消息不丢不重。**Broker 端**：`min.insync.replicas=2` + `default.replication.factor=3` + `unclean.leader.election.enable=false`，确保数据不丢失。**消费者端**：关闭自动提交（`enable.auto.commit=false`），手动提交 Offset；消费处理和 Offset 提交在同一事务中（Spring @Transactional + Kafka）。**业务一致性**：使用本地消息表 + 补偿机制：1）order-service 创建订单时同时写入本地消息表（同一事务）；2）定时任务扫描本地消息表，将未发送的消息发送到 Kafka；3）消费者处理成功后发送确认消息；4）如果确认超时，生产者重发消息（消费者需要幂等处理）。**幂等消费**：消费者使用数据库唯一约束或 Redis 去重，确保重复消息不会产生副作用。

### Q7: 在你的架构中，如果 Worker 节点全部宕机，会发生什么？（难度：困难）

**答案：** Worker 节点全部宕机的影响分析：**Pod 影响**：所有 Worker 上的 Pod 会被终止（包括应用服务、Kafka Broker、Ingress Controller NodePort），Master 节点上的系统组件（etcd、kube-apiserver、Controller Manager、Scheduler）不受影响但无法调度新 Pod。**服务影响**：Ingress Controller 不可用，外部流量无法进入集群；应用服务全部不可用；Kafka Broker 如果全部在 Worker 上则不可用。**恢复过程**：1）Worker 节点恢复后，kubelet 重新注册到 Master；2）Controller Manager 检测到 Pod 不在期望节点上，触发重新调度；3）由于 PVC 使用 local-path，如果 Worker 节点磁盘数据丢失，PVC 无法恢复（需要从备份恢复）；4）Kafka Broker 数据存储在 PVC 中，如果数据丢失需要从副本同步。**预防措施**：使用多可用区部署；PVC 使用支持多副本的 StorageClass（如 Ceph RBD、NFS）；Kafka Broker 分布在不同节点；配置 PDB 保障最小可用副本数；定期备份 etcd 和应用数据。

### Q8: 如何设计微服务的灰度发布方案？（难度：困难）

**答案：** 微服务灰度发布方案分多层面设计：**流量层灰度（Istio）**：使用 Istio VirtualService + DestinationRule 实现金丝雀发布。部署新版本 Pod（version: v2），通过 VirtualService 的 weight 字段控制流量比例（如 v1:90%, v2:10%）。可以基于 Header 路由特定用户到新版本（`x-canary: true`）。**数据层灰度（Kafka）**：新版本消费者使用新的 Consumer Group，独立消费 Kafka 消息，避免新旧版本消费冲突。**配置层灰度（Nacos）**：使用 Nacos 的 Beta 灰度发布功能，将新版本配置先推送给特定 IP 的实例验证。**前端灰度（Ingress）**：使用 Nginx Ingress Canary 注解，基于 Cookie 或 Header 将特定用户路由到新版本。**自动化灰度分析**：结合 Prometheus 指标（错误率、延迟 P99）自动判断新版本是否健康，健康则逐步增加流量比例，异常则自动回滚。完整流程：部署 v2（10% 流量）-> 观察 5 分钟（错误率 < 1%、P99 < 500ms）-> 增加到 30% -> 观察 10 分钟 -> 增加到 50% -> 观察 30 分钟 -> 全量切换 -> 下线 v1。

### Q9: 如何设计微服务的日志规范？（难度：中等）

**答案：** 微服务日志规范设计：**日志格式**：统一使用 JSON 格式，便于结构化查询和日志收集。关键字段：`timestamp`（ISO 8601）、`level`（INFO/WARN/ERROR）、`traceId`（OTel TraceID）、`spanId`（OTel SpanID（操作跨度 ID））、`service`（服务名）、`instance`（实例标识）、`thread`（线程名）、`logger`（Logger 名称）、`message`（日志消息）、`stackTrace`（异常堆栈）。**日志级别**：ERROR（系统错误，需要立即处理）、WARN（潜在问题，需要关注）、INFO（业务关键操作，如订单创建、支付成功）、DEBUG（开发调试信息，生产环境关闭）。**日志规范**：不输出敏感信息（密码、Token、身份证号）；使用占位符而非字符串拼接；异常日志必须包含上下文信息（请求参数、用户ID、订单号）。**日志采集**：Promtail 以 DaemonSet 模式采集容器日志，通过 Loki Label（namespace、app、container）索引，日志正文包含 traceId 用于关联 Trace。**日志查询**：Grafana Loki 支持正则表达式和 JSON 解析，支持从日志跳转到 Trace（通过 traceId）。

### Q10: 如何评估和优化微服务架构的性能？（难度：困难）

**答案：** 性能评估和优化从以下维度进行：**基准测试**：使用 JMeter/Gatling 对核心接口进行压力测试，确定系统吞吐量上限（QPS）、响应时间（P50/P90/P99）和资源消耗（CPU/内存）。**性能指标**：通过 Prometheus + Grafana 监控：服务级 QPS、错误率、延迟分布；JVM 级 GC 频率、堆内存使用、线程池状态；Kafka 级消息积压、生产/消费延迟；数据库级连接池使用率、慢查询。**瓶颈定位**：使用 Trace 分析请求链路中每个服务的耗时，定位瓶颈服务；使用 Flame Graph 分析 CPU 热点；使用 Heap Dump 分析内存泄漏。**优化策略**：1）应用层：缓存（Redis）、异步化（Kafka）、连接池优化、SQL 优化、JVM 调优；2）架构层：服务拆分、读写分离、CDN 加速、限流降级；3）基础设施层：Pod 水平扩展（HPA）、JVM 资源调整、Node 扩容。**持续优化**：建立性能基线，每次发布前回归测试；定期压测评估系统容量；配置性能告警（P99 > 阈值、QPS > 容量 80%）。

### Q11: 如何实现微服务的安全加固？（难度：困难）

**答案：** 微服务安全加固从多个层面实施：**网络层**：NetworkPolicy 限制命名空间间和 Pod 间的网络访问；Istio mTLS 加密所有服务间通信；Ingress 配置安全 Header（CSP、X-Frame-Options、HSTS）。**认证层**：API Gateway（API 网关） 统一认证（JWT Token），验证用户身份后传递用户信息到下游服务；Istio AuthorizationPolicy 控制服务间访问权限。**数据层**：敏感数据加密存储（数据库字段加密）；Secret 使用 Sealed Secrets 或 External Secrets 管理；配置中心（Nacos）中的敏感配置加密。**镜像安全**：Harbor 启用镜像扫描（Trivy），阻止高危漏洞镜像部署；使用私有镜像仓库，不使用公共镜像。**CI/CD 安全**：Tekton Pipeline 使用最小权限 ServiceAccount；Git Webhook 验证签名；镜像签名（Cosign/Notation）验证。**审计**：K8s API Server 审计日志；ArgoCD 操作审计；应用访问日志。**依赖安全**：定期扫描依赖漏洞（OWASP Dependency-Check）；使用受信任的依赖版本。**容灾**：定期备份 etcd 和应用数据；配置灾备集群。

### Q12: 如何设计微服务的监控告警体系？（难度：中等）

**答案：** 监控告警体系分四个层次：**基础设施层**：Node CPU/内存/磁盘/网络（Node Exporter + Prometheus）；Pod 资源使用率（cAdvisor）；PVC 使用率。**应用层**：服务 QPS、错误率、延迟 P50/P90/P99（OTel + Prometheus + spanmetrics）；JVM GC、堆内存、线程池（Micrometer + Prometheus）；Kafka 消息积压、Consumer Lag（Kafka Exporter）。**业务层**：订单创建成功率、支付成功率、库存扣减成功率（自定义 Metrics）；活跃用户数、GMV、转化率。**平台层**：Tekton Pipeline 成功率和耗时；ArgoCD Application Sync 状态和健康状态；Nacos 服务注册数量和健康比例。告警规则：P0（立即响应）：服务不可用、错误率 > 10%、数据丢失；P1（30 分钟响应）：P99 延迟 > 2s、Consumer Lag > 10000、Pod 重启；P2（4 小时响应）：CPU > 80%、磁盘 > 85%、证书即将过期。告警渠道：P0 -> 电话 + 短信 + 钉钉；P1 -> 短信 + 钉钉；P2 -> 钉钉 + 邮件。

### Q13: 如何处理微服务的数据一致性？（难度：困难）

**答案：** 微服务数据一致性方案：**强一致性场景**（如库存扣减）：使用分布式事务（Seata AT 模式）或数据库事务（单库场景）。Seata AT 模式通过 Undo Log 实现自动回滚，性能优于 2PC。**最终一致性场景**（如订单-支付）：使用 Saga 模式 + Kafka 事件驱动。Saga 编排：1）order-service 创建订单（PENDING）；2）inventory-service 扣减库存；3）payment-service 发起支付；4）如果任何步骤失败，发送补偿事件（取消订单、恢复库存、退款）。**事件溯源**：所有状态变更以事件形式存储（Kafka + 数据库），通过重放事件重建状态。**读一致性**：使用 Redis 缓存 + 版本号机制，写入数据库后更新缓存，读取时先读缓存再读数据库（Cache Aside Pattern）。**一致性保障**：幂等消费（数据库唯一约束 + Redis 去重）；消息不丢失（Kafka acks=all + 手动提交 Offset）；补偿机制（定时任务扫描超时未确认的事件）。选择建议：核心交易链路使用 Saga + Kafka，查询场景使用 CQRS（命令查询职责分离）。

### Q14: 如何设计微服务的容灾方案？（难度：困难）

**答案：** 微服务容灾方案分几个层面：**应用容灾**：多副本部署（至少 2 副本）+ PDB（PodDisruptionBudget）保障最小可用副本数；HPA（HorizontalPodAutoscaler）根据 CPU/内存/QPS 自动扩缩容；Pod 反亲和性确保副本分布在不同节点。**数据容灾**：MySQL 主从复制 + 自动故障切换（MHA/Orchestrator）；Redis Sentinel 自动故障切换；Kafka replication.factor=3 跨节点副本。**集群容灾**：多 Master 节点（3 Master 高可用）；etcd 3 节点集群；etcd 定期备份（每 6 小时）。**网络容灾**：CoreDNS 多副本；Service 自动负载均衡；Istio Sidecar 自动重试和故障转移。**应用级容灾**：Sentinel 熔断降级，当依赖服务不可用时返回降级响应；超时配置（连接超时 5s，读取超时 10s）；重试策略（指数退避，最多 3 次）。**灾备演练**：定期进行故障演练（Chaos Engineering）：模拟节点宕机、网络分区、服务不可用、数据库故障等场景，验证容灾方案的有效性。**RTO/RPO 目标**：RTO（恢复时间目标）< 5 分钟，RPO（恢复点目标）< 1 分钟。

### Q15: 请总结云原生技术栈的学习路径。（难度：简单）

**答案：** 云原生技术栈推荐学习路径：**基础层**：K8s 核心概念（Pod/Deployment/Service/ConfigMap（配置映射）/Secret/PVC/Ingress）-> Helm 包管理 -> K8s 网络（CNI、Service、Ingress）-> K8s 存储（PV/PVC/StorageClass）。**可观测性层**：Prometheus + Grafana（指标监控）-> OTel + Tempo（分布式追踪）-> Loki（日志聚合）-> 三大支柱关联分析。**服务治理层**：Nacos（服务注册发现 + 配置中心）-> Istio（服务网格、mTLS、流量管理）-> Sentinel（限流降级）。**消息队列层**：Kafka 核心概念（Topic/Partition/Consumer Group）-> 生产者和消费者 -> Exactly-Once 语义 -> KRaft 模式。**交付层**：Tekton（CI 流水线）-> ArgoCD（CD GitOps 交付）-> CI/CD 全链路集成。**实战层**：综合项目（电商微服务）串联所有模块 -> 故障演练 -> 性能优化 -> 安全加固。**认证考试**：CKA（K8s 管理员）-> CKS（K8s 安全专家）-> CKA/CKS 考点融入每个模块的学习中。建议：理论 + 实践结合，每个模块至少完成一次完整的部署和验证。

### 8.5.2 资源利用率监控（CPU/内存请求 vs 实际）

|   - kube_pod_container_resource_limits (限制值)                 |
|                                                                 |
+================================================================+

**资源利用率分析脚本：**

```bash
#!/bin/bash
# resource-utilization-analysis.sh

echo "============================================"
echo "  资源利用率分析"
echo "============================================"

NAMESPACE="mall"

# 1. 获取所有 Pod 的资源配置
echo ""
echo "[1/4] Pod 资源配置分析..."
echo ""
printf "%-30s %-10s %-10s %-10s %-10s\n" "Pod名称" "CPU请求" "CPU限制" "内存请求" "内存限制"
printf "%-30s %-10s %-10s %-10s %-10s\n" "------------------------------" "----------" "----------" "----------" "----------"

kubectl get pods -n $NAMESPACE -o json | jq -r '
  .items[] |
  select(.status.phase == "Running") |
  .metadata.name as $name |
  .spec.containers[] |
  [$name, 
   (.resources.requests.cpu // "0"),
   (.resources.limits.cpu // "无"),
   (.resources.requests.memory // "0"),
   (.resources.limits.memory // "无")] |
  @tsv
' 2>/dev/null | while read line; do
  printf "%-30s %-10s %-10s %-10s %-10s\n" $line
done

# 2. 获取实际资源使用
echo ""
echo "[2/4] 实际资源使用..."
echo ""
printf "%-30s %-15s %-15s\n" "Pod名称" "CPU使用" "内存使用"
printf "%-30s %-15s %-15s\n" "------------------------------" "---------------" "---------------"

kubectl top pods -n $NAMESPACE --no-headers 2>/dev/null | while read line; do
  printf "%-30s %-15s %-15s\n" $line
done

# 3. 计算资源利用率
echo ""
echo "[3/4] 资源利用率计算..."
echo ""

# 获取所有 Deployment 的资源配置和实际使用
for deploy in $(kubectl get deployment -n $NAMESPACE -o name); do
  DEPLOY_NAME=$(echo $deploy | cut -d'/' -f2)
  
  # 获取请求值
  CPU_REQUEST=$(kubectl get deployment $DEPLOY_NAME -n $NAMESPACE -o jsonpath='{.spec.template.spec.containers[0].resources.requests.cpu}' 2>/dev/null || echo "0")
  MEM_REQUEST=$(kubectl get deployment $DEPLOY_NAME -n $NAMESPACE -o jsonpath='{.spec.template.spec.containers[0].resources.requests.memory}' 2>/dev/null || echo "0")
  
  # 获取实际使用（平均值）
  ACTUAL_CPU=$(kubectl top pods -n $NAMESPACE -l app=$DEPLOY_NAME --no-headers 2>/dev/null | awk '{sum+=$2; count++} END {if(count>0) print sum/count; else print 0}')
  ACTUAL_MEM=$(kubectl top pods -n $NAMESPACE -l app=$DEPLOY_NAME --no-headers 2>/dev/null | awk '{sum+=$3; count++} END {if(count>0) print sum/count; else print 0}')
  
  echo "  $DEPLOY_NAME:"
  echo "    CPU请求: $CPU_REQUEST, 实际使用: ${ACTUAL_CPU}m, 利用率: [计算中]"
  echo "    内存请求: $MEM_REQUEST, 实际使用: $ACTUAL_MEM"
done

# 4. 生成优化建议
echo ""
echo "[4/4] 资源优化建议..."
echo ""

# 找出资源使用低于 30% 的 Pod
echo "  低利用率 Pod（可能需要缩容）："
kubectl top pods -n $NAMESPACE --no-headers 2>/dev/null | while read name cpu mem; do
  # 获取请求值
  CPU_REQ=$(kubectl get pod $name -n $NAMESPACE -o jsonpath='{.spec.containers[0].resources.requests.cpu}' 2>/dev/null | sed 's/m//' || echo "100")
  CPU_VAL=$(echo $cpu | sed 's/m//')
  
  if [ -n "$CPU_VAL" ] && [ -n "$CPU_REQ" ] && [ "$CPU_REQ" -gt 0 ]; then
    USAGE=$((CPU_VAL * 100 / CPU_REQ))
    if [ $USAGE -lt 30 ]; then
      echo "    - $name: CPU 使用率 ${USAGE}% (请求: ${CPU_REQ}m, 使用: ${CPU_VAL}m)"
    fi
  fi
done

echo ""
echo "  高利用率 Pod（可能需要扩容）："
kubectl top pods -n $NAMESPACE --no-headers 2>/dev/null | while read name cpu mem; do
  CPU_REQ=$(kubectl get pod $name -n $NAMESPACE -o jsonpath='{.spec.containers[0].resources.requests.cpu}' 2>/dev/null | sed 's/m//' || echo "100")
  CPU_VAL=$(echo $cpu | sed 's/m//')
  
  if [ -n "$CPU_VAL" ] && [ -n "$CPU_REQ" ] && [ "$CPU_REQ" -gt 0 ]; then
    USAGE=$((CPU_VAL * 100 / CPU_REQ))
    if [ $USAGE -gt 80 ]; then
      echo "    - $name: CPU 使用率 ${USAGE}% (请求: ${CPU_REQ}m, 使用: ${CPU_VAL}m)"
    fi
  fi
done
```

**Prometheus 成本监控规则：**

```yaml
# cost-monitoring-rules.yaml

kind: PrometheusRule
metadata:
  name: cost-monitoring-rules
  namespace: monitoring
spec:
  groups:
  - name: resource-utilization
    interval: 60s
    rules:
    # CPU 利用率
    - record: namespace:cpu_utilization_ratio
      expr: |
        sum(rate(container_cpu_usage_seconds_total{container!=""}[5m])) by (namespace)
        /
        sum(kube_pod_container_resource_requests{resource="cpu"}) by (namespace)
    
    # 内存利用率
    - record: namespace:memory_utilization_ratio
      expr: |
        sum(container_memory_working_set_bytes{container!=""}) by (namespace)
        /
        sum(kube_pod_container_resource_requests{resource="memory"}) by (namespace)
    
    # Pod 资源浪费（请求远大于实际使用）
    - alert: HighResourceRequestLowUsage
      expr: |
        (
          sum by (pod, namespace) (
            kube_pod_container_resource_requests{resource="cpu"}
          ) - 
          sum by (pod, namespace) (
            rate(container_cpu_usage_seconds_total[5m])
          )
        ) / 
        sum by (pod, namespace) (
          kube_pod_container_resource_requests{resource="cpu"}
        ) > 0.7
      for: 1h
      labels:
        severity: warning
      annotations:
        summary: "Pod {{ $labels.pod }} CPU 请求过高"
        description: "Pod {{ $labels.pod }} 在 {{ $labels.namespace }} 中 CPU 请求远高于实际使用"
    
    # 资源配额使用率
    - record: namespace:resource_quota_usage
      expr: |
        kube_resourcequota{type="used"} / kube_resourcequota{type="hard"}
  
  - name: cost-estimation
    interval: 300s
    rules:
    # 估算 CPU 成本（假设每核每小时 $0.05）
    - record: namespace:estimated_cpu_cost_hourly
      expr: |
        sum(kube_pod_container_resource_requests{resource="cpu"}) by (namespace) * 0.05
    
    # 估算内存成本（假设每 GB 每小时 $0.01）
    - record: namespace:estimated_memory_cost_hourly
      expr: |
        sum(kube_pod_container_resource_requests{resource="memory"}) by (namespace) / 1024 / 1024 / 1024 * 0.01

### 8.5.3 闲置资源清理

**闲置资源识别脚本：**

```bash
#!/bin/bash
# unused-resource-cleanup.sh

echo "============================================"
echo "  闲置资源清理工具"
echo "============================================"

DRY_RUN=${1:-"true"}  # 默认 dry-run 模式

if [ "$DRY_RUN" == "true" ]; then
  echo "[模式] 预览模式（不会实际删除资源）"
  echo "[提示] 执行实际清理请运行: $0 false"
else
  echo "[模式] 实际清理模式"
  echo "[警告] 即将删除闲置资源，按 Ctrl+C 取消..."
  sleep 5
fi

echo ""

# 1. 查找闲置 ConfigMap
echo "[1/6] 查找未使用的 ConfigMap..."
for ns in $(kubectl get ns -o name | cut -d'/' -f2); do
  for cm in $(kubectl get cm -n $ns -o name | cut -d'/' -f2); do
    # 检查是否有 Pod 引用该 ConfigMap
    USAGE=$(kubectl get pods -n $ns -o yaml | grep "name: $cm" | wc -l)
    if [ $USAGE -eq 0 ]; then
      echo "  [未使用] $ns/$cm"
      if [ "$DRY_RUN" == "false" ]; then
        kubectl delete cm $cm -n $ns
      fi
    fi
  done
done

# 2. 查找闲置 Secret
echo ""
echo "[2/6] 查找未使用的 Secret..."
for ns in $(kubectl get ns -o name | cut -d'/' -f2); do
  for secret in $(kubectl get secret -n $ns -o name | cut -d'/' -f2); do
    # 排除默认 secret
    if [[ "$secret" == "default"* ]] || [[ "$secret" == "istio"* ]]; then
      continue
    fi
    
    USAGE=$(kubectl get pods -n $ns -o yaml | grep "name: $secret" | wc -l)
    if [ $USAGE -eq 0 ]; then
      echo "  [未使用] $ns/$secret"
      if [ "$DRY_RUN" == "false" ]; then
        kubectl delete secret $secret -n $ns
      fi
    fi
  done
done

# 3. 查找闲置 PVC
echo ""
echo "[3/6] 查找未绑定的 PVC..."
for pvc in $(kubectl get pvc --all-namespaces | grep Lost | awk '{print $2":"$1}'); do
  PVC_NAME=$(echo $pvc | cut -d':' -f1)
  PVC_NS=$(echo $pvc | cut -d':' -f2)

```bash
#!/bin/bash
# cost-allocation-report.sh

echo "============================================"
echo "  成本分摊报告"
echo "============================================"

REPORT_DATE=$(date +%Y-%m)
REPORT_FILE="/tmp/cost-report-${REPORT_DATE}.csv"

# CSV 头部
echo "Namespace,Pod,Team,Project,Environment,CPU请求(核),内存请求(GB),估算成本(元/小时)" > $REPORT_FILE

# 遍历所有 Pod
for ns in mall middleware monitoring kafka; do
  echo "处理命名空间: $ns"
  
  kubectl get pods -n $ns -o json 2>/dev/null | jq -r '
    .items[] |
    select(.status.phase == "Running") |
    {
      name: .metadata.name,
      namespace: .metadata.namespace,
      team: (.metadata.labels.team // "unknown"),
      project: (.metadata.labels.project // "unknown"),
      environment: (.metadata.labels.environment // "unknown"),
      cpu_request: (.spec.containers[0].resources.requests.cpu // "0"),
      mem_request: (.spec.containers[0].resources.requests.memory // "0")
    } |
    [.namespace, .name, .team, .project, .environment, .cpu_request, .mem_request] |
    @csv
  ' 2>/dev/null | while read line; do
    # 解析 CSV 行
    NS=$(echo $line | cut -d',' -f1 | tr -d '"')
    POD=$(echo $line | cut -d',' -f2 | tr -d '"')
    TEAM=$(echo $line | cut -d',' -f3 | tr -d '"')
    PROJECT=$(echo $line | cut -d',' -f4 | tr -d '"')
    ENV=$(echo $line | cut -d',' -f5 | tr -d '"')
    CPU=$(echo $line | cut -d',' -f6 | tr -d '"')
    MEM=$(echo $line | cut -d',' -f7 | tr -d '"')
    
    # 转换 CPU 值为核数
    if [[ $CPU == *"m" ]]; then
      CPU_CORES=$(echo $CPU | sed 's/m//' | awk '{printf "%.3f", $1/1000}')
    else
      CPU_CORES=$CPU
    fi
    
    # 转换内存值为 GB
    if [[ $MEM == *"Gi" ]]; then
      MEM_GB=$(echo $MEM | sed 's/Gi//')
    elif [[ $MEM == *"Mi" ]]; then
      MEM_GB=$(echo $MEM | sed 's/Mi//' | awk '{printf "%.3f", $1/1024}')
    else
      MEM_GB=0
    fi
    
    # 计算估算成本（示例：CPU 0.5元/核/小时，内存 0.1元/GB/小时）
    CPU_COST=$(echo "$CPU_CORES * 0.5" | bc -l 2>/dev/null || echo "0")
    MEM_COST=$(echo "$MEM_GB * 0.1" | bc -l 2>/dev/null || echo "0")
    TOTAL_COST=$(echo "$CPU_COST + $MEM_COST" | bc -l 2>/dev/null || echo "0")
    
    echo "$NS,$POD,$TEAM,$PROJECT,$ENV,$CPU_CORES,$MEM_GB,$TOTAL_COST" >> $REPORT_FILE
  done
done

echo ""
echo "============================================"
echo "  成本汇总（按团队）"
echo "============================================"

# 按团队汇总
awk -F',' 'NR>1 {team[$3]+=$8} END {for (t in team) printf "%-20s %.2f 元/小时\n", t, team[t]}' $REPORT_FILE | sort -k2 -nr

echo ""
echo "============================================"
echo "  成本汇总（按项目）"
echo "============================================"

# 按项目汇总
awk -F',' 'NR>1 {project[$4]+=$8} END {for (p in project) printf "%-20s %.2f 元/小时\n", p, project[p]}' $REPORT_FILE | sort -k2 -nr

echo ""
echo "============================================"
echo "  详细报告已保存: $REPORT_FILE"
echo "============================================"

#!/bin/bash
# cost-optimization-apply.sh

echo "============================================"
echo "  成本优化实施工具"

OPTIMIZATION_TYPE=${1:-"all"}

# 1. 资源请求优化
optimize_requests() {

}

# 2. 副本数优化
optimize_replicas() {

  echo "  所有服务副本数已调整为 1"

  - type: Resource

        averageUtilization: 70

### 案例 5：Istio mTLS 导致跨命名空间调用失败

**现象：**
```
order-service（mall 命名空间）调用 user-service（mall 命名空间）正常，
但调用 payment-service（也是 mall 命名空间）返回 403 Forbidden。
```

**排查步骤：**
```bash
# 1. 检查 AuthorizationPolicy
kubectl get authorizationpolicy -n mall
# 发现：存在 user-service-policy 和 kafka-policy

# 2. 检查是否有 payment-service 的策略
kubectl get authorizationpolicy payment-service-policy -n mall -o yaml
# 发现：存在严格策略，只允许来自特定 ServiceAccount 的调用

# 3. 检查 order-service 的 ServiceAccount
kubectl get deployment order-service -n mall -o yaml | grep serviceAccountName
# 发现：order-service 使用 default ServiceAccount

# 4. 检查 AuthorizationPolicy 中的 principals
# 发现：只允许 "cluster.local/ns/mall/sa/payment-processor"
```

**解决方案：**
```bash
# 方案一：更新 AuthorizationPolicy 允许 order-service 调用
kubectl patch authorizationpolicy payment-service-policy -n mall --type merge \
  -p '{"spec":{"rules":[{"from":[{"source":{"principals":["cluster.local/ns/mall/sa/default","cluster.local/ns/mall/sa/order-service"]}}],"to":[{"operation":{"methods":["GET","POST"],"paths":["/api/payment/*"]}}]}]}}'

# 方案二：为 order-service 创建专用 ServiceAccount
kubectl create sa order-service-sa -n mall
kubectl patch deployment order-service -n mall --type merge \
  -p '{"spec":{"template":{"spec":{"serviceAccountName":"order-service-sa"}}}}'

# 更新 AuthorizationPolicy
kubectl patch authorizationpolicy payment-service-policy -n mall --type merge \
  -p '{"spec":{"rules":[{"from":[{"source":{"principals":["cluster.local/ns/mall/sa/order-service-sa"]}}]}]}}'
```

### 案例 8：全链路性能下降

**现象：**
```
