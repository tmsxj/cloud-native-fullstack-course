## 2. StorageClass 部署（local-path-provisioner）

### 2.1 为什么需要 StorageClass

kubeadm 部署的 Kubernetes 集群**默认不提供任何 StorageClass**。这意味着：

- 无法自动创建 PV（PersistentVolume，持久卷）
- 所有需要持久化存储的组件（Prometheus、Loki、Kafka、Nacos 等）都需要手动创建 PV
- 后续课程中大量使用 PVC（PersistentVolumeClaim，持久卷声明）声明存储，如果没有 StorageClass，PVC 将永远处于 Pending（等待绑定）状态

### 2.2 为什么选择 local-path-provisioner

| 方案 | 优点 | 缺点 | 适合场景 |
|------|------|------|----------|
| local-path-provisioner | 轻量、无需额外存储服务、部署简单 | 数据绑定节点，Pod 迁移后数据不跟随 | **教学环境、开发测试** |
| NFS + nfs-subdir-external-provisioner | 数据集中、Pod 可跨节点迁移 | 需要额外 NFS 服务器（网络文件系统） | 生产环境 |
| Ceph RBD / CephFS | 高可用、企业级分布式存储 | 架构复杂、资源消耗大 | 大规模生产 |
| OpenEBS | 云原生存储、功能丰富 | 资源消耗较大 | 中型集群 |

**选择 local-path-provisioner 的理由：**
1. 资源消耗极低（适合 2 核 4GB 的 Master 节点）
2. 部署简单，一个 YAML 即可完成
3. 基于 Node 本地磁盘，无需额外存储服务
4. 完全满足教学和实验需求

### 2.3 完整部署步骤

#### 步骤 1：在美国服务器下载镜像和 YAML

```bash
# ============================================
# 在美国服务器 (43.135.167.116) 上执行
# ============================================

# 下载 local-path-provisioner 的部署 YAML
curl -LO https://raw.githubusercontent.com/rancher/local-path-provisioner/v0.0.26/deploy/local-path-storage.yaml

# 下载镜像
docker pull rancher/local-path-provisioner:v0.0.26

# 给镜像打 tag，准备推送到 Harbor
docker tag rancher/local-path-provisioner:v0.0.26 192.168.1.61/registry.k8s.io/local-path-provisioner:v0.0.26

# 登录 Harbor
docker login 192.168.1.61 -u admin -p Harbor12345

# 推送镜像到 Harbor
docker push 192.168.1.61/registry.k8s.io/local-path-provisioner:v0.0.26
```

#### 步骤 2：将 YAML 传到集群 Master 节点

```bash
# ============================================
# 在美国服务器 (43.168.1.116) 上执行
# ============================================

scp local-path-storage.yaml root@192.168.1.51:/root/
```

#### 步骤 3：修改 YAML 中的镜像地址

```bash
# ============================================
# 在 master1 (192.168.1.51) 上执行
# ============================================

cd /root

# 备份原始 YAML
cp local-path-storage.yaml local-path-storage.yaml.bak

# 修改镜像地址为 Harbor 地址
sed -i 's|image: rancher/local-path-provisioner:.*|image: 192.168.1.61/registry.k8s.io/local-path-provisioner:v0.0.26|g' local-path-storage.yaml

# 确认修改成功
grep "image:" local-path-storage.yaml
```

#### 步骤 4：部署 local-path-provisioner

```bash
# ============================================
# 在 master1 (192.168.1.51) 上执行
# ============================================

# 创建命名空间（如果 YAML 中已定义可跳过）
kubectl apply -f local-path-storage.yaml

# 查看部署状态
kubectl get pods -n local-path-storage -o wide

# 等待 Pod 就绪
kubectl wait --for=condition=ready pod -l app=local-path-provisioner -n local-path-storage --timeout=120s
```

#### 步骤 5：验证 StorageClass

```bash
# ============================================
# 在 master1 (192.168.1.51) 上执行
# ============================================

# 查看 StorageClass
kubectl get sc

# 预期输出：
# NAME                   PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
# local-path (default)   rancher.io/local-path   Delete          WaitForFirstConsumer   false                  1m

# 查看详细信息
kubectl describe sc local-path
```

#### 步骤 6：设置为默认 StorageClass

如果 `local-path` 没有自动成为默认 StorageClass，手动设置：

```bash
# ============================================
# 在 master1 (192.168.1.51) 上执行
# ============================================

# 将 local-path 设置为默认 StorageClass
kubectl patch storageclass local-path -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'

# 验证：确认 (default) 标记
kubectl get sc
```

#### 步骤 7：功能验证（创建测试 PVC）

```bash
# ============================================
# 在 master1 (192.168.1.51) 上执行
# ============================================

# 创建测试 PVC
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: PersistentVolumeClaim # PVC：持久卷声明
metadata:
  name: test-pvc
spec:
  accessModes:
    - ReadWriteOnce # 单节点读写
  storageClassName: local-path # 本地路径存储类
  resources:
    requests:
      storage: 1Gi # 请求1Gi存储
EOF

# 查看 PVC 状态（应该很快变为 Bound）
kubectl get pvc test-pvc

# 创建测试 Pod 使用该 PVC
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod # Pod：K8s最小调度单元
metadata:
  name: test-pod
spec:
  containers:
  - name: test-container
    image: 192.168.1.61/registry.k8s.io/pause:3.9
    volumeMounts:
    - name: test-volume
      mountPath: /data # 容器内挂载路径
  volumes:
  - name: test-volume
    persistentVolumeClaim:
      claimName: test-pvc # 引用上面创建的PVC
EOF

# 查看 Pod 状态
kubectl get pod test-pvc -o wide

# 确认 PVC 已绑定
kubectl get pvc

# 清理测试资源
kubectl delete pod test-pod
kubectl delete pvc test-pvc
```

## 3. Gitea 轻量 Git 服务器部署

### 3.1 为什么需要 Gitea

后续课程中多个模块依赖 Git 仓库：

| 模块 | 依赖方式 |
|------|----------|
| ArgoCD (GitOps) | 从 Git 仓库拉取应用清单，自动同步到集群 |
| Tekton (CI) | 从 Git 仓库拉取源代码进行构建 |
| 全链路综合实战 | 完整的 GitOps 工作流：代码提交 -> CI 构建 -> GitOps 部署 |

在离线环境中，无法访问 GitHub/GitLab，因此需要在内部部署一个轻量 Git 服务器。

### 3.2 为什么选择 Gitea

| 方案 | 资源消耗 | 部署复杂度 | 功能 |
|------|----------|-----------|------|
| Gitea | 极低 (~200MB RAM) | 简单 | Git、PR（Pull Request）、Issue、Webhook |
| GitLab CE | 高 (4GB+ RAM) | 复杂 | 完整 DevOps 平台 |
| Gogs | 极低 | 简单 | 功能较少，维护不活跃 |

Gitea 是最合适的选择：资源占用极低，功能完善，部署简单。

```bash

version: "3.8"

services:
  mysql:
    image: 192.168.1.61/registry.k8s.io/mysql:8.0
    container_name: gitea-mysql
    restart: always
    environment:
      - MYSQL_ROOT_PASSWORD=Harbor12345 # MySQL root密码

      - MYSQL_USER=gitea # 数据库用户
      - MYSQL_PASSWORD=Harbor12345 # 数据库密码

      - /data/gitea/mysql-data:/var/lib/mysql
    ports:
      - "3306:3306" # MySQL端口
    command: --character-set-server=utf8mb4 --collation-server=utf8mb4_unicode_ci
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-pHarbor12345"]
      interval: 10s
      timeout: 5s
      retries: 5

  gitea:
    image: 192.168.1.61/registry.k8s.io/gitea:latest
    container_name: gitea
    restart: always
    depends_on:
      mysql:
        condition: service_healthy
    environment:
      - USER_UID=1000
      - USER_GID=1000
      - DB_TYPE=mysql
      - DB_HOST=mysql:3306
      - DB_NAME=gitea
      - DB_USER=gitea
      - DB_PASSWD=Harbor12345
      - ROOT_URL=http://192.168.1.61:3000

      - /data/gitea/gitea-data:/data

      - "2222:22" # Gitea SSH端口
```

## 4. Nexus Maven 私服部署（可选）

### 4.1 为什么需要 Nexus

在 Tekton CI 流水线中，构建 Spring Boot 应用需要下载 Maven 依赖（JAR 包）。在离线环境中，无法访问 Maven Central（Maven 中央仓库），因此需要部署一个 Maven 私服，预先缓存所有需要的依赖。

> **注意：如果不需要 CI 流水线（跳过 Tekton 章节），可以跳过本节。**

### 4.2 完整部署步骤

#### 步骤 1：在美国服务器下载镜像

```bash
# ============================================
# 在美国服务器 (43.135.167.116) 上执行
# ============================================

# 下载 Nexus3 镜像
docker pull sonatype/nexus3:latest

# 打 tag
docker tag sonatype/nexus3:latest 192.168.1.61/registry.k8s.io/nexus3:latest

# 登录 Harbor
docker login 192.168.1.61 -u admin -p Harbor12345

# 推送镜像到 Harbor
docker push 192.168.1.61/registry.k8s.io/nexus3:latest
```

#### 步骤 2：在 Harbor 节点创建目录

```bash
# ============================================
# 在 Harbor 节点 (192.168.1.61) 上执行
# ============================================

# 创建 Nexus 数据目录
mkdir -p /data/nexus/nexus-data

# 设置目录权限（Nexus 容器内使用 UID 200 运行）
chown -R 200:200 /data/nexus/nexus-data
```

#### 步骤 3：创建 docker-compose.yml

```bash
# ============================================
# 在 Harbor 节点 (192.168.1.61) 上执行
# ============================================

cat <<'EOF' > /data/nexus/docker-compose.yml
version: "3.8"

services:
  nexus:
    image: 192.168.1.61/registry.k8s.io/nexus3:latest
    container_name: nexus
    restart: always
    environment:
      - INSTALL4J_ADD_VM_PARAMS=-Xms512m -Xmx512m -XX:MaxDirectMemorySize=512m
    volumes:
      - /data/nexus/nexus-data:/nexus-data
    ports:
      - "8081:8081" # Nexus Web端口
EOF
```

#### 步骤 4：启动 Nexus

```bash
# ============================================
# 在 Harbor 节点 (192.168.1.61) 上执行
# ============================================

cd /data/nexus
```

## 6. 镜像同步通用流程

### 6.1 统一镜像同步脚本

以下脚本用于在美国服务器上批量下载镜像并推送到 Harbor，可在后续每个章节中复用。

```bash
# ============================================
# 文件名：sync_images.sh
# 位置：美国服务器 (43.135.167.116) /root/sync_images.sh
# 用法：bash sync_images.sh <镜像列表文件>
# ============================================

#!/bin/bash
set -e

# ========== 配置 ==========
HARBOR_ADDR="192.168.1.61"
HARBOR_USER="admin"
HARBOR_PASS="Harbor12345"
HARBOR_PROJECT="registry.k8s.io"

# 登录 Harbor
echo "========================================"
echo "登录 Harbor: ${HARBOR_ADDR}"
echo "========================================"
echo "${HARBOR_PASS}" | docker login ${HARBOR_ADDR} -u ${HARBOR_USER} --password-stdin

# 读取镜像列表
IMAGE_LIST_FILE="${1:-images.txt}"

if [ ! -f "${IMAGE_LIST_FILE}" ]; then
    echo "错误：镜像列表文件 ${IMAGE_LIST_FILE} 不存在"
    echo "用法：bash $0 <镜像列表文件>"
    echo "镜像列表文件格式（每行一个镜像）："
    echo "  rancher/local-path-provisioner:v0.0.26"
    echo "  prom/prometheus:v2.48.1"
    echo "  grafana/loki:2.9.3"
    exit 1
fi

# 处理每个镜像
TOTAL=$(wc -l < "${IMAGE_LIST_FILE}")
SUCCESS=0
FAIL=0
SKIP=0

echo ""
echo "========================================"
echo "开始同步镜像，共 ${TOTAL} 个"
echo "========================================"

while IFS= read -r IMAGE || [ -n "${IMAGE}" ]; do
    # 跳过空行和注释
    if [[ -z "${IMAGE}" || "${IMAGE}" == \#* ]]; then
        continue
    fi

    # 提取镜像名和 tag
    if [[ "${IMAGE}" == *":"* ]]; then
        IMAGE_NAME=$(echo "${IMAGE}" | cut -d':' -f1 | sed 's|.*/||')
        IMAGE_TAG=$(echo "${IMAGE}" | cut -d':' -f2)
    else
        IMAGE_NAME=$(echo "${IMAGE}" | sed 's|.*/||')
        IMAGE_TAG="latest"
    fi

    HARBOR_IMAGE="${HARBOR_ADDR}/${HARBOR_PROJECT}/${IMAGE_NAME}:${IMAGE_TAG}"

    echo ""
    echo "----------------------------------------"
    echo "[${SUCCESS}/${TOTAL}] 处理: ${IMAGE}"
    echo "  目标: ${HARBOR_IMAGE}"

    # 检查 Harbor 中是否已存在
    if docker manifest inspect "${HARBOR_IMAGE}" > /dev/null 2>&1; then
        echo "  状态: 已存在，跳过"
        SKIP=$((SKIP + 1))
        continue
    fi

    # 拉取原始镜像
    echo "  拉取: docker pull ${IMAGE}"
    if ! docker pull "${IMAGE}"; then
        echo "  错误: 拉取失败"
        FAIL=$((FAIL + 1))
        continue
    fi

    # 打 tag
    echo "  打标: docker tag ${IMAGE} ${HARBOR_IMAGE}"
    docker tag "${IMAGE}" "${HARBOR_IMAGE}"

    # 推送到 Harbor
    echo "  推送: docker push ${HARBOR_IMAGE}"
    if ! docker push "${HARBOR_IMAGE}"; then
        echo "  错误: 推送失败"
        FAIL=$((FAIL + 1))
        continue
    fi

    # 清理本地原始镜像（节省空间）
    docker rmi "${IMAGE}" 2>/dev/null || true

    echo "  状态: 成功"
    SUCCESS=$((SUCCESS + 1))

done < "${IMAGE_LIST_FILE}"

echo ""
echo "========================================"
echo "同步完成！"
echo "  成功: ${SUCCESS}"
echo "  跳过: ${SKIP}"
echo "  失败: ${FAIL}"
echo "========================================"
```

| PV/PVC 绑定 | 理解 PV 和 PVC 的绑定关系 | `kubectl get pv,pvc` |
| 动态供给 | StorageClass 自动创建 PV | `kubectl apply -f pvc.yaml` |
| 默认 StorageClass | 多个 SC 时的默认选择逻辑 | `kubectl patch storageclass` |
| 存储访问模式 | ReadWriteOnce、ReadOnlyMany、ReadWriteMany | PVC 的 accessModes |

**关键概念：**

- **ReclaimPolicy（回收策略）**：
  - `Delete`：PVC 删除时，PV 和底层存储一起删除（local-path 默认）
  - `Retain`：PVC 删除时，PV 保留，需手动清理

- **VolumeBindingMode（卷绑定模式）**：
  - `WaitForFirstConsumer`：等待 Pod 调度到节点后再绑定 PV（local-path 使用此模式，确保数据在 Pod 所在节点）
  - `Immediate`：PVC 创建后立即绑定 PV

### 8.2 本地存储 vs 网络存储

| 特性 | 本地存储 (local-path) | 网络存储 (NFS/Ceph) |
|------|----------------------|---------------------|
| 数据位置 | 节点本地磁盘 | 独立存储服务器 |
| Pod 迁移 | 数据不跟随 Pod | 数据跟随 Pod |
| 性能 | 高（本地磁盘） | 取决于网络 |
| 可靠性 | 低（节点故障数据丢失） | 高（多副本） |
| 适用场景 | 临时数据、缓存 | 持久数据、数据库 |

### 8.3 GitOps 基础概念

**CKA/CKS 考试中可能涉及 GitOps 相关概念：**

- **GitOps 核心原则**：
  1. 声明式：所有配置以 YAML 声明式文件存储在 Git 中
  2. 版本化：Git 提供完整的变更历史和审计
  3. 自动化：Git 变更自动触发集群同步
  4. 不可变：每次变更都是新的提交，而非修改现有状态

- **GitOps 工具链**：Gitea（本章部署）+ ArgoCD（第 12 章）= 完整的 GitOps 方案

## 9. 面试题

### 面试题 1：什么是 StorageClass？kubeadm 部署的集群为什么没有默认 StorageClass？

**参考答案：**

StorageClass 是 Kubernetes 中用于定义存储"类别"的资源对象，它描述了存储的类型、Provisioner（存储供给者）、ReclaimPolicy（回收策略）等属性。当用户创建 PVC 时，通过指定 StorageClass，系统可以自动创建对应的 PV，这就是"动态供给"（Dynamic Provisioning）。

kubeadm 部署的集群默认不安装任何 StorageClass，因为 kubeadm 是一个最小化的集群引导工具，不会预设任何存储方案。存储方案的选择取决于底层基础设施（云厂商的云盘、NFS、Ceph 等），kubeadm 无法替用户做这个决定。相比之下，托管型 K8s 服务（如 EKS（AWS）、GKE（Google））通常会自动配置与云盘集成的默认 StorageClass。

在我们的离线环境中，部署 local-path-provisioner 提供了 `local-path` StorageClass，并将其设置为默认，使得后续所有 PVC 请求都能自动满足。

---

### 面试题 2：解释 VolumeBindingMode 中 WaitForFirstConsumer 和 Immediate 的区别。

**参考答案：**

- **Immediate**：PVC 创建后立即绑定 PV，不等待任何 Pod 使用该 PVC。这意味着 PV 可能被绑定到任意节点，而后续调度使用该 PVC 的 Pod 时，可能因为 PV 不在 Pod 调度的节点上而导致 Pod 无法启动（特别是对于本地存储）。

- **WaitForFirstConsumer**：PVC 创建后保持 Pending 状态，直到有 Pod 引用该 PVC 并完成调度。Kubernetes 调度器会综合考虑 Pod 的资源需求、亲和性约束以及 PVC 的存储需求，将 Pod 调度到合适的节点，然后才在该节点上创建并绑定 PV。

对于 local-path-provisioner 这种基于节点本地存储的方案，**必须使用 WaitForFirstConsumer**，否则 PV 可能被分配到节点 A，而 Pod 被调度到节点 B，导致 Pod 无法挂载存储。这也是为什么 local-path-provisioner 默认使用 WaitForFirstConsumer 的原因。

### 面试题 3：在完全离线环境中，如何设计一个可靠的镜像分发方案？

**参考答案：**

在完全离线环境中，镜像分发的核心思路是"引入一个有外网的跳板机"：

1. **架构设计**：美国云服务器（有外网）-> Harbor（内网镜像仓库）-> K8s 节点（离线）
2. **同步流程**：
   - 在美国服务器上通过 `docker pull` 从 Docker Hub 等公共仓库拉取镜像
   - `docker tag` 修改镜像标签为 Harbor 地址前缀
   - `docker push` 推送到内网 Harbor
   - K8s 节点通过 containerd 从 Harbor 拉取镜像
3. **自动化**：编写统一的 `sync_images.sh` 脚本，通过镜像列表文件批量处理
4. **版本管理**：每个模块维护自己的镜像列表文件，便于复现和管理
5. **Helm Chart**：类似地，通过 `helm pull` 下载 Chart 包，`helm push` 推送到 Harbor Helm 仓库

关键注意事项：
- Harbor 必须配置为 HTTP 模式（或配置自签名证书的 HTTPS）
- containerd 需要配置 `/etc/containerd/config.toml` 中的 `plugins."io.containerd.grpc.v1.cri".registry.configs` 以信任 Harbor
- 镜像同步脚本应支持断点续传和已存在跳过

### 面试题 4：什么是 GitOps？与传统 CI/CD 相比有什么优势？

**参考答案：**

GitOps 是一种将 Git 作为"单一事实来源"（Single Source of Truth）的持续交付方法。核心思想是：所有基础设施和应用配置都存储在 Git 仓库中，任何变更都通过 Git 提交触发，由自动化工具（如 ArgoCD）将 Git 中的期望状态同步到 Kubernetes 集群。

**与传统 CI/CD 的对比：**

| 维度 | 传统 CI/CD | GitOps |
|------|-----------|--------|
| 变更方式 | 手动执行 kubectl apply 或 CI 脚本 | Git 提交自动触发 |
| 审计追踪 | 分散在 CI 日志中 | Git 提交历史天然提供完整审计 |
| 回滚 | 手动操作或 CI 脚本 | `git revert` 即可 |
| 安全性 | CI 需要集群管理员权限 | GitOps 工具只需只读 Git + 集群写入权限 |
| 一致性 | 依赖 CI 脚本正确性 | 集群状态始终与 Git 仓库保持一致 |
| 灾难恢复 | 需要备份和恢复脚本 | 重新连接 Git 仓库即可恢复 |

**GitOps 的四大原则：**
1. 声明式：系统的一切都声明式描述
2. 版本化：所有状态存储在 Git 中
3. 自动拉取：自动应用变更
4. 持续协调：持续检测并修正偏差

### 面试题 5：local-path-provisioner 有哪些局限性？在生产环境中应该使用什么方案？

**参考答案：**

**local-path-provisioner 的局限性：**

1. **数据不跟随 Pod 迁移**：PV 绑定到特定节点，Pod 被重新调度到其他节点后无法访问原数据
2. **节点故障导致数据丢失**：节点宕机后，该节点上的所有 PV 数据将不可用
3. **不支持跨节点读写**：Volume 的 accessModes 仅支持 ReadWriteOnce
4. **容量无限制**：不会检查节点实际磁盘空间，可能导致磁盘写满
5. **不支持快照和克隆**：无法创建存储快照或克隆卷

**生产环境推荐方案：**

| 场景 | 推荐方案 | 说明 |
|------|----------|------|
| 通用生产 | Ceph RBD / CephFS | 高可用、支持快照、克隆、扩容 |
| 云上环境 | 云厂商 CSI（AWS EBS、阿里云云盘等） | 与云平台深度集成 |
| 中等规模 | OpenEBS (Mayastor/Jiva) | 云原生存储，支持大部分 CSI 特性 |
| NFS 环境 | nfs-subdir-external-provisioner | 轻量级网络存储方案 |
| 需要文件存储 | Longhorn | 轻量级、K8s 原生、支持快照 |

选择存储方案时需要考虑：数据可靠性要求、性能要求、预算、运维复杂度、是否需要快照/克隆等高级特性。

### 案例 1：StorageClass 不生效 -- PVC 一直处于 Pending 状态

**现象：**

```bash
kubectl get pvc
# NAME      STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   AGE
# test-pvc  Pending                                       local-path     5m
```

**排查步骤：**

```bash
# 1. 检查 StorageClass 是否存在
kubectl get sc
# 如果没有 local-path，说明部署失败，回到第 2 节重新部署

# 2. 检查 StorageClass 是否为默认
kubectl get sc -o wide
# 如果没有 (default) 标记，PVC 未指定 storageClassName 时将无法自动绑定
# 解决：设置为默认
kubectl patch storageclass local-path -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'

# 3. 检查 local-path-provisioner Pod 是否正常运行
kubectl get pods -n local-path-storage
# 如果 Pod 异常，查看日志
kubectl logs -n local-path-storage -l app=local-path-provisioner

# 4. 检查 PVC 事件
kubectl describe pvc test-pvc
# 重点关注 Events 部分，常见错误：
# - "waiting for a volume to be created" -> provisioner 未就绪
# - "failed to provision volume" -> 节点磁盘空间不足或路径权限问题

# 5. 检查节点磁盘空间
df -h /var/lib/rancher/local-path-storage/
# 如果磁盘空间不足，清理空间或更换存储路径

# 6. 检查 containerd 是否能拉取 provisioner 镜像
crictl images | grep local-path-provisioner
# 如果镜像不存在，手动从 Harbor 拉取
# 在对应 Worker 节点上执行：
ctr -n k8s.io images pull 192.168.1.61/registry.k8s.io/local-path-provisioner:v0.0.26
```

**常见原因和解决方案：**

| 原因 | 解决方案 |
|------|----------|
| Provisioner Pod 未运行 | `kubectl delete pod -n local-path-storage -l app=local-path-provisioner` 重建 |
| 镜像拉取失败 | 检查 containerd 配置，确认 Harbor 地址可达 |
| 磁盘空间不足 | `df -h` 检查，清理空间 |
| StorageClass 未设为默认 | `kubectl patch storageclass` 设置默认 |
| helper Pod 无法调度 | 检查节点资源是否充足 |

## 11. CSI存储驱动实战（进阶）

> 本节介绍生产级CSI存储驱动部署，补充local-path的局限性。
> 
> **适用场景**: 生产环境需要数据持久化、跨节点迁移、快照备份
> **推荐方案**: Longhorn（轻量级）或 Rook-Ceph（企业级）

### 11.1 CSI驱动概述

**CSI (Container Storage Interface，容器存储接口)** 是Kubernetes存储的标准接口，让存储厂商可以编写通用驱动程序。

```
┌─────────────────────────────────────────────────────────────┐
│                    Kubernetes                               │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐     │
│  │     PVC     │───►│ StorageClass│───►│  CSI Driver │     │
│  └─────────────┘    └─────────────┘    └──────┬──────┘     │
└─────────────────────────────────────────────────┼───────────┘
                                                  │
                                                  ▼
┌─────────────────────────────────────────────────────────────┐
│                    存储后端                                  │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐     │
│  │   Longhorn  │    │  Rook-Ceph  │    │   NFS CSI   │     │
│  │  (本地分布式)│    │  (企业级)   │    │  (网络存储)  │     │
│  └─────────────┘    └─────────────┘    └─────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

### 11.2 CSI驱动对比

| 驱动 | 类型 | 特性 | 内存需求 | 适用场景 |
|------|------|------|----------|----------|
| **Longhorn** | 分布式块存储 | 快照、克隆、备份、HA | ~500MB/节点 | 中小规模生产 |
| **Rook-Ceph** | 企业级存储 | 块/文件/对象、多副本 | ~2GB/节点 | 大规模生产 |
| **NFS CSI** | 网络存储 | 简单、依赖NFS服务器 | ~100MB | 已有NFS环境 |
| **local-path** | 本地存储 | 无网络开销、最快 | ~50MB | 测试/缓存 |

### 11.4 Longhorn快照与备份

#### 11.4.1 创建快照

```yaml
# volume-snapshot.yaml
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot # 存储快照
metadata:
  name: mysql-snapshot-20250124
spec:
  volumeSnapshotClassName: longhorn-snapshot # 快照类
  source:
    persistentVolumeClaimName: mysql-pvc # 源PVC
```

```bash
# 创建VolumeSnapshotClass（如果不存在）
kubectl apply -f - <<EOF
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass # 快照类定义
metadata:
  name: longhorn-snapshot
driver: driver.longhorn.io # Longhorn CSI驱动
deletionPolicy: Delete # 快照删除策略
EOF

# 创建快照
kubectl apply -f volume-snapshot.yaml

# 查看快照
kubectl get volumesnapshot
```

#### 11.4.2 从快照恢复

```yaml
# restore-pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim # 从快照恢复的PVC
metadata:
  name: mysql-pvc-restored
spec:
  storageClassName: longhorn
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi # 存储容量
  dataSource: # 数据源为快照
    name: mysql-snapshot-20250124
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
```

### 11.6 Rook-Ceph简介（企业级方案）

### 11.7 存储选型决策树

```
需要生产级存储？
├── 是 → 数据规模？
│   ├── 小（<10TB）→ Longhorn（推荐）
│   └── 大（>10TB）→ Rook-Ceph
└── 否 → local-path（测试环境）

需要跨节点迁移？
├── 是 → Longhorn / Rook-Ceph
└── 否 → local-path

需要快照备份？
├── 是 → Longhorn / Rook-Ceph
└── 否 → local-path / NFS CSI

已有NFS服务器？
├── 是 → NFS CSI Driver
└── 否 → Longhorn
```
