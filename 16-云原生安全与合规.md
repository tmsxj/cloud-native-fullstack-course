## 2. 核心概念

### 2.1 镜像安全

镜像安全是云原生安全的第一道防线。

| 安全实践 | 说明 | 工具 |
|----------|------|------|
| 漏洞扫描 | 扫描OS包和语言依赖的已知CVE(通用漏洞披露) | Trivy, Grype, Clair |
| 镜像签名 | 使用数字签名验证镜像来源 | Cosign, Notary |
| SBOM生成 | 生成软件物料清单(Software Bill of Materials) | Trivy, Syft |
| 最小镜像 | 使用distroless或scratch减少攻击面 | Google Distroless |
| 镜像策略 | 禁止latest标签,强制使用digest | OPA, Kyverno |

**镜像分层安全模型：**

```
+--------------------------------------------------+
| Layer 5: Application (你的应用代码)               |
| Layer 4: Runtime (Node.js/Python/Go Runtime)     |
| Layer 3: System Libraries (glibc, openssl)       |
| Layer 2: Base OS (Alpine/Debian/RedHat)          |
| Layer 1: Base Image (scratch/distroless)         |
+--------------------------------------------------+
| 每层都需要扫描, 漏洞可能在任何一层               |
+--------------------------------------------------+
```

### 2.2 Secret管理

Kubernetes Secret(密钥)的安全挑战与解决方案。

| 方案 | 特点 | 适用场景 |
|------|------|----------|
| 原生Secret | Base64编码, etcd加密 | 简单场景,开发测试 |
| Sealed Secrets | 加密存储在Git, 集群内解密 | GitOps工作流 |
| External Secrets | 同步外部KMS(密钥管理服务)/Vault | 企业级密钥管理 |
| HashiCorp Vault | 动态密钥, 自动轮换 | 高安全要求场景 |
| AWS/GCP/Azure KMS | 云厂商托管 | 公有云环境 |

**Secret安全最佳实践：**
- etcd启用静态加密 (Encryption at Rest)
- 使用RBAC限制Secret访问范围
- 启用Secret审计日志
- 定期轮换Secret
- 避免将Secret提交到Git仓库

### 2.3 RBAC (基于角色的访问控制，Role-Based Access Control)

```
+----------------------------------------------------------+
|                    RBAC 核心概念                          |
+----------------------------------------------------------+
|                                                          |
|  +--------+      +--------+      +--------+              |
|  |  User  |----->|  Role  |----->|  Verb  |              |
|  | (用户) |      | (角色) |      | (操作) |              |
|  +--------+      +--------+      +--------+              |
|       |               |               |                  |
|       |          +----+----+   +------+------+           |
|       |          | Cluster |   | get, list   |           |
|       |          |  Role   |   | create      |           |
|       |          +----+----+   | update      |           |
|       |               |        | delete      |           |
|       |               |        +-------------+           |
|       |               |                                  |
|       |          +----+----+      +--------+             |
|       +--------->| RoleBind |----->| Resource|            |
|                  | Cluster  |      | (资源)  |            |
|                  | RoleBind |      +--------+             |
|                  +----------+                             |
|                                                          |
+----------------------------------------------------------+
```

**RBAC最佳实践：**
- 遵循最小权限原则
- 避免使用cluster-admin绑定到ServiceAccount
- 定期审计RBAC配置
- 使用命名空间隔离权限
- 区分人类用户和机器用户(ServiceAccount)

### 2.4 网络策略 (NetworkPolicy)

网络策略(NetworkPolicy)实现Pod级别的微隔离(Micro-segmentation)。

```
+----------------------------------------------------------+
|                    NetworkPolicy 模型                     |
+----------------------------------------------------------+
|                                                          |
|  默认允许所有流量 (Default Allow All)                     |
|  +--------------------------------------------------+   |
|  |  Pod A  <----->  Pod B  <----->  Pod C          |   |
|  |     ^               ^               ^            |   |
|  |     |               |               |            |   |
|  |     +---------------+---------------+            |   |
|  +--------------------------------------------------+   |
|                                                          |
|  应用NetworkPolicy后 (Default Deny + 显式允许)           |
|  +--------------------------------------------------+   |
|  |  Pod A  ------>  Pod B  ------>  Pod C          |   |
|  |     X               ^               X            |   |
|  |                     |                            |   |
|  |              [允许规则]                          |   |
|  +--------------------------------------------------+   |
|                                                          |
|  典型策略模式:                                           |
|  1. 默认拒绝所有入站 (Default Deny Ingress)              |
|  2. 默认拒绝所有出站 (Default Deny Egress)               |
|  3. 按标签选择器精细化控制                                |
|  4. 命名空间级别隔离                                      |
|                                                          |
+----------------------------------------------------------+
```

### 2.5 运行时安全

运行时安全(Runtime Security)关注容器运行期间的异常行为检测。

| 安全机制 | 作用 | 实现方式 |
|----------|------|----------|
| Pod Security | Pod安全标准执行 | PSA (Pod安全准入控制) |
| seccomp | 系统调用过滤(安全计算模式) | securityContext.seccompProfile |
| AppArmor | 程序访问控制 | 节点Profile + Pod注解 |
| SELinux | 强制访问控制 | securityContext.seLinuxOptions |
| Falco | 异常行为检测(云原生运行时安全工具) | eBPF/内核模块监控系统调用 |

**运行时安全层次：**

```
+----------------------------------------------------------+
|                    运行时安全层次                         |
+----------------------------------------------------------+
|                                                          |
|  Layer 1: Pod Security Standards                         |
|  - privileged/baseline/restricted                        |
|  - 命名空间标签控制                                       |
|                                                          |
|  Layer 2: SecurityContext                                |
|  - runAsNonRoot, readOnlyRootFilesystem                  |
|  - capabilities drop/add                                 |
|  - seccompProfile                                        |
|                                                          |
|  Layer 3: 内核级安全                                      |
|  - seccomp (系统调用过滤)                                 |
|  - AppArmor (程序访问控制)                                |
|  - SELinux (强制访问控制)                                 |
|                                                          |
|  Layer 4: 运行时监控                                      |
|  - Falco (异常行为检测)                                   |
|  - 审计日志                                               |
|  - 进程监控                                               |
|                                                          |
+----------------------------------------------------------+
```

### 3.2 镜像拉取与推送脚本

```bash
#!/bin/bash
# pull-and-push-security-images.sh
# 用于离线环境镜像准备

HARBOR_URL="harbor.example.com"
HARBOR_PROJECT="security"
IMAGE_LIST="all-security-images.txt"

# 登录Harbor
docker login ${HARBOR_URL}

# 拉取并推送镜像
while IFS= read -r image; do
    [[ -z "$image" || "$image" =~ ^# ]] && continue
    
    echo "处理镜像: $image"
    
    # 拉取镜像
    docker pull "$image"
    
    # 重新打标签
    image_name=$(echo "$image" | awk -F'/' '{print $NF}')
    new_tag="${HARBOR_URL}/${HARBOR_PROJECT}/${image_name}"
    
    docker tag "$image" "$new_tag"
    
    # 推送到Harbor
    docker push "$new_tag"
    
    echo "已推送: $new_tag"
    echo "----------------------------------------"
done < "$IMAGE_LIST"

echo "所有镜像处理完成"
```

```bash
echo "Trivy离线数据库准备完成"
```

## 4. 实战部署

### 4.1 Harbor镜像漏洞扫描 (Trivy离线部署)

#### 4.1.1 部署Trivy Scanner

```bash
# 在Harbor服务器上部署Trivy (离线模式)
mkdir -p /opt/trivy
cd /opt/trivy

# 创建Trivy配置文件
cat > trivy.yaml << 'EOF'
# Trivy配置文件
severity:
  - HIGH  # 高危漏洞
  - CRITICAL  # 严重漏洞

scan:
  scanners:
    - vuln  # 漏洞扫描
    - secret  # 密钥泄露检测
    - config  # 配置文件检查
  skip-dirs:
    - /var/lib/docker
    - /tmp

db:
  # 离线模式配置
  skip-update: true
  no-progress: true
  repository: ""

cache:
  backend: "fs"
  dir: "/root/.cache/trivy"

format: "table"
EOF

# 创建systemd服务
cat > /etc/systemd/system/trivy-server.service << 'EOF'
[Unit]
Description=Trivy Vulnerability Scanner Server
After=network.target

[Service]
Type=simple
User=root
ExecStart=/usr/local/bin/trivy server --listen 0.0.0.0:8080 --skip-update
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
EOF

# 启动服务
systemctl daemon-reload
systemctl enable trivy-server
systemctl start trivy-server

# 验证服务
systemctl status trivy-server
curl http://localhost:8080/healthz
```

#### 4.4.3 配置镜像扫描策略

```yaml
# harbor-scan-policy.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: harbor-scan-policy
  namespace: harbor
data:
  policy.json: |
    {
      "severity": ["HIGH", "CRITICAL"],
      "vulnerability": {
        "ignore_unfixed": false,
        "package_types": ["os", "library"]
      },
      "application/vnd.security.vulnerability.report": {
        "schema_version": "1.1"
      }
    }
---
# 配置项目级别的镜像准入策略
apiVersion: goharbor.io/v1beta1
kind: Project  # Harbor项目
```

```bash
# cosign-sign-images.sh
# 镜像签名脚本

# 设置环境变量

# 签名镜像函数
sign_image() {
    local image=$1
    local tag=$2
    local full_image="${image}:${tag}"
    
    echo "正在签名镜像: ${full_image}"
    
    cosign sign \
        --key ${COSIGN_KEY} \
        --allow-insecure-registry \
        ${full_image}
    
    echo "签名完成: ${full_image}"
}

    --allow-insecure-registry \
```

### 4.4 K8s RBAC精细化权限控制

#### 4.4.1 命名空间隔离权限模型

```yaml
# rbac-namespace-isolation.yaml
# 命名空间级别的权限隔离

---
# 1. 创建命名空间
apiVersion: v1
kind: Namespace

  name: team-alpha
  labels:
    team: alpha
    environment: production

---
# 2. 创建ServiceAccount

  name: team-alpha-sa
  namespace: team-alpha

---
# 3. 创建Role (命名空间级别)
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: team-alpha-role
  namespace: team-alpha
rules:
# Pod权限
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
# Service权限
- apiGroups: [""]
  resources: ["services"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
# ConfigMap权限
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
# Deployment权限
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
# 禁止访问Secret (最小权限原则)
# 禁止访问Node, ClusterRole等集群资源

---
# 4. 绑定Role到ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: team-alpha-binding
  namespace: team-alpha
subjects:
- kind: ServiceAccount
  name: team-alpha-sa
  namespace: team-alpha
roleRef:
  kind: Role
  name: team-alpha-role
  apiGroup: rbac.authorization.k8s.io
```

#### 生成和归档流程

**Tekton Pipeline中生成SBOM:**
```yaml
apiVersion: tekton.dev/v1beta1
kind: Task
metadata:
  name: generate-and-archive-sbom
  namespace: tekton-pipelines
spec:
  description: 生成SBOM并归档到Harbor
  params:
  - name: image
    type: string
  - name: app-name
    type: string
  - name: app-version
    type: string
  - name: sbom-format
    type: string
    default: "cyclonedx-json"
  workspaces:
  - name: sbom
  - name: dockerconfig
  steps:
  - name: generate-sbom
    image: 192.168.1.61/tekton/anchore/syft:latest
    script: |
      #!/bin/bash
      set -e
      
      mkdir -p $(workspaces.sbom.path)
      SBOM_FILE="$(workspaces.sbom.path)/sbom-$(params.app-name)-$(params.app-version).json"
      
      echo "=== 生成SBOM ==="
      echo "镜像: $(params.image)"
      echo "格式: $(params.sbom-format)"
      
      syft $(params.image) \
        -o $(params.sbom-format)=$SBOM_FILE \
        --scope all-layers
      
      echo "SBOM已生成: $SBOM_FILE"
      
      # 显示统计信息
      COMPONENT_COUNT=$(cat $SBOM_FILE | jq '.components | length' 2>/dev/null || echo "N/A")
      echo "组件数量: $COMPONENT_COUNT"
  
  - name: sign-sbom
    image: 192.168.1.61/tekton/sigstore/cosign:latest
    script: |
      #!/bin/bash
      
      SBOM_FILE="$(workspaces.sbom.path)/sbom-$(params.app-name)-$(params.app-version).json"
      
      echo "=== 签名SBOM ==="
      cosign sign-blob \
        --key env://COSIGN_PRIVATE_KEY \
        --output-signature ${SBOM_FILE}.sig \
        $SBOM_FILE
      
      echo "SBOM签名完成"
    env:
    - name: COSIGN_PRIVATE_KEY
      valueFrom:
        secretKeyRef:
          name: cosign-key
          key: private-key
  
  - name: upload-to-harbor
    image: 192.168.1.61/tekton/curlimages/curl:latest
    script: |
      #!/bin/sh
      
      SBOM_FILE="sbom-$(params.app-name)-$(params.app-version).json"
      
      echo "=== 上传SBOM到Harbor ==="
      
      # Harbor 2.8+ 支持SBOM附件上传
      # 使用Harbor API上传SBOM作为镜像附件
      
      HARBOR_URL="192.168.1.61"
      PROJECT="library"
      REPOSITORY="$(params.app-name)"
      
      # 获取镜像digest
      DIGEST=$(curl -s -u "${HARBOR_USER}:${HARBOR_PASS}" \
        "https://${HARBOR_URL}/v2/${PROJECT}/${REPOSITORY}/manifests/$(params.app-version)" \
        | jq -r '.config.digest')
      
      # 上传SBOM附件
      curl -X POST \
        -u "${HARBOR_USER}:${HARBOR_PASS}" \
        -H "Content-Type: application/vnd.cyclonedx+json" \
        --data-binary @$(workspaces.sbom.path)/${SBOM_FILE} \
        "https://${HARBOR_URL}/api/v2.0/projects/${PROJECT}/repositories/${REPOSITORY}/artifacts/${DIGEST}/attachments/sbom"
      
      echo "SBOM上传完成"
    env:
    - name: HARBOR_USER
      valueFrom:
        secretKeyRef:
          name: harbor-credentials
          key: username
    - name: HARBOR_PASS
      valueFrom:
        secretKeyRef:
          name: harbor-credentials
          key: password
```

#### 4.6.2 自定义Falco规则

```yaml
# falco-custom-rules.yaml
# 自定义Falco规则ConfigMap

apiVersion: v1
kind: ConfigMap
metadata:
  name: falco-custom-rules
  namespace: falco
  labels:
    app.kubernetes.io/name: falco
data:
  custom-rules.yaml: |
    # ==================== 自定义规则 ====================
    
    # 规则1: 检测特权容器启动
    - rule: Privileged Container Started
      desc: Detect privileged container started
      condition: >
        spawned_process and
        container and
        container.privileged=true
      output: >
        Privileged container started
        (user=%user.name command=%proc.cmdline
         container=%container.name image=%container.image.repository)
      priority: CRITICAL
    
    # 规则2: 检测敏感文件访问
    - rule: Sensitive File Access
      desc: Detect access to sensitive files
      condition: >
        open_read and
        container and
        (fd.name contains "/etc/shadow" or
         fd.name contains "/etc/passwd" or
         fd.name contains "/etc/hosts")
      output: >
        Sensitive file accessed
        (user=%user.name file=%fd.name
         command=%proc.cmdline container=%container.name)
      priority: WARNING
    
    # 规则3: 检测异常出站连接
    - rule: Outbound Connection from Sensitive Container
      desc: Detect outbound connections from sensitive containers
      condition: >
        outbound and
        container and
        (container.name contains "database" or
         container.name contains "redis" or
         container.name contains "mysql")
      output: >
        Outbound connection from sensitive container
        (connection=%fd.name container=%container.name
         command=%proc.cmdline)
      priority: NOTICE
    
    # 规则4: 检测Kubernetes API异常访问
    - rule: K8s API Unusual Access
      desc: Detect unusual access to K8s API
      condition: >
        spawned_process and
        container and
        (proc.name in (kubectl, helm) or
         proc.cmdline contains "kubernetes" or
         proc.cmdline contains ".kube")
      output: >
        K8s tool executed in container
        (user=%user.name command=%proc.cmdline container=%container.name)
      priority: WARNING
    
    # 规则5: 检测反向Shell
    - rule: Reverse Shell
      desc: Detect potential reverse shell
      condition: >
        spawned_process and
        container and
        (proc.name in (bash, sh, zsh) and
         (fd.name contains "socket" or
          fd.name contains "tcp"))
      output: >
        Potential reverse shell detected
        (user=%user.name command=%proc.cmdline container=%container.name)
      priority: CRITICAL
    
    # 规则6: 检测挂载敏感目录
    - rule: Sensitive Mount
      desc: Detect container with sensitive hostPath mount
      condition: >
        spawned_process and
        container and
        (container.mounts contains "/var/run/docker.sock" or
         container.mounts contains "/etc/kubernetes" or
         container.mounts contains "/root/.ssh")
      output: >
        Container with sensitive mount started
        (container=%container.name mounts=%container.mounts)
      priority: WARNING
```

### 案例 1：SonarQube扫描误报处理

**现象：**
```
SonarQube报告检测到大量"未使用的导入"问题，
但实际上这些导入在编译时会被使用。
Pipeline因此无法通过质量门禁。
```

**排查步骤：**
```bash
# 1. 查看SonarQube报告详情
# 登录SonarQube Web界面，查看具体问题
# 问题类型：java:S1128 - 未使用的导入

# 2. 检查代码
# 发现问题：Lombok注解生成的代码被误判
cat src/main/java/com/mall/entity/Order.java
# @Data注解会生成getter/setter，但SonarQube静态分析无法识别

# 3. 查看SonarQube规则配置
curl -u admin:admin \
  "http://sonarqube:9000/api/rules/show?key=java:S1128" | jq .

# 4. 检查项目排除配置
# 发现没有配置Lombok相关排除
```

**解决方案：**
```bash
# 方案一：配置SonarQube排除规则（推荐）
# 在sonar-project.properties中添加：
cat >> sonar-project.properties << 'EOF'
# 排除Lombok生成的代码
sonar.java.libraries=target/classes
sonar.exclusions=**/generated-sources/**,**/target/**

# 禁用特定规则
sonar.issue.ignore.multicriteria=e1
e1.ruleKey=java:S1128
e1.resourceKey=**/*.java
EOF

# 方案二：在代码中添加抑制注解
# 对于个别误报，使用@SuppressWarnings
@SuppressWarnings("java:S1128")
public class Order { ... }

# 方案三：调整质量门禁阈值
# 允许一定数量的"次要"问题
curl -u admin:admin \
  -X POST "http://sonarqube:9000/api/qualitygates/create_condition" \
  -d "gateName=MyCompany Gate" \
  -d "metric=minor_violations" \
  -d "op=GT" \
  -d "warning=50" \
  -d "error=100"
```

**预防措施：**
```yaml
# 在Pipeline中添加预处理步骤
- name: sonar-pre-check
  runAfter:
  - build
  script: |
    # 检查是否有Lombok依赖
    if grep -q "lombok" pom.xml; then
      echo "检测到Lombok，自动配置排除规则"
      echo "sonar.java.libraries=target/classes" >> sonar-project.properties
    fi
```

### 案例 2：SBOM合规检查失败

**现象：**
```
安全审计要求提供SBOM，但生成的SBOM缺少部分组件信息。
审计工具报告："SBOM不完整，缺少transitive依赖"
```

**排查步骤：**
```bash
# 1. 检查当前SBOM生成命令
syft packages dir:. -o cyclonedx-json > sbom.json

# 2. 查看生成的SBOM
cat sbom.json | jq '.components | length'
# 发现只有直接依赖，缺少传递依赖

# 3. 检查Maven依赖树
mvn dependency:tree -DoutputFile=dependency-tree.txt
cat dependency-tree.txt | wc -l
# 发现传递依赖数量远多于SBOM中的组件

# 4. 检查Syft扫描范围
syft packages dir:. -vv 2>&1 | grep -i "scope"
# 发现默认只扫描compile scope
```

**解决方案：**
```bash
# 方案一：使用正确的扫描范围
# 扫描所有scope的依赖
syft packages dir:. -o cyclonedx-json \
  --source-scope all-layers \
  > sbom-complete.json

# 方案二：从已构建的JAR扫描（推荐）
# 先构建项目
mvn clean package -DskipTests

# 扫描JAR文件（包含所有传递依赖）
syft packages file:./target/order-service-1.0.0.jar \
  -o cyclonedx-json \
  > sbom-complete.json

# 方案三：使用Maven插件生成
# 在pom.xml中添加CycloneDX插件
cat >> pom.xml << 'EOF'
<plugin>
  <groupId>org.cyclonedx</groupId>
  <artifactId>cyclonedx-maven-plugin</artifactId>
  <version>2.7.9</version>
  <executions>
    <execution>
      <phase>package</phase>
      <goals>
        <goal>makeAggregateBom</goal>
      </goals>
    </execution>
  </executions>
</plugin>
EOF

# 重新构建生成SBOM
mvn clean package
# SBOM将生成在target/bom.json
```

**验证修复：**
```bash

cat sbom.json | jq '.components | length'
# 应该包含所有传递依赖

# 2. 检查关键组件是否存在
cat sbom.json | jq '.components[] | select(.name | contains("log4j"))'

# 3. 验证SBOM格式
```

### 案例 3：依赖漏洞升级冲突

**现象：**
```
SCA扫描发现log4j-core存在CVE-2021-44228漏洞。
尝试升级到2.17.1版本，但出现依赖冲突。
mvn dependency:tree显示多个版本共存。
```

**排查步骤：**
```bash
# 1. 查找log4j依赖
dependency:tree | grep log4j
# 发现：
# [INFO] +- org.apache.logging.log4j:log4j-core:jar:2.14.1:compile
# [INFO] |  \- org.apache.logging.log4j:log4j-api:jar:2.14.1:compile
# [INFO] \- some.library:problematic-lib:jar:1.0:compile
# [INFO]    \- org.apache.logging.log4j:log4j-core:jar:2.14.1:compile

# 2. 查找引入log4j的依赖
mvn dependency:tree -Dincludes=org.apache.logging.log4j

# 3. 检查dependencyManagement
# 发现没有统一管理log4j版本

# 4. 查看effective POM
mvn help:effective-pom | grep -A 5 log4j
```

**解决方案：**
```bash
# 方案一：使用dependencyManagement统一管理（推荐）
# 在pom.xml中添加：
cat >> pom.xml << 'EOF'
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.apache.logging.log4j</groupId>
      <artifactId>log4j-bom</artifactId>
      <version>2.17.1</version>
      <scope>import</scope>
      <type>pom</type>
    </dependency>
  </dependencies>
</dependencyManagement>
EOF

# 方案二：排除传递依赖
# 在引入问题库的依赖中排除log4j
cat >> pom.xml << 'EOF'
<dependency>
  <groupId>some.library</groupId>
  <artifactId>problematic-lib</artifactId>
  <version>1.0</version>
  <exclusions>
    <exclusion>
      <groupId>org.apache.logging.log4j</groupId>
      <artifactId>log4j-core</artifactId>
    </exclusion>
  </exclusions>
</dependency>

<!-- 显式声明安全版本 -->
<dependency>
  <groupId>org.apache.logging.log4j</groupId>
  <artifactId>log4j-core</artifactId>
  <version>2.17.1</version>
</dependency>
EOF

# 方案三：使用Maven Enforcer插件
# 强制依赖版本一致性
cat >> pom.xml << 'EOF'
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-enforcer-plugin</artifactId>
  <version>3.1.0</version>
  <executions>
    <execution>
      <goals>
        <goal>enforce</goal>
      </goals>
      <configuration>
        <rules>
          <dependencyConvergence/>
        </rules>
      </configuration>
    </execution>
  </executions>
</plugin>
EOF
```

**验证修复：**
```bash
# 1. 重新构建
mvn clean package

# 2. 验证依赖树
mvn dependency:tree | grep log4j
# 应该只显示2.17.1版本

# 3. 重新扫描验证
dependency-check.sh --project order-service --scan .
# 确认CVE-2021-44228不再报告

# 4. 运行测试确保功能正常
mvn test
```

**长期预防措施：**
```yaml

  script: |
    # 检查是否有重复依赖
    mvn dependency:analyze-duplicate

    mvn enforcer:enforce
```

## 8. CKA/CKS考点

### 考点1: Pod Security Admission (CKS重点)

**考点内容：**
- 为命名空间配置PSA标签
- 理解privileged/baseline/restricted三个级别
- 配置enforce/audit/warn三种模式

**实操命令：**

```bash
# 配置命名空间PSA
kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/warn=restricted

# 查看PSA配置
kubectl get namespace production --show-labels

# 移除标签
kubectl label namespace production \
  pod-security.kubernetes.io/enforce- \
  pod-security.kubernetes.io/audit- \
  pod-security.kubernetes.io/warn-
```

**考试技巧：**
- 记住三个级别: privileged < baseline < restricted
- enforce模式直接拒绝, audit记录日志, warn返回警告
- restricted级别要求: runAsNonRoot, drop ALL capabilities, allowPrivilegeEscalation=false

### 考点2: SecurityContext配置 (CKS重点)

**考点内容：**
- Pod级和容器级securityContext配置
- capabilities管理
- seccompProfile配置

**实操命令：**


**考点内容：**

- 配置入站/出站规则
- 按标签选择器过滤

```bash

  name: default-deny-ingress
  namespace: production

  - Ingress
EOF

  name: allow-frontend
  namespace: production

      app: backend
  policyTypes:
  - Ingress

  - from:

        matchLabels:

    - protocol: TCP
```

**考试技巧：**

## 9. 面试题

### 面试题1: 如何加固一个Kubernetes Pod?请列举至少10项措施

**参考答案：**

1. **非Root运行**: `runAsNonRoot: true`, `runAsUser: 1000`
2. **只读根文件系统**: `readOnlyRootFilesystem: true`
3. **禁用特权模式**: `privileged: false`
4. **移除Capabilities**: `capabilities.drop: ["ALL"]`, 仅添加必要的
5. **防止特权提升**: `allowPrivilegeEscalation: false`
6. **启用seccomp**: `seccompProfile.type: RuntimeDefault`
7. **资源限制**: 设置requests和limits
8. **健康检查**: 配置livenessProbe和readinessProbe
9. **不自动挂载SA Token**: `automountServiceAccountToken: false`
10. **安全镜像**: 使用distroless或最小化基础镜像
11. **镜像签名验证**: 使用Cosign验证镜像来源
12. **NetworkPolicy**: 限制Pod网络访问

**延伸知识：**
- 这些措施对应Pod Security Standards的restricted级别
- 可以使用OPA/Gatekeeper/Kyverno强制实施这些策略

### 面试题2: 什么是零信任架构?在Kubernetes中如何实现?

**参考答案：**

**零信任核心原则：**
1. 永不信任, 始终验证
2. 假设网络已被攻破
3. 最小权限访问
4. 持续监控和验证

**K8s中的实现：**

| 层次 | 实现方式 |
|------|----------|
| 身份认证 | mTLS (Istio/Linkerd), OIDC认证 |
| 访问控制 | RBAC, Pod Security Admission |
| 网络隔离 | NetworkPolicy, ServiceMesh |
| 运行时安全 | Falco, seccomp, AppArmor |
| 镜像安全 | 镜像签名, 漏洞扫描, SBOM |

**延伸知识：**
- 零信任不是单一产品, 而是安全理念
- 需要结合组织流程和技术手段共同实现

### 面试题3: 如何保护Kubernetes中的Secret?

**参考答案：**

**Secret保护层次：**

1. **传输安全**
   - etcd启用静态加密 (Encryption at Rest)
   - 配置: `--encryption-provider-config`

2. **访问控制**
   - 使用RBAC限制Secret读取权限
   - 按命名空间隔离Secret

3. **运行时保护**
   - 不将Secret作为环境变量 (使用Volume挂载)
   - 设置Secret的resourceVersion防止并发修改

4. **外部密钥管理**
   - HashiCorp Vault
   - AWS/GCP/Azure KMS
   - External Secrets Operator

5. **GitOps安全**
   - Sealed Secrets加密存储
   - SOPS加密

**延伸知识：**
- Secret数据在etcd中是base64编码, 必须启用加密
- 考虑使用SPIFFE/SPIRE进行工作负载身份认证

### 面试题4: 描述一下Falco的工作原理和典型使用场景

**参考答案：**

**工作原理：**
1. **内核态监控**: 通过eBPF或内核模块挂载到系统调用路径
2. **事件捕获**: 捕获所有系统调用和内核事件
3. **规则匹配**: 将事件与预定义规则进行匹配
4. **告警输出**: 匹配规则后触发告警, 支持多种输出方式

**部署架构：**
```
+-----------+     +-----------+     +-----------+
|  Kernel   |---->|   Falco   |---->|  Alerts   |
|  Events   |     |   Engine  |     |  Output   |
+-----------+     +-----------+     +-----------+
                        |
                   +----+----+
                   |  Rules  |
                   +---------+
```

**典型场景：**
- 特权容器启动检测
- 容器内执行shell检测
- 敏感文件访问检测 (/etc/shadow等)
- 异常出站连接检测
- Kubernetes API异常访问
- 反向shell检测

**延伸知识：**
- Falco规则基于Sysdig过滤语法
- 可以与Falcosidekick集成发送到Slack/PagerDuty

### 面试题5: 什么是供应链攻击?如何在K8s环境中防范?

**参考答案：**

**供应链攻击类型：**
1. **依赖混淆**: 上传恶意包到公共仓库
2. **构建系统入侵**: 篡改CI/CD流程
3. **镜像污染**: 使用包含后门的镜像
4. **仓库劫持**: 接管废弃的依赖包

**K8s防范措施：**

| 阶段 | 措施 |
|------|------|
| 开发 | 依赖扫描 (Snyk, Dependabot), 代码签名 |
| 构建 | 加固CI/CD, 构建环境隔离, SLSA来源证明 |
| 镜像 | 镜像签名 (Cosign), 漏洞扫描 (Trivy), SBOM |
| 部署 | 准入控制 (验证签名), OPA/Kyverno策略 |
| 运行 | 运行时监控 (Falco), 网络隔离 |

**延伸知识：**
- SLSA框架提供供应链安全成熟度模型
- Sigstore提供免费的代码签名服务
- 考虑使用distroless镜像减少攻击面

## 10. Kyverno 准入控制

### 10.1 什么是 Kyverno

**Kyverno** 是 CNCF 孵化项目，是一个 Kubernetes 原生的策略管理引擎。它使用声明式 YAML 配置定义策略，无需学习复杂的 Rego 语言（OPA Gatekeeper 使用）。

**Kyverno vs OPA Gatekeeper：**

| 特性 | Kyverno | OPA Gatekeeper |
|------|---------|----------------|
| **策略语言** | YAML（声明式） | Rego（编程语言） |
| **学习曲线** | 低 | 高 |
| **策略验证** | 自动验证 YAML | 需要测试 Rego |
| **策略生成** | 原生支持 | 需要额外配置 |
| **镜像验证** | 原生支持 | 需要配置 |
| **性能** | 略低（Go 实现） | 略高 |
| **社区活跃度** | 高 | 高 |

**Kyverno 架构：**

```
+================================================================+
|                   Kyverno 架构                                  |
+================================================================+
|                                                                 |
|   +----------------+                                            |
|   |  API Server    |                                            |
|   +-------+--------+                                            |
|           |                                                     |
|           v                                                     |
|   +-------+--------+     +----------------+                    |
|   | Admission      |---->|  Kyverno       |                    |
|   | Webhook        |     |  Controller    |                    |
|   +----------------+     +--------+-------+                    |
|                                   |                             |
|           +-----------------------+----------------+            |
|           |                       |                |            |
|   +-------v-------+       +--------v------+  +-----v-----+     |
|   | Policy Engine |       | Policy Store  |  | Generator |     |
|   | (验证/变更)   |       | (ETCD)        |  | (资源生成) |     |
|   +---------------+       +---------------+  +-----------+     |
|                                                                 |
|   策略类型：                                                     |
|   - validate: 验证策略（拒绝不合规资源）                          |
|   - mutate: 变更策略（自动修改资源）                              |
|   - generate: 生成策略（自动创建资源）                            |
|   - verifyImages: 镜像验证策略                                   |
|                                                                 |
+================================================================+
```

### 10.3 策略示例

**镜像仓库验证策略：**

```yaml
# policies/trusted-registry.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: trusted-registry
  annotations:
    policies.kyverno.io/title: "镜像仓库验证"
    policies.kyverno.io/description: "只允许从 Harbor 私有仓库拉取镜像"
    policies.kyverno.io/category: "安全"
    policies.kyverno.io/severity: "high"
spec:
  validationFailureAction: Enforce  # Enforce=拒绝, Audit=仅记录
  background: true
  rules:
  - name: validate-registry
    match:
      any:
      - resources:
          kinds:
          - Pod
    validate:
      message: "镜像必须来自受信任的仓库: 192.168.1.61:80"
      pattern:
        spec:
          containers:
          - image: "192.168.1.61:80/*"  # 只允许 Harbor 镜像
```

**资源限制验证策略：**

```yaml
# policies/require-resource-limits.yaml

  name: require-resource-limits

    policies.kyverno.io/title: "要求资源限制"
    policies.kyverno.io/description: "所有容器必须设置 CPU 和内存 limits"
spec:
  validationFailureAction: Enforce
  background: true
  rules:
  - name: validate-limits
    match:
      any:
      - resources:
          kinds:
          - Pod
    validate:
      message: "容器必须设置 CPU 和内存 limits"

        spec:
          containers:
          - resources:
              limits:
```

### 12.2 构建时扫描（Trivy in CI）

**Tekton Task 集成 Trivy：**

```yaml
# tasks/trivy-scan.yaml
apiVersion: tekton.dev/v1beta1
kind: Task
metadata:
  name: trivy-scan
  namespace: tekton-pipelines
spec:
  description: Trivy 镜像漏洞扫描
  params:
  - name: image
    description: 要扫描的镜像
    type: string
  - name: severity
    description: 严重级别阈值
    type: string
    default: "HIGH,CRITICAL"
  
  results:
  - name: vulnerabilities
    description: 发现的漏洞数量
  
  steps:
  - name: trivy-scan
    image: aquasec/trivy:latest
    script: |
      #!/bin/sh
      set -e
      
      echo ">>> 扫描镜像: $(params.image)"
      
      # 执行扫描
      trivy image --severity $(params.severity) --format json --output /tmp/result.json $(params.image)
      
      # 统计漏洞数量
      HIGH=$(cat /tmp/result.json | jq '[.Results[].Vulnerabilities[] | select(.Severity=="HIGH")] | length')
      CRITICAL=$(cat /tmp/result.json | jq '[.Results[].Vulnerabilities[] | select(.Severity=="CRITICAL")] | length')
      
      echo ">>> 发现漏洞: CRITICAL=${CRITICAL}, HIGH=${HIGH}"
      
      # 输出结果
      echo -n "${HIGH},${CRITICAL}" > $(results.vulnerabilities.path)
      
      # 高危漏洞阻断构建
      if [ "${CRITICAL}" -gt 0 ]; then
        echo "!!! 发现 CRITICAL 漏洞，构建失败"
        exit 1
      fi
```

**Falco 规则示例：**

```yaml
# falco-rules-local.yaml
# 自定义 Falco 规则

- rule: 敏感文件访问
  desc: 检测对敏感文件的访问
  condition: >
    open_read and
    (fd.name in (/etc/shadow, /etc/passwd, /root/.ssh/id_rsa) or
     fd.name startswith /etc/kubernetes/)
  output: "敏感文件访问 (user=%user.name file=%fd.name)"
  priority: WARNING
  tags: [security, filesystem]

- rule: 容器逃逸尝试
  desc: 检测可能的容器逃逸行为
  condition: >
    spawned_process and
    container and
    (proc.name in (nsenter, setns) or
     proc.args contains "--privileged" or
     fd.name startswith /proc/1/ns/)
  output: "可能的容器逃逸 (user=%user.name container=%container.id)"
  priority: CRITICAL
  tags: [security, container]

- rule: 数据库异常访问
  desc: 检测非预期的数据库连接
  condition: >
    inbound_outbound and
    container and
    fd.sport in (3306, 5432, 27017) and
    not container.image.repository in (mysql, postgres, mongo)
  output: "非数据库容器访问数据库 (container=%container.id port=%fd.sport)"
  priority: WARNING
  tags: [security, database]
```

### 12.5 纵深防御策略

**分层防御矩阵：**

| 层级 | 防御措施 | 工具 | 阻断能力 |
|------|----------|------|----------|
| **代码层** | SAST、依赖扫描 | SonarQube、Snyk | CI 阻断 |
| **镜像层** | 漏洞扫描、签名验证 | Trivy、Cosign | 构建阻断 |
| **配置层** | 策略验证、准入控制 | Kyverno、OPA | 部署阻断 |
| **网络层** | 网络隔离、mTLS | NetworkPolicy、Istio | 运行时阻断 |
| **运行时层** | 行为监控、异常检测 | Falco、Tetragon | 告警+可选阻断 |
| **数据层** | 加密存储、访问控制 | KMS、RBAC | 数据保护 |

**纵深防御配置示例：**

```yaml
# defense-in-depth.yaml
# 综合防御策略配置

# 1. 构建时：Trivy 扫描
build:
  vulnerability_scan:
    enabled: true
    severity_threshold: HIGH
    block_on_critical: true
  sbom:
    enabled: true
    format: spdx

# 2. 部署时：Kyverno 策略
deploy:
  admission_control:
    policies:
      - trusted-registry
      - require-resource-limits
      - disallow-privileged
    enforcement: Enforce

# 3. 运行时：Falco 监控
runtime:
  falco:
    rules:
      - 敏感文件访问
      - 容器逃逸尝试
      - 数据库异常访问
    action: alert  # alert 或 terminate

# 4. 网络：NetworkPolicy
network:
  default_deny: true
  allowed_namespaces:
    - monitoring -> mall
    - mall -> middleware
  mtls:
    mode: STRICT

# 5. 数据：加密
data:
  secret_encryption:
    provider: kms
    key_id: alias/k8s-secrets
  pv_encryption:
    enabled: true
    storage_classes:
      - encrypted-local-path
```

### 12.6 CKA/CKS 考点关联

| 考点 | 关联内容 |
|------|----------|
| **准入控制** | 理解 Admission Webhook 机制 |
| **安全策略** | Kyverno/OPA 策略配置 |
| **镜像安全** | 镜像扫描、签名验证 |
| **运行时安全** | Falco 规则配置 |
| **合规** | CIS、SOC2、PCI-DSS 要求 |

**高频面试题：**

1. **Q: Kyverno 和 OPA Gatekeeper 的主要区别？**
   - A: Kyverno 使用 YAML 声明式配置，学习曲线低；OPA Gatekeeper 使用 Rego 编程语言，功能更强大但学习成本高。Kyverno 原生支持镜像验证和资源生成。

2. **Q: 安全扫描应该在哪一层实施？**
   - A: 应该实施分层扫描：构建时扫描镜像漏洞（Trivy），部署时通过准入控制验证（Kyverno），运行时监控异常行为（Falco）。这叫纵深防御（Defense in Depth）。

3. **Q: SOC2 合规在 Kubernetes 中如何实现？**
   - A: SOC2 五大原则对应 Kubernetes 控制：安全性（RBAC、NetworkPolicy）、可用性（PDB、HPA）、处理完整性（镜像签名）、机密性（Secret 加密）、隐私性（数据分类）。

### 10.4 安全加固检查清单

```bash
#!/bin/bash
# security-hardening-checklist.sh
# 生产环境安全加固检查清单

echo "========== Kubernetes 安全加固检查清单 =========="
echo "检查时间: $(date)"
echo ""

CHECKS_PASSED=0
CHECKS_FAILED=0

check_item() {
    local name=$1
    local command=$2
    echo -n "[检查] $name ... "
    if eval "$command" > /dev/null 2>&1; then
        echo "通过"
        ((CHECKS_PASSED++))
    else
        echo "失败"
        ((CHECKS_FAILED++))
    fi
}

# 1. API Server安全
echo "1. API Server安全配置"
echo "----------------------------------------"
check_item "匿名认证已禁用" "kubectl get pods -n kube-system -l component=kube-apiserver -o yaml | grep -q 'anonymous-auth=false'"
check_item "Basic认证已禁用" "kubectl get pods -n kube-system -l component=kube-apiserver -o yaml | grep -q 'basic-auth-file' && false || true"
check_item "Token认证已启用" "kubectl get pods -n kube-system -l component=kube-apiserver -o yaml | grep -q 'token-auth-file'"

# 2. etcd安全
echo ""
echo "2. etcd安全配置"
echo "----------------------------------------"
check_item "etcd启用TLS" "kubectl get pods -n kube-system -l component=etcd -o yaml | grep -q 'cert-file'"
check_item "etcd启用客户端认证" "kubectl get pods -n kube-system -l component=etcd -o yaml | grep -q 'client-cert-auth=true'"

# 3. Kubelet安全
echo ""
echo "3. Kubelet安全配置"
echo "----------------------------------------"
check_item "Kubelet启用匿名认证禁用" "kubectl get nodes -o yaml | grep -q 'anonymous-auth=false'"
check_item "Kubelet启用只读端口禁用" "kubectl get nodes -o yaml | grep -q 'read-only-port=0'"

# 4. Pod安全
echo ""
echo "4. Pod安全配置"
echo "----------------------------------------"
check_item "PSA已启用" "kubectl get namespace production -o yaml | grep -q 'pod-security.kubernetes.io'"
check_item "默认ServiceAccount未自动挂载" "kubectl get namespace production -o yaml | grep -q 'automountServiceAccountToken: false'"

# 5. 网络安全
echo ""
echo "5. 网络安全配置"
echo "----------------------------------------"
check_item "NetworkPolicy已部署" "kubectl get networkpolicy -n production | grep -q 'default-deny'"
check_item "CNI插件已启用" "kubectl get pods -n kube-system | grep -q 'calico\|cilium\|flannel'"

# 6. 审计日志
echo ""
echo "6. 审计日志配置"
echo "----------------------------------------"
check_item "审计日志已启用" "kubectl get pods -n kube-system -l component=kube-apisenter -o yaml | grep -q 'audit-policy-file'"

echo ""
echo "========== 检查结果汇总 =========="
echo "通过: $CHECKS_PASSED"
echo "失败: $CHECKS_FAILED"
echo ""

if [ $CHECKS_FAILED -eq 0 ]; then
    echo "所有安全检查通过!"
else
    echo "存在 $CHECKS_FAILED 项检查未通过, 请修复后重新检查"
fi
```
